חלק 2 — שאלות פתוחות (כתיבת SQL)

נסו לכתוב את השאילתה בעצמכם, ואז גלו את הפתרון לדוגמה.

שאלה פתוחה 1

נתונות הטבלאות:

Event(event_id, time, neighborhood, description)   -- אירוע
Person(ID, f_name, l_name, age, height)             -- אדם
Connection(ID, event_id)                            -- טבלת קשר אדם–אירוע

א. כתבו שאילתה המחזירה את האירועים בהם היו יותר מ-5 אנשים.

ב. כתבו שאילתה המחזירה את כל האנשים (שם פרטי ושם משפחה) שהשתתפו באירוע יום הסטודנט (description = 'student day') או באירוע מבחן סוף שנה (description = 'exam') וגם גילם מעל 25.

ג. כתבו את פקודות ה-SQL: יצירת טבלת ערים city (קוד עיר, שם עיר); יצירת טבלת שכונה neighborhood (קוד שכונה, שם שכונה, קוד העיר שאליה היא שייכת); הוספת שכונה; ומחיקת עיר כך שהשכונות ששייכות אליה יימחקו גם הן.

א.

SELECT event_id
FROM Connection
GROUP BY event_id
HAVING COUNT(*) > 5;

מקבצים לפי אירוע, סופרים אנשים, ומסננים count>1 עם HAVING. (אם רוצים גם תיאור: לצרף JOIN Event.)

ב.

SELECT DISTINCT p.f_name, p.l_name
FROM Person p
JOIN Connection c ON p.ID = c.ID
JOIN Event e      ON c.event_id = e.event_id
WHERE e.description = 'student day'
   OR (e.description = 'exam' AND p.age > 25);

שימו לב לסוגריים סביב תנאי ה-exam: הגיל >25 חל רק עליו. DISTINCT מונע כפילויות אם אדם השתתף בכמה אירועים.

ג.

CREATE TABLE city (
  city_code INT PRIMARY KEY,
  city_name VARCHAR(50)
);

CREATE TABLE neighborhood (
  neighborhood_code INT PRIMARY KEY,
  neighborhood_name VARCHAR(50),
  city_code INT,
  FOREIGN KEY (city_code) REFERENCES city(city_code)
    ON DELETE CASCADE
);

-- הוספת שכונה
INSERT INTO neighborhood (neighborhood_code, neighborhood_name, city_code)
VALUES (1, 'Florentin', 5);

-- מחיקת עיר; השכונות שלה נמחקות אוטומטית בזכות ON DELETE CASCADE
DELETE FROM city WHERE city_code = 5;

המפתח: ON DELETE CASCADE על מפתח הזר גורם למחיקת השכונות התלויות כשמוחקים את העיר.

🧪 משימה: כתבו את הפתרון שלכם כאן והריצו אותו על נתוני דוגמה אמיתיים — ורק אז השוו לפתרון. אפשר לנסות את כל הסעיפים; כפתור «בדיקה» בודק את סעיף ב׳ (הטבלאות כבר טעונות עם נתונים).

SELECT DISTINCT p.f_name, p.l_name
FROM Person p
JOIN Connection c ON p.ID = c.ID
JOIN Event e      ON c.event_id = e.event_id
WHERE e.description = 'student day'
   OR (e.description = 'exam' AND p.age > 25);

שאלה פתוחה 2

נתונה טבלה לא-מנורמלת על השאלת ספרים, עם העמודות:

reader_id, reader_name, phone_number, address,
book_name, author_name, book_category, rent_from, rent_to

(קורא יכול לשאול כמה ספרים; אותו ספר יכול להיות מושאל בזמנים שונים.)

א. פרקו את הטבלה לטבלאות מנורמלות (מספיק לרשום את שם הטבלה והעמודות, עם ציון מפתחות).

ב. תארו דיאגרמת ERD עבור הטבלאות.

א. פירוק מנורמל (עד 3NF):

Reader(reader_id PK, reader_name, phone_number, address)

Book(book_id PK, book_name, author_id FK, book_category)
   -- ניתן גם להפריד קטגוריה: Category(category_id PK, category_name)

Author(author_id PK, author_name)

Rental(rental_id PK,
       reader_id FK -> Reader,
       book_id  FK -> Book,
       rent_from, rent_to)
   -- לחלופין מפתח מורכב (reader_id, book_id, rent_from)

העיקרון: כל עובדה נשמרת פעם אחת. פרטי הקורא תלויים רק ב-reader_id; פרטי הספר/מחבר רק ב-book_id; פרטי ההשאלה (תאריכים) הם הקשר בין קורא לספר. כך נמנעת כפילות (שם הקורא לא חוזר בכל שורת השאלה) ואנומליות עדכון/מחיקה.

ב. ERD (מילולית):

  • Reader (1) ⟵שואל⟶ (N) Rental (N) ⟵של⟶ (1) Book — כלומר קשר רבים-לרבים בין Reader ל-Book מפורק דרך ישות הקשר Rental (שנושאת את rent_from/rent_to).
  • Author (1) ⟵כתב⟶ (N) Book — מחבר אחד לכמה ספרים.
  • מפתחות ראשיים מסומנים בקו תחתון; קשרי הרבים מיוצגים במפתחות זרים ב-Rental וב-Book.

🧪 משימה: רוצים לבדוק את הפירוק שלכם בפועל? צרו כאן את הטבלאות המנורמלות (CREATE TABLE), הכניסו כמה שורות והריצו שאילתות על המבנה שהצעתם.
