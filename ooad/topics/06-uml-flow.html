Activity Diagram — זרימת בקרה

תרשים פעילות (Activity Diagram) מתאר זרימת עבודה ובקרה — פעולות, החלטות, פעילויות מקביליות והמעברים ביניהן. מעין תרשים זרימה (flowchart) מתקדם.

  • התחלה: עיגול מלא ●. סיום: עיגול בתוך עיגול ◉.
  • Action: מלבן מעוגל.
  • Decision/Merge: מעוין ◇ עם תנאים (guards) על היציאות.
  • Fork/Join: פס עבה — לפיצול/איחוד זרימות מקביליות.
  • Swimlanes: «מסלולים» שמייחסים פעולות לאחראי (מחלקה/תפקיד).

שימוש: ניתוח ותכן — מידול תהליכים עסקיים ואלגוריתמים/לוגיקת מתודות מורכבת.

מקור: Exercise5-BehavioralDiagrams.pdf

דוגמה מלאה: תרשים Activity לעיבוד הזמנה

נבנה תרשים Activity «מאפס» עבור תהליך עסקי אמיתי — עיבוד הזמנה בחנות. נעקוב אחרי הזרימה צעד-אחר-צעד וניצור כל סימן במקומו:

  • התחלה — עיגול מלא ●. קריטי למבחן סימן ההתחלה של תרשים Activity הוא עיגול שחור מלא (solid filled circle), בדיוק כמו ב-State. לא מעוין, לא מלבן.
  • Action: «קבל הזמנה» (Receive Order) — מלבן מעוגל.
  • Fork (פס עבה) — התהליך מתפצל לשתי זרימות שרצות במקביל: «מלא את ההזמנה» (Fill Order) ו-«שלח חשבונית» (Send Invoice). שתיהן יכולות לקרות בו-זמנית — אין תלות ביניהן.
  • Decision (מעוין ◇) על ענף המשלוח: [rush order?]. אם התנאי (guard) מתקיים — «משלוח לילה» (Overnight Delivery); אחרת — «משלוח רגיל» (Regular Delivery).
  • Join (פס עבה) — שתי הזרימות המקבילות נפגשות ומתאחדות שוב לזרימה אחת. הזרימה ממתינה עד ששתיהן הסתיימו.
  • Action: «סגור הזמנה» (Close Order), ואז סיום — עיגול בתוך עיגול ◉.
הכלל של Fork/Join: Fork פותח זרימות מקביליות, Join סוגר אותן ומסנכרן. Join אינו ממשיך עד שכל הזרימות שנכנסות אליו הגיעו — זה נקודת סנכרון. אל תבלבלו עם Decision/Merge (מעוין) שבוחר נתיב אחד מתוך כמה, בלי מקביליות.

Swimlanes — דוגמה: תשלום ב-Checkout

מסלולים (Swimlanes) הם עמודות אנכיות שמחלקות את התרשים לפי מי אחראי לכל פעולה — כל Action נמצא במסלול של הגורם שמבצע אותו. זה מראה בבירור «מי עושה מה» בתהליך. נדגים תהליך checkout באתר מסחר עם שני מסלולים: |User| ו-|System|.

  • |User|: מוסיף פריט לעגלה ← «המשך קניות?» (לולאת חזרה — repeat loop שחוזרת לבחירת פריטים) ← «עבור לתשלום».
  • |System|: בודק אם המשתמש מחובר — ◇ [logged in?]; אם לא, מפנה למסך התחברות. מעבד תשלום ← ◇ [payment ok?]: אם הצליח → «צור הזמנה» ו-«שלח אישור» (fork מקבילי); אם נכשל → «הצג שגיאה» וחזרה לתשלום.
העשרה — לולאה בתרשים Activity: אין סימן «loop» ייעודי. לולאה נבנית מ-Decision שאחד מענפיו חוזר אחורה אל Action קודם (בדיוק כמו «המשך קניות?» שחוזר לעגלה). כך מייצגים חזרתיות בתהליך.

State Diagram — מחזור חיים של אובייקט

תרשים מצבים (State Diagram / Statechart) מתאר את מחזור החיים של אובייקט יחיד — כל המצבים שהוא יכול להיות בהם, והאירועים/תנאים שמפעילים מעברים ביניהם.

  • State: מלבן מעוגל (עם פעולות entry/, exit/).
  • Transition: חץ מתויג event [guard] / action.
  • Initial: עיגול מלא ●. Final: ◉.
  • Composite state: מצב המכיל תת-מצבים.

שימוש: תכן — אובייקטים בעלי התנהגות תלוית-מצב: רכיבי UI, פרוטוקולים, מחזור חיים של הזמנה, התקני חומרה.

מקור: Exercise5-BehavioralDiagrams.pdf

Guard, entry/exit/do, ומעבר-עצמי

מעבר בתרשים State מתויג בפורמט event [guard] / action:

  • event — האירוע שמפעיל את המעבר (למשל motionDetected()).
  • [guard] — קריטי למבחן תנאי בוליאני בסוגריים מרובעים שחייב להיות אמת כדי שהמעבר יתרחש (fire). אם ה-guard שקרי, האירוע קורה אך המעבר אינו מתבצע. דוגמה: withdraw() [balance >= amount] — משיכה עוברת רק אם יש כיסוי.
  • / action — פעולה שמתבצעת כשהמעבר קורה (למשל / switchToNight()).

בתוך מצב אפשר להגדיר שלושה סוגי פעילות פנימית:

  • entry / … — מתבצע פעם אחת ברגע הכניסה למצב (למשל entry / turnOnLight).
  • exit / … — מתבצע פעם אחת ברגע היציאה מהמצב (למשל exit / logTimestamp).
  • do / … — פעילות מתמשכת שרצה כל עוד האובייקט במצב (do-activity), עד שאירוע מוציא אותו. דוגמה: במצב Opening של דלת מעלית — do / disable open-detector; או do / manageDayLights בבית חכם.
מעבר-עצמי (Self-transition): חץ שיוצא ממצב וחוזר אליו עצמו. הוא מפעיל מחדש את exit ואז entry ומריץ את ה-action. דוגמה: מצב Night בבית חכם — motionDetected() / adjustLighting() משאיר את המערכת ב-Night אך מכוונן את התאורה בכל תנועה.

History ואזורים מקביליים במצב מורכב

מצב מורכב (composite) יכול להוסיף שתי יכולות מתקדמות מעבר לתת-מצבים פשוטים: זיכרון היסטוריה ו-אזורים מקביליים.

מצב היסטוריה (History, H)

סמל H בתוך עיגול, בתוך מצב מורכב (composite). כשחוזרים למצב המורכב, ה-history «זוכר» את תת-המצב האחרון שהאובייקט היה בו — במקום להתחיל מ-Initial מחדש.

  • דוגמה אמיתית: נגן מוזיקה במצב מורכב Playing עם תת-מצבים Track 1/2/3. אירוע pause() יוצא ל-Paused; resume() חוזר דרך [H] בדיוק ל-Track 2 שהתנגן — ולא לתחילת הרשימה.
  • H* (deep history) זוכר גם תת-מצבים מקוננים לעומק, לא רק את הרמה הראשונה.

אזורים מקביליים (Concurrent / Orthogonal Regions)

מצב מורכב יכול להתחלק לכמה אזורים מקביליים ע"י קו מקווקו (dashed line). האובייקט נמצא בו-זמנית במצב אחד מכל אזור — כל אזור מריץ תת-מכונת-מצבים משלו במקביל.

  • דוגמה אמיתית — Bounded Buffer: אזור אחד עוקב אחרי המילוי — Empty ⇄ Partial ⇄ Full (מעברים כמו put [items = MAX] / PUT); אזור שני, מקווקו-מופרד, עוקב אחרי הגישה — Idle ⇄ Writing ⇄ Reading. הבאפר נמצא בו-זמנית במצב מילוי ובמצב גישה.
  • דוגמה יומיומית — טלפון: אזור «שיחה» (Idle/OnCall) ואזור «סוללה» (Charging/Discharging) רצים במקביל — אפשר להיות OnCall ו-Charging בו-זמנית.
מקביליות — Activity מול State: ב-Activity המקביליות מסומנת ב-Fork/Join (פס עבה). ב-State המקביליות מסומנת ב-אזורים אורתוגונליים מופרדים בקו מקווקו בתוך מצב מורכב.

דוגמה מלאה: תרשים State למחזור חיים של הזמנה

תרשים State מתאר את מחזור החיים של אובייקט יחיד — כאן, אובייקט Order בודד. הוא מראה בכל רגע באיזה מצב ההזמנה נמצאת ואיזה אירוע מעביר אותה למצב הבא:

  • מצבים (States): New (נוצרה), Paid (שולמה), Shipped (נשלחה), Delivered (נמסרה), ו-Cancelled (בוטלה).
  • מעברים (Transitions): כל חץ מתויג באירוע שמפעיל אותו — placeOrder(), confirmPayment(), shipOrder(), deliverOrder(), cancelOrder().
  • כמה נתיבים לסיום: הזמנה יכולה להגיע ל-◉ דרך Delivered (הצלחה) או דרך Cancelled (ביטול מ-New או מ-Paid).
קריטי למבחן תרשים State מתאר את מחזור החיים של אובייקט אחד בלבד בעל התנהגות תלוית-מצב (state-dependent behavior): אותה קריאת מתודה עשויה להתנהג אחרת בהתאם למצב. למשל cancelOrder() חוקית מ-New/Paid, אך לא מ-Shipped — ההזמנה כבר בדרך.

Activity מול State — לא לבלבל

  • Activity עוקב אחרי זרימת פעולות/תהליך (מה קורה, בסדר). מתאים לתהליך עסקי או אלגוריתם.
  • State עוקב אחרי מצבי אובייקט והמעברים ביניהם (באיזה מצב הוא נמצא). מתאים למחזור חיים תלוי-אירועים.
