תבניות התנהגות — תקשורת

תבניות התנהגות (Behavioral) עוסקות באופן התקשורת וחלוקת האחריות בין אובייקטים — אלגוריתמים והקצאת תפקידים. בקורס: Observer, Template Method, Strategy, Iterator, Command.

מקור: Behavioral-Design-Patterns.pdf

Observer

מטרה: תלות אחד-לרבים — כשאובייקט (Subject) משנה מצב, כל התלויים בו (Observers) מקבלים הודעה ומתעדכנים אוטומטית. שימושים אופייניים: מערכות אירועים, התראות MVC, pub/sub. שאלת המבחן Q27 מבקשת במפורש לתאר את המשתתפים (participants) והיחסים — לכן חשוב לדעת בעל-פה את ארבעת התפקידים ואת רצף ההודעות.

המשתתפים (Participants)

  • Subject — הנושא המופשט (ממשק). מחזיק רשימת מנויים ומגדיר שלוש פעולות: attach(o) (הוסף מנוי), detach(o) (הסר מנוי) ו-notify() (הודע לכולם). אינו מכיר את המחלקות הקונקרטיות של המשקיפים — רק את הממשק Observer.
  • Observer — הממשק המופשט של המשקיף, עם פעולה אחת: update(). כל מי שרוצה להתעדכן מממש אותו.
  • ConcreteSubject — הנושא הקונקרטי. שומר את המצב (subjectState), מספק getState()/setState(), וקורא ל-notify() כשהמצב משתנה.
  • ConcreteObserver — המשקיף הקונקרטי. מחזיק הפניה ל-ConcreteSubject, וב-update() מושך את המצב העדכני דרך subject->getState() ומסנכרן את עצמו.

היחסים: Subject מצביע על רבים Observer (אסוציאציה אחד-לרבים); ConcreteSubject יורש מ-Subject; ConcreteObserver יורש מ-Observer ומחזיק הפניה חזרה ל-ConcreteSubject.

דוגמה מהעולם: גיליון אלקטרוני (MVC)

אובייקט Data (Subject) מחזיק את נתוני הגיליון. שלוש תצוגות מנויות עליו: גרף עוגה, טבלה והיסטוגרמה — כולן Observers. כשהמשתמש משנה מספר בתא, Data קורא ל-notify(), וכל התצוגות מציירות את עצמן מחדש — מבלי ש-Data יודע בכלל מי הן. דוגמה שנייה: תחנת מזג-אוויר — חיישן טמפרטורה (Subject) → תצוגה נוכחית, סטטיסטיקה, תחזית (Observers).

הזרימה: setState → notify → update → getState

class Observer { public: virtual void update() = 0; virtual ~Observer(){} };

class Subject {
  vector<Observer*> obs;
public:
  void attach(Observer* o){ obs.push_back(o); }          // הרשמה
  void detach(Observer* o){ /* מחיקה מהרשימה */ }         // ביטול הרשמה
  void notify(){ for (auto o : obs) o->update(); }        // שידור לכולם
};

class WeatherData : public Subject {   // ConcreteSubject
  float temp;
public:
  float getState(){ return temp; }
  void setState(float t){ temp = t; notify(); }          // שינוי מצב מפעיל שידור
};

class Display : public Observer {      // ConcreteObserver
  WeatherData* subj;
public:
  Display(WeatherData* s):subj(s){ s->attach(this); }
  void update() override { render(subj->getState()); }    // מושך את המצב (pull)
};

Push מול Pull: ב-pull (כמו בקוד למעלה) notify() רק מודיע "משהו השתנה", וכל משקיף מושך בעצמו את מה שהוא צריך דרך getState() — גמיש אך דורש קריאה נוספת. ב-push ה-Subject שולח את הנתונים כפרמטר של update(data) — יעיל אך מצמיד את ה-Subject לצרכים של המשקיפים.

יתרונות שהמבחן אוהב: (1) צימוד מופשט — ה-Subject מכיר רק את הממשק Observer, כך שנושא ומשקיפים יכולים להשתנות בנפרד. (2) תקשורת שידור (broadcast) — ה-Subject משדר לכל המנויים בלי לציין מי המקבלים; אפשר להוסיף/להסיר מנויים בזמן ריצה.

חסרונות (drawbacks)

  • שכחת detach() → מצביע תלוי (dangling pointer) — אם משקיף נמחק אך לא הוסר מרשימת ה-Subject, הקריאה הבאה ל-update() תיגש לזיכרון משוחרר. זהו הבאג הנפוץ ביותר של Observer ב-C++.
  • סדר התראה לא מוגדר — אין ערובה לסדר שבו המשקיפים מקבלים update(); משקיפים תלויים זה בזה עלולים לגרום ל-race conditions.
  • מפלי עדכונים בלתי צפויים (cascades) — שינוי תמים אחד יכול להפעיל שרשרת עדכונים (משקיף שמשנה מצב שגורר notify נוסף), עם סיכון לעדכונים מעגליים ולבעיות ביצועים; קשה לאתר מי הפעיל את מי.
  • update() "שמנה" מפרה SRP — משקיף שדוחס לתוך update() טיפול בנתונים + עדכון UI + קריאת API + לוגים — שביר וקשה לבדיקה.

מקור: Behavioral-Design-Patterns.pdf, Exercise11-DesignPatternsProxyObserver.pdf

Strategy

מטרה: להגדיר משפחת אלגוריתמים, לעטוף כל אחד, ולהפוך אותם לניתנים להחלפה בזמן ריצה דרך ממשק משותף — הלקוח בוחר אסטרטגיה. דוגמאות: אלגוריתמי מיון, שיטות תשלום, אסטרטגיות דחיסה. שאלת המבחן Q28c מציגה חנות עם כמה אסטרטגיות הנחה הניתנות להחלפה בזמן ריצה ושואלת איזו תבנית, לאיזו קטגוריה היא שייכת, ולמה היא עדיפה על שרשרת if/else.

המשתתפים

  • Strategy — הממשק המשותף לכל האלגוריתמים, עם פעולה כמו algorithmInterface().
  • ConcreteStrategy — מימוש קונקרטי אחד של האלגוריתם (וריאנט). לכל וריאנט מחלקה משלו.
  • Context — מחזיק הפניה ל-Strategy, מאפשר להחליף אותה (setStrategy()), ומאציל אליה את העבודה. הלקוח מגדיר איזו אסטרטגיה לשים.

דוגמה מהעולם: עגלת קניות ותשלום

ShoppingCart הוא ה-Context; PaymentStrategy הוא הממשק; CreditCardPayment ו-PayPalPayment הם ה-ConcreteStrategy. המשתמש בוחר אמצעי תשלום ב-checkout — והעגלה מחליפה אסטרטגיה בזמן ריצה בלי שום if. דוגמה נוספת מהדק: Container (Context) מאציל את פריסת הרכיבים ל-LayoutManager — BorderLayout או GridLayout (ConcreteStrategies).

struct PaymentStrategy {                 // Strategy
  virtual void pay(int amount) = 0;
  virtual ~PaymentStrategy(){}
};
struct CreditCardPayment : PaymentStrategy {   // ConcreteStrategy
  void pay(int a) override { /* חיוב כרטיס */ }
};
struct PayPalPayment : PaymentStrategy {       // ConcreteStrategy
  void pay(int a) override { /* חיוב PayPal */ }
};
class ShoppingCart {                     // Context
  PaymentStrategy* strat = nullptr;
public:
  void setStrategy(PaymentStrategy* s){ strat = s; }   // החלפה בזמן ריצה
  void checkout(int total){ strat->pay(total); }       // האצלה
};
// שימוש:
ShoppingCart cart;
cart.setStrategy(&card);   cart.checkout(100);
cart.setStrategy(&paypal); cart.checkout(50);          // הוחלף בלי if/else

למה עדיף על if/else? שרשרת if/else ענקית מפרה את OCP (כל אלגוריתם חדש דורש עריכת אותה פונקציה), מרכזת קוד לא-קשור בגוש אחד, וקשה לבדיקה. Strategy מפריד כל אלגוריתם למחלקה נבדקת בנפרד, ומאפשר להוסיף וריאנט חדש בלי לגעת ב-Context — הרחבה דרך הרכבה (composition) במקום תנאים.

חסרונות

  • הלקוח חייב להכיר את האסטרטגיות — הבחירה ידנית: מישהו חייב לדעת אילו וריאנטים קיימים כדי לבחור אחד; מוסיף צימוד למשפחת האסטרטגיות.
  • ריבוי מחלקות (class explosion) — כל וריאנט = מחלקה נוספת (SortAscending, SortDescending...).
  • העברת נתוני הקשר — לעיתים האסטרטגיה זקוקה לנתונים מה-Context (פרופיל לקוח וכו') שיש להעביר ידנית; עלול לשבור אנקפסולציה או לנפח את הממשק בפרמטרים שחלק מהאסטרטגיות לא צריכות.
  • תקורת עקיפה (indirection) — קריאה וירטואלית לכל הפעלה; במסלולים קריטיים לביצועים אסטרטגיה פשוטה שאפשר היה להטמיע (inline) עולה יותר.

מקור: Exercise12-DesignPatternsStrategyTemplateIterator.pdf

Template Method

מטרה: להגדיר את שלד האלגוריתם במחלקת הבסיס ולהשאיר לתת-מחלקות למלא שלבים ספציפיים — שלד קבוע, שלבים משתנים. דוגמאות: frameworks, לולאות משחק, צינורות ייבוא נתונים.

Strategy מול Template Method: Strategy מחליף אלגוריתם שלם דרך הרכבה (composition) בזמן ריצה; Template Method משנה שלבים דרך הורשה (inheritance) בזמן קומפילציה.

המשתתפים

  • AbstractClass — מגדיר את templateMethod() (השלד, סדר השלבים הקבוע) ואת ה-primitiveOperation() המופשטות (השלבים שתת-מחלקות ימלאו).
  • ConcreteClass — מממש את השלבים הפרימיטיביים, בלי לגעת בשלד.

הכלל: את templateMethod() עצמה אסור לדרוס — רק את השלבים הפרימיטיביים. ב-C++ מכריזים עליה non-virtual (ב-Java/C# — final) כדי לאכוף זאת.

עקרון הוליווד (Hollywood Principle): "Don't call us, we'll call you" — מחלקת הבסיס היא זו שקוראת אל תת-המחלקות בנקודות ההרחבה, לא להפך. זהו היפוך בקרה קלאסי ומונח שהמבחן אוהב.

דוגמה מהעולם: ייצוא נתונים

DataExporter (AbstractClass) מגדיר exportData() ציבורית שקוראת בסדר קבוע ל-fetchData() → formatData() → save() (שלבים מוגנים/מופשטים). CSVExporter ו-JSONExporter דורסים רק את formatData(). דוגמה נוספת מהדק: Shape עם moveTo() כתבנית שקוראת ל-setX/setY/setVisible, ו-Circle/Rectangle דורסים רק את setVisible().

class DataExporter {                          // AbstractClass
public:
  void exportData() {                         // template method — non-virtual
    auto rows = fetchData();
    string out = formatData(rows);            // primitive — משתנה
    save(out);
  }
protected:
  virtual string formatData(Rows) = 0;        // primitiveOperation
  Rows fetchData(){ /* משותף */ }
  void save(string){ /* משותף */ }
};
class CSVExporter : public DataExporter {     // ConcreteClass
  string formatData(Rows r) override { return toCSV(r); }
};

מתי להשתמש: כשלכמה תהליכים מבנה זהה אך שלבים שונים — התבנית מונעת שכפול קוד בתת-מחלקות ואוכפת סדר ביצוע קבוע של השלבים.

חסרונות

  • נעילה להורשה (inheritance lock-in) — אי אפשר לשנות התנהגות בלי לרשת ולדרוס; בזמן קומפילציה בלבד (שלא כמו Strategy הדינמי).
  • ריבוי מחלקות — כל וריאציה דורשת תת-מחלקה חדשה.
  • זרימה "מוסתרת" — הלוגיקה מפוצלת בין הבסיס לתת-המחלקה; קשה לעקוב אחר הריצה השלמה.
  • בדיקה קשה יותר — צריך להקים היררכיה שלמה כדי לבדוק את הזרימה.

מקור: Behavioral-Design-Patterns.pdf, Exercise12-DesignPatternsStrategyTemplateIterator.pdf

Iterator

מטרה: לספק דרך אחידה לעבור על אוסף ברצף, בלי לחשוף את הייצוג הפנימי שלו. דוגמאות: איטרטורים של STL, לולאות range-based, cursors של מסד נתונים.

המשתתפים

  • Iterator — ממשק המעבר: first(), next(), isDone(), currentItem() (או בגרסה מקוצרת hasNext() / next()).
  • ConcreteIterator — מממש את המעבר ושומר את מיקום הסמן על אוסף קונקרטי. מצב הסמן שלו הוא מה שמאפשר מספר מעברים במקביל.
  • Aggregate — ממשק האוסף עם createIterator() (מפעל שמחזיר איטרטור).
  • ConcreteAggregate — האוסף הממשי; createIterator() מחזיר ConcreteIterator קשור אליו.

דוגמה מהעולם: רשימת השמעה (playlist)

אוסף Playlist (ConcreteAggregate) חושף createIterator(). הלקוח עובר על השירים דרך hasNext()/next() בלי לדעת אם מאחורי הקלעים יש מערך, רשימה מקושרת או עץ. אפשר אפילו איטרטורים שונים לסדרי מעבר שונים (רגיל / ערבוב / א"ב), ושני איטרטורים בו-זמנית על אותה רשימה.

struct Iterator {                        // Iterator
  virtual bool hasNext() = 0;
  virtual string next() = 0;
};
class NameRepository {                    // ConcreteAggregate
  vector<string> names{"Ann","Bob","Cid"};
public:
  Iterator* createIterator();
  friend class NameIterator;
};
class NameIterator : public Iterator {   // ConcreteIterator — שומר מיקום
  NameRepository& repo; size_t i = 0;
public:
  NameIterator(NameRepository& r):repo(r){}
  bool hasNext() override { return i < repo.names.size(); }
  string next()   override { return repo.names[i++]; }
};
// שימוש:
for (Iterator* it = repo.createIterator(); it->hasNext(); )
  cout << it->next();

יתרונות: (1) מפשט את ממשק האוסף — האוסף לא צריך להעמיס עליו פעולות מעבר. (2) מספר מעברים בו-זמנית על אותו אוסף (כל איטרטור עם סמן משלו). (3) מעבר אחיד על מבנים שונים (מערך, עץ, גרף) דרך אותו ממשק.

חסרונות

  • פסילת איטרטור (invalidation) — שינוי האוסף באמצע מעבר עלול להפוך את הסמן ללא-תקף; פתרונות: robust iterators, copy-on-write, גרסאות (versioning) או fail-fast.
  • תקורה מול range-for — כבד ומרובה boilerplate עבור אוספים פשוטים בגודל קבוע, לעומת לולאת range-based native ב-C++ מודרני.
  • בקרת לולאה ידנית — איטרטור חיצוני (external) דורש שהלקוח ינהל בעצמו את הלולאה; קל לשכוח תנאי עצירה.
  • חשיפת פנים — איטרטור שמחזיר Node* חושף מבנה פנימי ומהדק צימוד; קשה גם לייעל (אין גישה מקבילית/עצלה).

העשרה — external מול internal: ב-external iterator הלקוח שולט בקצב (next()); ב-internal iterator האיטרטור עצמו מריץ את הלולאה ומקבל פונקציה (כמו std::for_each). קיים גם NullIterator ש-isDone() שלו תמיד true — שימושי לטיפול אחיד בעלים בעץ (Composite).

מקור: Behavioral-Design-Patterns.pdf, Exercise12-DesignPatternsStrategyTemplateIterator.pdf

Command

מטרה: לעטוף בקשה כאובייקט — מאפשר לפרמטר בקשות, לתייר/לתעד אותן ולתמוך בפעולות ניתנות לביטול (undo/redo). דוגמאות: פעולות GUI, תורי משימות, הקלטת מאקרו. (שאלת מבחן Q20.)

המשתתפים

  • Command — הממשק עם execute() (ולרוב גם undo()).
  • ConcreteCommand — קושר בין Receiver לפעולה; execute() מאציל ל-receiver->action(), ושומר מצב לצורך undo().
  • Invoker — מפעיל את הפקודה (למשל כפתור בסרגל כלים); מחזיק את הפקודה וקורא ל-execute() בלי לדעת מה היא עושה.
  • Receiver — מי שיודע לבצע את העבודה בפועל (כל מחלקה יכולה להיות Receiver).
  • Client — יוצר את ה-ConcreteCommand ומקשר לו את ה-Receiver.

דוגמה מהעולם: כפתור Copy בעורך טקסט

כפתור בסרגל הכלים הוא ה-Invoker; לחיצה מפעילה CopyCommand (ConcreteCommand) שקורא ל-Document (Receiver). כל פקודה שבוצעה נדחפת למחסנית היסטוריה (history stack) — לכן אפשר undo/redo מרובה-רמות. אותה תבנית מאפשרת גם תור פקודות ולוג/סריאליזציה של בקשות.

struct Command {                          // Command
  virtual void execute() = 0;
  virtual void undo() = 0;
};
class Document {                          // Receiver
public: void copy(){ /* עבודה אמיתית */ } void paste(){}
};
class CopyCommand : public Command {      // ConcreteCommand
  Document* doc;                          // מקשר Receiver + פעולה
public:
  CopyCommand(Document* d):doc(d){}
  void execute() override { doc->copy(); }
  void undo()    override { /* שחזור מצב */ }
};
class Button {                            // Invoker
  Command* cmd; vector<Command*> history;
public:
  void click(){ cmd->execute(); history.push_back(cmd); }
  void undoLast(){ history.back()->undo(); history.pop_back(); }
};

יתרונות: מנתק את ה-Invoker מה-Receiver (הכפתור לא יודע מה קורה); מאפשר undo/redo דרך מחסנית, תיור (queue), תיעוד ולוג. MacroCommand היא פקודה מורכבת (Composite) שמחזיקה רשימת פקודות ומריצה את כולן ברצף — וגם מבטלת אותן בסדר הפוך.

חסרונות

  • ריבוי מחלקות קטנות — מחלקת ConcreteCommand לכל פעולה; אפשר למתן זאת עם lambdas.
  • undo() קשה לתופעות לוואי חיצוניות — קל לבטל שינוי בזיכרון, אך קשה לבטל כתיבה לקובץ, שליחת רשת או הדפסה.
  • תקורת מצב לשחזור — לתמיכה ב-undo יש לשמור snapshot של המצב לפני הפעולה, מה שצורך זיכרון.
