מצביעים חכמים (Smart Pointers)

למה מצביעים חכמים?

מצביע חכם (smart pointer) הוא אובייקט העוטף מצביע גולמי (raw pointer) ומנהל את משך החיים של המשאב אוטומטית — בלי delete ידני. הוא מבוסס על עקרון RAII (Resource Acquisition Is Initialization): המשאב משוחרר כשהאובייקט העוטף יוצא מ-scope. כך נמנעות דליפות זיכרון (memory leaks) ומצביעים תלויים (dangling pointers).

ב-C++ המודרני (מ-C++11) מעדיפים מצביעים חכמים על פני new/delete ידני. שלושת הסוגים ב-<memory>: unique_ptr, shared_ptr, weak_ptr.

מקור: Smart-Pointers.pdf

הבעיה: שלוש התקלות של ניהול זיכרון ידני

ב-C++ מקצים זיכרון דינמי עם new ומשחררים אותו עם delete. כל new חייב delete תואם — לא פחות ולא יותר. שלושת סוגי התקלות שנובעים מטעות בכך הם בדיוק מה שמצביעים חכמים באים למנוע:

  • דליפת זיכרון (memory leak): הקצינו זיכרון ושכחנו לשחרר אותו. הזיכרון נשאר תפוס עד שהתוכנית מסתיימת. קורה בקלות ב-early return, כשנזרקת exception באמצע הפונקציה, או כשלא ברור למי אחריות השחרור.
  • מצביע תלוי (dangling pointer): שחררנו את הזיכרון (delete) אבל עדיין מחזיקים מצביע אליו ומנסים להשתמש בו. המצביע מצביע לזיכרון שכבר לא שלנו — התנהגות בלתי מוגדרת (undefined behavior).
  • שחרור כפול (double delete / double free): קוראים ל-delete על אותו כתובת זיכרון פעמיים. גם זו התנהגות בלתי מוגדרת — עלול לקרוס, להשחית את מבני הנתונים של מנהל הזיכרון, או לפתוח פרצת אבטחה. קורה כששני מצביעים חושבים שהם הבעלים של אותו אובייקט וכל אחד משחרר אותו.
void process() {
    int* data = new int[100];   // הקצאה דינמית
    if (error) return;          // דליפה! שכחנו delete
    delete[] data;              // ...ואם נגיע לכאן פעמיים → double delete
}                               // data שנשאר בשימוש אחרי delete = dangling

מצביעים חכמים מבטלים את שלוש התקלות בבת אחת: אין delete ידני (אין דליפה), הבעלות מנוהלת (אין שחרור כפול), והמצביע יודע מתי המשאב כבר לא קיים (אין dangling).

יש שפות שבהן הבעיה נפתרת אחרת: C ו-C++ משתמשות בניהול זיכרון ידני (המתכנת אחראי ל-new/delete), בעוד שפות כמו Java משתמשות בניהול אוטומטי. לכל גישה יתרונות וחסרונות: ידני נותן שליטה מלאה ודטרמיניזם אך פגיע לתקלות; אוטומטי בטוח יותר אך פחות צפוי בתזמון השחרור.

Garbage Collection (GC): מנגנון runtime שמשחרר אוטומטית אובייקטים שלא ניתן עוד להגיע אליהם. ל-Java ו-C# יש GC; ל-C ול-C++ אין — ולכן דווקא שם צריך מצביעים חכמים. הבדל מהותי: מצביע חכם משחרר דטרמיניסטית ביציאה מה-scope, בעוד GC משחרר "מתישהו" בזמן לא צפוי.

unique_ptr — בעלות בלעדית

בעלות בלעדית (exclusive ownership): בעלים יחיד של המשאב. לא ניתן להעתיק — רק להעביר (move). משמיד את האובייקט אוטומטית ביציאה מ-scope. אפס תקורה (zero overhead) לעומת מצביע גולמי.

std::unique_ptr<Widget> w = std::make_unique<Widget>();
auto w2 = std::move(w);   // העברת בעלות; w עכשיו null
// w2 משוחרר אוטומטית בסוף ה-scope

ברירת המחדל כשהבעלות יחידה וברורה. השתמשו ב-make_unique ליצירה.

איך נאכפת הבלעדיות? unique_ptr מוחק (= delete) את שתי הפעולות שהיו משכפלות בעלים: את ה-copy constructor ואת ה-copy assignment operator (operator=). לכן ניסיון להעתיק unique_ptr כלל לא מתקמפל — נשארת רק העברה (move) באמצעות std::move, שמעבירה את הבעלות ומאפסת את המקור ל-null. כך מובטח שבכל רגע יש בדיוק בעלים אחד, ואין סיכון ל-double delete.

מקור: Smart-Pointers.pdf

shared_ptr — בעלות משותפת

בעלות משותפת (shared ownership): כמה בעלים חולקים את אותו משאב באמצעות ספירת הפניות (reference counting). האובייקט מושמד כאשר ה-shared_ptr האחרון משוחרר (הספירה יורדת ל-0). יש תקורה קטנה מעדכוני מונה אטומיים.

auto a = std::make_shared<Node>();
auto b = a;              // ספירה = 2
b.reset();               // ספירה = 1
// המשאב ישוחרר כשגם a ייהרס (ספירה = 0)

שימוש: כשכמה אובייקטים חייבים להחזיק את אותו משאב יחד.

מקור: Smart-Pointers.pdf

weak_ptr — צופה ללא בעלות

צופה ללא בעלות (non-owning observer): מתבונן ב-shared_ptr בלי להעלות את ספירת ההפניות. חייבים לנעול אותו (lock() → shared_ptr) לפני שימוש, מה שגם בודק בבטחה אם האובייקט עדיין קיים.

std::weak_ptr<Node> wp = a;         // לא מעלה ספירה
if (auto sp = wp.lock()) {          // sp הוא shared_ptr תקף
    sp->use();
} // else: האובייקט כבר נהרס

התפקיד המרכזי של weak_ptr: לשבור הפניות מעגליות (circular references) שגורמות לדליפת זיכרון — וכן ל-caches ו-back-pointers.

איך מצביע חכם עובד מבפנים? — SmartPtr<T>

מצביע חכם אינו קסם — הוא מחלקת תבנית (template) שעוטפת מצביע גולמי ומשתמשת בשני מנגנוני C++ קלאסיים כדי "להיראות ולהתנהג" כמו מצביע:

  • העמסת אופרטורים (operator overloading): אנחנו מגדירים מחדש את * ואת -> כדי שהגישה לאובייקט העטוף תיראה בדיוק כמו גישה דרך מצביע רגיל.
  • RAII: ה-constructor מקבל ומחזיק את המצביע הגולמי, וה-destructor קורא ל-delete — כך שהשחרור קורה אוטומטית ביציאה מ-scope.
template <typename T>
class SmartPtr {
    T* pointee;                          // המצביע הגולמי העטוף
public:
    explicit SmartPtr(T* p) : pointee(p) {}   // הבנאי מקבל בעלות
    ~SmartPtr() { if (pointee) delete pointee; } // RAII: שחרור אוטומטי
    T& operator*()  const { return *pointee; }   // גישה לערך
    T* operator->() const { return  pointee; }    // גישה לחברים
};

עכשיו אפשר לכתוב sp->method() או (*sp).field בדיוק כמו עם מצביע גולמי — אבל השחרור מובטח. ההבדל בין סוגי המצביעים החכמים הוא בהחלטה על ההעתקה: להעביר בעלות ולאפס את המקור (unique_ptr), לספור הפניות ולשחרר בספירה 0 (shared_ptr), או לצפות בלי בעלות (weak_ptr).

העשרה: מימוש מצביע חכם כללי לחלוטין קשה — במיוחד לגרום לו להמיר בין טיפוסים לאורך היררכיית ירושה כמו שמצביע גולמי עושה (למשל Derived* ל-Base*), ולתמוך בבדיקת null טבעית. אלה מהחסרונות של מצביעים חכמים: קשים יותר לכתיבה ולהבנה, וההמרה למצביע גולמי בעייתית. לא נדרש למבחן — רק דעו שזה קיים.
