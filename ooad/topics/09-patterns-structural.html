תבניות מבנה — הרכבה

תבניות מבנה (Structural) עוסקות באופן הרכבת מחלקות ואובייקטים למבנים גדולים תוך שמירה על גמישות ויעילות. חמש תבניות בקורס: Composite, Decorator, Proxy, Adapter, Facade.

מקור: Design-Patterns-Structural-Patterns.pdf

Composite

מטרה: להרכיב אובייקטים למבני עץ (part-whole), כך שהלקוח מתייחס לאובייקט יחיד ולהרכב באופן אחיד — אותו ממשק לעלה ולצומת.

class Component { virtual void render() = 0; };
class Leaf : public Component { void render() override; };
class Composite : public Component {
  vector<Component*> children;
  void render() override { for (auto c : children) c->render(); }
};

שימוש: מערכת קבצים, עצי UI, תרשימי ארגון.

מקור: Design-Patterns-Structural-Patterns.pdf

Decorator ו-Proxy

Decorator: להוסיף אחריות לאובייקט דינמית, בזמן ריצה — חלופה גמישה לתת-מחלוק (subclassing). ה-Decorator עוטף את האובייקט ומממש את אותו ממשק. דוגמה: זרמי I/O, רכיבי GUI עם מסגרת/גלילה.

Proxy: «נציג» (surrogate) לאובייקט אחר כדי לשלוט בגישה אליו — טעינה עצלה (lazy loading), caching, logging, בקרת הרשאות. סוגים: virtual proxy, protection proxy, remote proxy.

Decorator מול Proxy: שניהם עוטפים אובייקט עם אותו ממשק — אבל Decorator מוסיף התנהגות/אחריות, ו-Proxy שולט בגישה (בלי לשנות את ההתנהגות עצמה).

מקור: Exercise10-DesignPatternsBuilderDecorator.pdf

Adapter ו-Facade

Adapter: להמיר את הממשק של מחלקה לממשק אחר שהלקוח מצפה לו — מאפשר לממשקים לא-תואמים לעבוד יחד. דוגמה: עטיפת API ישן, התאמת ספריית צד-שלישי לממשק מקומי.

Facade: לספק ממשק פשוט אחד לתת-מערכת מורכבת — מפחית צימוד בין הלקוח לפנימיות תת-המערכת. דוגמה: חזית של קומפיילר, בקר מערכת קולנוע ביתי, API אחיד מעל כמה שירותים.

Adapter מול Facade: Adapter מתאים ממשק קיים אחד לממשק אחר (לתאימות); Facade מפשט גישה לתת-מערכת שלמה מרובת מחלקות.

מקור: Design-Patterns-Structural-Patterns.pdf

Composite — משתתפים, ניהול ילדים ודוגמה מלאה

עד כה ראינו את הרעיון הכללי. עכשיו נפרק את התבנית לתפקידים (participants) — שמות המחלקות שהמבחן עשוי לשאול עליהם — ונראה איך בונים את העץ בפועל.

  • Component — הממשק המשותף שגם לעלה וגם להרכב. מכריז על הפעולה המשותפת (למשל getSize()), ולעיתים מגדיר גישה להורה (parent) והתנהגות ברירת-מחדל.
  • Leaf (עלה) — עצם ללא ילדים; מבצע את הפעולה בעצמו. דוגמה: קובץ בודד.
  • Composite (הרכב/צומת) — מחזיק אוסף ילדים מסוג Component, מממש את פעולות ניהול הילדים (add / remove / getChild), ומאציל את הפעולה המשותפת לכל ילדיו. דוגמה: תיקייה.
  • Client — עובד אך ורק דרך ממשק Component, ולכן לא מבחין בין עלה להרכב.

דוגמה מהעולם האמיתי — מערכת קבצים: FileSystemNode הוא ה-Component; File הוא Leaf; Directory הוא Composite. קריאה ל-getSize() על קובץ מחזירה את גודלו; על תיקייה — סוכמת רקורסיבית את גדלי כל הילדים.

// ניהול הילדים חי על ה-Composite, ופעולת הליבה רקורסיבית
class FileSystemNode { public: virtual int getSize() = 0; };

class File : public FileSystemNode {          // Leaf
  int bytes;
public: int getSize() override { return bytes; }
};

class Directory : public FileSystemNode {     // Composite
  vector<FileSystemNode*> children;
public:
  void add(FileSystemNode* c)    { children.push_back(c); }   // ניהול ילדים
  void remove(FileSystemNode* c) { /* מחיקה מהאוסף */ }
  FileSystemNode* getChild(int i){ return children[i]; }
  int getSize() override {                                    // רקורסיה על העץ
    int sum = 0; for (auto c : children) sum += c->getSize();
    return sum;
  }
};
// Client בונה את העץ דרך אותו ממשק:
Directory* root = new Directory();
root->add(new File());          // עלה בתוך הרכב
Directory* sub = new Directory();
sub->add(new File());
root->add(sub);                  // הרכב בתוך הרכב

העשרה — שקיפות מול בטיחות (GoF): אפשר לשים את add/remove/getChild על ממשק ה-Component עצמו (גישת transparency — הלקוח מטפל בכולם אחיד), או רק על ה-Composite (גישת safety — עלה כלל לא חושף ניהול ילדים). זו התלבטות עיצוב קלאסית של התבנית.

חסרונות של Composite

  • הפרת ISP: אם שמים add/remove/getChild על ממשק הבסיס, כל Leaf נאלץ לממש אותם — בדרך כלל בזריקת חריגה (throw) — כי לעלה אין ילדים. הפתרון (הפרדת ממשקים לעלה ולהרכב) עולה במחיר אובדן האחידות.
  • אובדן בטיחות-טיפוסים (type safety): טיפול אחיד אומר שלקוח שמצפה רק ל-File עלול לקבל Directory. כדי להבחין הוא נדחף ל-dynamic_cast / instanceof — שהמרצה מכנה במפורש «ריח קוד» (code smell).
  • עלות רקורסיה: עצים גדולים צורכים זיכרון, ומעבר רקורסיבי עמוק יקר; גם «בעבוע» אירועים (event bubbling) נעשה מורכב.
  • קושי להגביל תוכן: אין דרך אלגנטית לאכוף ש-Composite יכיל רק סוגים מסוימים של ילדים (למשל «תיקייה שתכיל רק תמונות»). כפייה של Composite על כל היררכיה (ארגון, HTML, JSON) היא שימוש-יתר.

מקור: Design-Patterns-Structural-Patterns.pdf · Exercise9-DesignPatternsAbstractFactory.pdf

Decorator — עוטף שמוסיף אחריות

ה-Decorator מוסיף התנהגות לאובייקט בזמן ריצה בלי לגעת במחלקה המקורית ובלי פיצוץ של תת-מחלקות. הסוד שלו הוא קשר כפול: ה-Decorator גם יורש מ-Component (כדי שיהיה לו אותו ממשק) וגם מחזיק הפניה ל-Component אחר (העצם שהוא עוטף).

  • Component — הממשק המשותף (למשל Beverage עם cost()).
  • ConcreteComponent — האובייקט הבסיסי שנעטף (למשל Espresso).
  • Decorator — מחלקת בסיס מופשטת לעוטפים; מחזיקה הפניה ל-Component (מגיעה בבנאי) ומאצילה אליו את הפעולה.
  • ConcreteDecorator — עוטף קונקרטי שמוסיף מצב (addedState) או התנהגות (addedBehavior) לפני/אחרי ההאצלה (למשל Milk, Sugar).

דוגמה מהעולם האמיתי — קפה ותוספות: קפה בסיס עולה 8₪; עטיפה ב-Milk מוסיפה 2₪, עטיפה ב-Sugar מוסיפה 1₪. כל תוספת היא עוטף עצמאי שאפשר לערום.

class Beverage { public: virtual int cost() = 0; };         // Component
class Espresso : public Beverage {                          // ConcreteComponent
public: int cost() override { return 8; }
};
class AddOn : public Beverage {                             // Decorator מופשט
protected: Beverage* inner;                                 // מחזיק Component
public: AddOn(Beverage* b) : inner(b) {}
};
class Milk : public AddOn {                                 // ConcreteDecorator
public: Milk(Beverage* b) : AddOn(b) {}
  int cost() override { return inner->cost() + 2; }          // מאציל ואז מוסיף
};
class Sugar : public AddOn {
public: Sugar(Beverage* b) : AddOn(b) {}
  int cost() override { return inner->cost() + 1; }
};
// Client עורם עוטפים:
Beverage* order = new Milk(new Sugar(new Espresso()));      // 8+1+2 = 11

העשרה — Java I/O כדוגמה קלאסית: זרמי הקלט של Java הם Decorator. Node Streams (כמו FileInputStream) קוראים ממקור; Processing Streams (כמו BufferedInputStream, DataInputStream) עוטפים אותם ומוסיפים התנהגות. FilterInputStream הוא ה-Decorator הבסיסי, ולכן כותבים new BufferedInputStream(new FileInputStream(f)).

חסרונות של Decorator

  • הרבה מחלקות קטנות: כל תוספת = מחלקה, והדיבאג מסתבך כשיש שכבות עטיפה רבות.
  • הסדר משנה! Bold(Italic(Plain)) אינו בהכרח שווה ל-Italic(Bold(Plain)). עוטף ששַנֶּה סמנטיקה עלול גם להפר LSP. (שאלה אהובה במבחן.)
  • אין האצלה אוטומטית: אין דרך אלגנטית להעביר את כל הקריאות — כל פעולה חייבת להיות מְפוֹרוֶרֶדֶת ידנית. מומלץ לשמור את ממשק ה-Component קטן ככל האפשר.
  • ניהול בעלות (C++): העוטף מחזיק מצביע לעצם הפנימי — יש לנהל בעלות בזהירות (למשל smart pointers) כדי למנוע דליפות.

מקור: Exercise10-DesignPatternsBuilderDecorator.pdf · Design-Patterns-Structural-Patterns.pdf

Proxy — נציג ששולט בגישה

ה-Proxy הוא «נציג» (surrogate) שיושב במקום העצם האמיתי ושולט בגישה אליו — בלי לשנות את ההתנהגות. הלקוח לא יודע שהוא מדבר עם נציג.

  • Subject — הממשק המשותף שגם ה-Proxy וגם העצם האמיתי מממשים (למשל Image עם display()).
  • RealSubject — העצם ה«כבד» האמיתי שמבצע את העבודה (למשל RealImage שטוען תמונה מהדיסק).
  • Proxy — מספק ממשק זהה ל-Subject, מחזיק הפניה ל-RealSubject, ומאציל אליו לאחר שביצע את הבקרה שלו.

דוגמה מהעולם האמיתי — טעינה עצלה של תמונה (virtual proxy): ProxyImage לא טוען את קובץ התמונה הכבד עד שקוראים ל-display() בפעם הראשונה.

class Image { public: virtual void display() = 0; };        // Subject
class RealImage : public Image {                            // RealSubject
  string file;
public: RealImage(string f):file(f){ /* טעינה כבדה מהדיסק */ }
  void display() override { /* מצייר */ }
};
class ProxyImage : public Image {                           // Proxy
  string file; RealImage* real = nullptr;                   // מחזיק RealSubject
public: ProxyImage(string f):file(f){}
  void display() override {
    if (!real) real = new RealImage(file);                  // יצירה עצלה בקריאה ראשונה
    real->display();                                         // מאציל
  }
};

ארבעה סוגי Proxy:

  • Virtual Proxy — דוחה יצירה/טעינה של עצמים כבדים עד שצריך אותם (הדוגמה למעלה).
  • Protection Proxy — מגביל גישה לפי הרשאות.
  • Remote Proxy — נציג מקומי לעצם שנמצא במרחב-כתובות אחר / מעבר לרשת.
  • Smart Reference — תחליף למצביע גולמי שמבצע פעולות נוספות בכל גישה (למשל ספירת הפניות או logging). כאן נסגר המעגל ל-smart pointers מיחידה 1.

חסרונות של Proxy

  • שכבת תיווך נוספת: אינדירקציה נוספת שמקשה על מעקב וניפוי; מנהלים שתי מחלקות (Proxy + RealSubject) במקום אחת.
  • סכנת God Object: נציג שמערבב בקרת-גישה + logging + הרשאות + caching מפר SRP. עדיף לפצל ל-decorators/middleware נפרדים.
  • צימוד הדוק ל-RealSubject: אם ה-Proxy כותב new RealSubject() בפנים — קשה לבדוק ולעשות mock. הפתרון: הזרקת תלות (dependency injection).
  • הנדסת-יתר: לעיתים Decorator/Strategy מספיקים; ו-remote proxy נראה כמו עצם מקומי אך מסתיר השהיית רשת וכשלים — «אנקפסולציה שבורה» של התנהגות המערכת.

מקור: Exercise11-DesignPatternsProxyObserver.pdf · Design-Patterns-Structural-Patterns.pdf

Adapter — מתאם בין ממשקים לא-תואמים

ה-Adapter ממיר את הממשק של מחלקה קיימת לממשק שהלקוח מצפה לו — כמו מתאם תקע חשמלי. כך שתי מחלקות עם ממשקים לא-תואמים יכולות לעבוד יחד.

  • Target — הממשק הספציפי-לתחום שהלקוח משתמש בו (מה שהלקוח מצפה לו).
  • Adaptee — המחלקה הקיימת/החיצונית שרוצים לעשות בה שימוש חוזר, אך הממשק שלה לא מתאים.
  • Adapter — מממש את Target ומתרגם כל קריאה לקריאה מתאימה על ה-Adaptee.
  • Client — עובד מול Target בלבד ולא יודע על ה-Adaptee.

דוגמה מהעולם האמיתי — עטיפת ספריית צד-שלישי ישנה: הקוד שלנו מצפה ל-MediaPlayer.play() (ה-Target), אבל ברשותנו ספרייה ישנה LegacyAudio.startAudio() (ה-Adaptee). מתאם מגשר.

class MediaPlayer { public: virtual void play() = 0; };     // Target
class LegacyAudio {                                         // Adaptee (ממשק שונה)
public: void startAudio() { /* מנגן בדרך הישנה */ }
};
// Object Adapter — הרכבה (composition): מחזיק Adaptee
class AudioAdapter : public MediaPlayer {
  LegacyAudio* adaptee;
public: AudioAdapter(LegacyAudio* a):adaptee(a){}
  void play() override { adaptee->startAudio(); }            // מתרגם קריאה
};

Class Adapter מול Object Adapter:

  • Class Adapter — משתמש בירושה מרובה (יורש גם מ-Target וגם מ-Adaptee). מתחייב ל-Adaptee ספציפי אחד, אך יכול לדרוס את התנהגותו.
  • Object Adapter — משתמש בהרכבה (מחזיק הפניה ל-Adaptee, כמו בקוד). עובד עם ה-Adaptee וכל תת-מחלקותיו, אך קשה יותר לדרוס את התנהגות ה-Adaptee.

יתרונות וחסרונות

  • יתרון: מתאם-אובייקט יחיד יכול לשרת הרבה Adaptees; מוסיפים רק מחלקה אחת חדשה; מקדם OCP (שימוש חוזר בלי לשנות מחלקות קיימות).
  • חיסרון: Class Adapter דורש ירושה מרובה (לא נתמכת בכל שפה); Object Adapter מקשה על דריסת התנהגות ה-Adaptee; שימוש-יתר מוסיף מורכבות (לעיתים עדיף לשכתב את ה-Adaptee עצמו).

העשרה: Pluggable adapters בונים את ההתאמה פנימה ומצמצמים הנחות על הממשק; Two-way adapters מספקים שקיפות לשני כיוונים כדי לשרת שני לקוחות שונים בו-זמנית.

מקור: Design-Patterns-Structural-Patterns.pdf

Facade — חזית פשוטה לתת-מערכת מורכבת

ה-Facade מספק ממשק אחד פשוט מעל תת-מערכת של הרבה מחלקות, ומפחית את הצימוד בין הלקוח לפנימיות. זו בדיוק שאלה 16 במבחן.

  • Facade — יודע אילו מחלקות בתת-המערכת מטפלות בכל בקשה, ומאציל אליהן.
  • Subsystem Classes — המחלקות שעושות את העבודה בפועל. אין להן שום הפניה חזרה ל-Facade — הן ממשיכות לתפקד עצמאית לגמרי.
  • Client — מדבר (בדרך כלל) רק דרך ה-Facade.

דוגמה מהעולם האמיתי — קולנוע ביתי: HomeTheaterFacade.watchMovie() מפעיל בקריאה אחת את המגבר, המקרן ונגן ה-DVD — במקום שהלקוח יכיר ויתאם את שלושתם בעצמו.

// תת-מערכת — מחלקות ללא ידיעה על ה-Facade
class Amplifier { public: void on(){} void setVolume(int){} };
class Projector { public: void on(){} void wide(){} };
class DvdPlayer { public: void on(){} void play(string){} };

class HomeTheaterFacade {                                   // Facade
  Amplifier amp; Projector proj; DvdPlayer dvd;
public:
  void watchMovie(string movie) {                          // ממשק אחד פשוט
    proj.on(); proj.wide();
    amp.on();  amp.setVolume(7);
    dvd.on();  dvd.play(movie);
  }
};
// Client: theater.watchMovie("Inception");

חסרונות של Facade

  • God Object: החזית עלולה לתפוח למחלקת-ענק שיודעת «הכול» ומרכזת יותר מדי אחריות.
  • מסתיר מורכבות שימושית: לקוח מתקדם עלול להזדקק לפעולה עדינה של תת-מערכת שה-Facade לא חושף.
  • תקורת ביצועים: שכבת אינדירקציה נוספת.

העשרה — Facade אינו «חומה»: לקוחות מתקדמים עדיין יכולים לעקוף אותו ולגשת ישירות למחלקות תת-המערכת. אפשר גם להפוך את ה-Facade למופשט עם מימושים שונים, להסתיר מחלקות פנימיות (package-private) מאחורי ממשק ציבורי, ולספק כמה facades (פשוט + עשיר) ללקוחות שונים. Facade גם מפחית תלויות קומפילציה.

כל חמש תבניות המבנה תומכות ב-Open/Closed Principle

עיקרון הפתוח/סגור (OCP) אומר: פתוח להרחבה, סגור לשינוי. כל אחת מחמש התבניות מרחיבה יכולת בלי לגעת בקוד קיים:

  • Adapter — משתמש שוב במחלקה קיימת בלי לשנות אותה.
  • Decorator — מוסיף אחריות דרך עטיפה, לא דרך שינוי המחלקה.
  • Composite — מרחיב את העץ בסוגי צמתים חדשים דרך הרכבה, לא ירושה בקוד הקיים.
  • Proxy — מוסיף בקרה דרך הרכבה, בלי לשנות את ה-RealSubject.
  • Facade — מגן על הלקוח משינויים בתת-המערכת (שינוי פנימי לא דורש שינוי בלקוח).

ארבע מהתבניות נשענות על הרכבה על פני ירושה (composition over inheritance) — עוטפים/מחזיקים עצם במקום לרשת ממנו. זהו הרעיון של שאלה 19 במבחן, והוא מה שהופך אותן לגמישות בזמן ריצה.
