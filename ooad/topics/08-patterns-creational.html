מהי תבנית עיצוב?

תבנית עיצוב (Design Pattern) היא פתרון כללי, מנוסה ורב-שימושי לבעיית עיצוב חוזרת. חשוב: תבנית איננה קטע קוד מוכן להעתקה — היא תיאור של רעיון שאתה מתאים לבעיה שלך. כל תבנית מתעדת שלושה דברים: את הבעיה שהיא פותרת, את הנימוק והפשרות (rationale & trade-offs) — מה מרוויחים ומה מקריבים — ואת הפתרון כנקודת פתיחה שמתאימים, לא כמרשם נוקשה שמעתיקים כלשונו.

המקור: ספר «Gang of Four» — «Design Patterns: Elements of Reusable Object-Oriented Software» מאת Gamma, Helm, Johnson & Vlissides, שיצא לאור ב-1994. הספר מקטלג 23 תבניות בשלוש משפחות: יצירה (Creational), מבנה (Structural) והתנהגות (Behavioral).

תבניות יצירה (Creational) עוסקות באופן יצירת האובייקטים — הן מפרידות את הלקוח (Client) מפרטי היצירה, כך שהקוד לא תלוי במחלקות קונקרטיות ובאופן שבהן הן מכוננות.

העשרה — למה בכלל תבניות? (ארבע התועלות שהמרצה מדגיש)

  • הפשטה (Abstraction): עובדים עם פתרון מוכח בעל שם, במקום להרכיב הכול ממחלקות ומתודות גולמיות.
  • עיצוב טוב יותר (Design): ממחזרים רעיון, לא רק קוד — ונמנעים מלחזור על טעויות ידועות.
  • תקשורת (Communication): שם התבנית הוא אוצר מילים משותף — אומרים «נשתמש כאן ב-Observer» וכולם מבינים מיד את המבנה. עוזר בהצגת מערכת, בלימוד מערכת זרה מהר, ובתכנון קבוצתי.
  • הכשרה (Training): התבניות מפיצות ידע של מומחים מנוסים ומתעדות ניסיון שנצבר בשטח.

רקע היסטורי קצר: רעיון ה«תבנית» הגיע במקור מהאדריכל Christopher Alexander (1977), שהגדיר תבנית כפתרון חוזר וניתן-להתאמה בבנייה; חלוצי התוכנה אימצו אותו והתחילו לתעד ידע-סמוי מתוך מערכות קיימות (למשל ET++, NeXTStep) עד לספר ה-GoF.

מקור: Design-Patterns-Creational.pdf, Exercise8-DesignPatternsSingletonPrototype.pdf

Singleton

מטרה: להבטיח שמופע יחיד (single instance) של מחלקה קיים, ולספק אליו נקודת גישה גלובלית. בקצרה — Singleton הוא המקבילה מונחית-העצמים של משתנה גלובלי, אבל עם גישה מבוקרת (רק דרך מתודה אחת שאתה שולט בה).

המשתתפים (Participants):

  • Singleton — המחלקה עצמה: מחזיקה שדה סטטי פרטי עם המופע היחיד, בנאי פרטי (כדי שאיש לא יוכל ליצור מופע נוסף), ומתודה סטטית ציבורית getInstance() שיוצרת את המופע בפעם הראשונה ומחזירה אותו תמיד.
  • Client — כל קוד שמשתמש ב-Singleton: לעולם לא מכונן אותו עם new, אלא ניגש אליו אך ורק דרך getInstance().

דוגמה מהעולם: מנהל־הגדרות של משחק — GameConfig: קיים בדיוק אחד, וכל מסך (תפריט, שלב, חנות) קורא ממנו את עוצמת הקול והשפה דרך GameConfig::instance(). מקרים נפוצים נוספים: Logger, Config manager, Thread pool — משאב יחיד משותף לכל המערכת.

class Config {                       // ← Singleton
public:
  static Config& instance() {       // ← נקודת הגישה הגלובלית
    static Config c;   // נוצר פעם אחת, thread-safe מ-C++11
    return c;
  }
  Config(const Config&) = delete;   // אין העתקה
private:
  Config() {}                       // בנאי פרטי — אי־אפשר new מבחוץ
};
// Client:
Config::instance().setVolume(80);   // גישה רק דרך instance()

אתחול עצל (lazy initialization): המופע נוצר רק בקריאה הראשונה ל-getInstance, לא לפני. ב-C++ המשתנה הסטטי המקומי (static Config c;) מבטיח יצירה חד-פעמית ובטוחה-לחוטים החל מ-C++11. Singleton יכול גם לנהל מספר מבוקר של מופעים (לא בהכרח אחד), וניתן אף לרשת ולעדן אותו.

למה Singleton נחשב לעיתים אנטי-תבנית — החסרונות:

  • מצב גלובלי מוסתר (hidden global state): מתנהג כמו משתנה גלובלי — כל קוד ניגש אליו בלי שהועבר לו במפורש, וכך נוצרות תלויות סמויות שקשה לראות מחתימת הפונקציה.
  • קשה לבדיקות יחידה (unit-test): אי-אפשר להחליף אותו בקלות ב-mock, המצב שלו נשמר בין בדיקות (אלא אם מאפסים ידנית) והוא מקשה על הזרקת תלות.
  • מפר את SRP: נושא שתי אחריויות — הלוגיקה העסקית שלו וגם ניהול מחזור-החיים שלו עצמו (יצירת המופע היחיד). שאלת מבחן נפוצה: «איזה עקרון SOLID מפר Singleton?» → SRP.
  • מורכבות רב-חוטית: יצירה בטוחה בסביבה מרובת-חוטים דורשת מנגנונים כמו double-checked locking.

שורה תחתונה (המלצת המרצה): העדף הזרקת תלות (Dependency Injection, DI) — העברת האובייקט כפרמטר — על פני Singleton, כדי לחשוף את התלויות ולאפשר בדיקות.

Prototype

מטרה: ליצור אובייקטים חדשים על-ידי שכפול (clone) של אובייקט קיים (ה«אב-טיפוס»), במקום בנייה מאפס. שימושי כשהאתחול יקר, או כדי להימנע מתת-מחלקה נפרדת לכל וריאציה. יתרון מרכזי: הלקוח לא תלוי במחלקה הקונקרטית של המוצר — הוא רק מבקש «שכפל את עצמך».

המשתתפים (Participants):

  • Prototype — הממשק שמכריז על מתודת השכפול clone().
  • ConcretePrototype — מחלקה שמממשת את clone() ומחזירה עותק של עצמה.
  • Client — יוצר מוצרים חדשים בכך שהוא מבקש מאב-טיפוס לשכפל את עצמו, בלי לדעת מהי המחלקה הקונקרטית.

דוגמה מהעולם: עורך גרפי — יש אב-טיפוס של Circle בעל צבע וקו-מתאר מוגדרים; במקום להגדיר את כולם מחדש בכל פעם, המשתמש לוחץ «שכפל» והעורך קורא shape->clone() ומקבל עיגול זהה. דוגמה נוספת: תבנית מסמך (ReportTemplate) שנשמרת במרשם אב-טיפוסים (PrototypeManager / DocumentManager) תחת שם, וכל בקשה חדשה מקבלת עותק משוכפל.

class Shape {                        // ← Prototype
public:
  virtual Shape* clone() const = 0;  // מתודת השכפול
  virtual ~Shape() = default;
};
class Circle : public Shape {        // ← ConcretePrototype
  Shape* clone() const override { return new Circle(*this); }
};
// Client:
Shape* copy = original->clone();     // לא יודע שזה Circle

החסרונות והמלכודות של Prototype:

  • העתקה עמוקה מול רדודה (deep vs shallow copy) — הדאגה המרכזית: אם לאובייקט יש מצביעים/אוספים/אובייקטים מקוננים, העתקה רדודה מעתיקה רק את המצביע → שני העותקים חולקים אותו זיכרון, מה שגורם למצב משותף, שחרור-כפול (double free) ושחיתות. העתקה עמוקה משכפלת גם את התוכן — אך עולה בביצועים, וקשה במיוחד עם הפניות מעגליות.
  • עקביות — יש לאפס שדות ייחודיים: העותק לרוב זקוק לאתחול-מחדש: מזהה (ID) חדש, חותמת-זמן חדשה, ידית מסד-נתונים חדשה — אחרת נשאר מצב ישן ונוצרים באגים לוגיים.
  • הפרת אנקפסולציה: מימוש clone() נכון מחייב להכיר את המבנה הפנימי של האובייקט (איך הוא מורכב), מה ש«חושף» חלק מהמימוש.
  • ניהול זיכרון ב-C++: clone() מחזיר new גולמי — כל עותק חייב להימחק ידנית (delete) או להיעטף במצביע חכם (unique_ptr / shared_ptr), אחרת נוצרת דליפת זיכרון.

מקור: Exercise8-DesignPatternsSingletonPrototype.pdf, Design-Patterns-Creational.pdf

Factory Method

מטרה: להגדיר מתודה ליצירת אובייקט, אך לדחות את ההחלטה איזו מחלקה קונקרטית ליצור אל תת-המחלקות. מחלקת-העל מכריזה על מתודת-יצירה וירטואלית, וכל תת-מחלקה דורסת אותה ומחליטה מה בדיוק ליצור — כך היצירה נעשית פולימורפית.

המשתתפים (Participants):

  • Creator — מחלקה (לרוב אבסטרקטית) שמכריזה על מתודת-היצירה הווירטואלית, למשל createDocument(): Document.
  • ConcreteCreator — תת-מחלקה שדורסת את מתודת-היצירה ומחזירה מוצר קונקרטי מסוים.
  • Product — הממשק המשותף של המוצרים; ConcreteProduct — המימוש הקונקרטי שכל יוצרן מחזיר.

דוגמה מהעולם: אפליקציה שמנהלת מסמכים. Application (Creator) מכריזה על createDocument(); TextApp דורסת ומחזירה TextDocument, SpreadsheetApp מחזירה SpreadsheetDocument. הקוד הכללי ב-Application עובד מול Document בלבד ולא יודע איזה סוג נוצר.

class Application {                          // ← Creator
public:
  virtual Document* createDocument() = 0;   // Factory Method (וירטואלית)
  void newDoc() { Document* d = createDocument(); d->open(); }
};
class TextApp : public Application {         // ← ConcreteCreator
  Document* createDocument() override { return new TextDocument(); }
};

העשרה — «זה בכלל תבנית? זו סתם פולימורפיזם!» ביקורת נפוצה: מתודה וירטואלית שנדרסת — מה החידוש? התשובה: תבניות אינן נמדדות בחדשנות אלא בכך שהן מקטלגות פתרון מוכח לבעיה חוזרת (יצירה בלי כבילה למחלקה קונקרטית). הקריטריון המדויק (שאלת מבחן): זו Factory Method רק כאשר היצירה נדחית לתת-מחלקות באופן פולימורפי — ולא כשמדובר סתם בפונקציה בשם «יפה» שמחזירה new X.

Abstract Factory

מטרה: לספק ממשק ליצירת משפחה של אובייקטים קשורים בלי לציין את המחלקות הקונקרטיות שלהם. בעוד Factory Method יוצרת מוצר אחד, Abstract Factory יוצרת קבוצה שלמה של מוצרים תואמים יחד.

המשתתפים (Participants):

  • AbstractFactory — הממשק שמכריז על מתודת-יצירה לכל סוג מוצר במשפחה (למשל createButton(), createCheckbox()).
  • ConcreteFactory — מממש את הממשק ליצירת משפחה אחת (למשל WinFactory, MacFactory). לרוב מוגדר כ-Singleton — יוצרן אחד למשפחה.
  • AbstractProduct — ממשק לסוג מוצר (Button); ConcreteProduct — המימוש (WinButton, MacButton).
  • Client — מקבל AbstractFactory ומשתמש רק בממשקים האבסטרקטיים; לעולם לא יוצר מוצר קונקרטי ישירות.

דוגמה מהעולם: ערכת רכיבי-ממשק חוצת-פלטפורמות. GUIFactory מכריזה על createButton() ו-createCheckbox(); WinFactory מייצרת WinButton+WinCheckbox ו-MacFactory מייצרת את מקבילות ה-Mac. האפליקציה מקבלת יוצרן אחד לפי מערכת-ההפעלה וכל הרכיבים יוצאים בסגנון עקבי.

struct GUIFactory {                         // ← AbstractFactory
  virtual Button*   createButton()   = 0;
  virtual Checkbox* createCheckbox() = 0;
};
struct WinFactory : GUIFactory {            // ← ConcreteFactory
  Button*   createButton()   override { return new WinButton(); }
  Checkbox* createCheckbox() override { return new WinCheckbox(); }
};
// Client מקבל את היוצרן ולא יודע אם זה Win או Mac:
void buildUI(GUIFactory& f) { f.createButton()->render(); }

יתרונות: החלפת משפחה שלמה בקלות (רק מחליפים את היוצרן), והבטחת עקביות — כל המוצרים שייכים לאותה משפחה ולא מתערבבים.

החסרונות:

  • «פיצוץ יוצרנים» / הפרת OCP+ISP: הוספת סוג מוצר חדש (למשל Slider) מחייבת לשנות את ממשק ה-AbstractFactory ואת כל היוצרנים הקונקרטיים — הפרה של Open/Closed, וגם של ISP (ממשק «שמן» שכופה על יוצרנים לממש מוצרים שאינם צריכים).
  • משפחות נוקשות: אי-אפשר לערבב — למשל MacButton עם WinCheckbox; לפעמים העקביות הכפויה מגבילה מדי.
  • הנדסת-יתר (over-engineering): למקרה של אובייקט אחד-שניים ללא שינוי צפוי — יותר מדי ממשקים ועקיפות מיותרת עבור מה שהוא בסך הכול new Button().
  • היפוך-שליטה מוסתר: היוצרן מסתיר מה נוצר — לוגיקה סמויה שמקשה על ניפוי-שגיאות. חלופה מודרנית: מיכל Dependency Injection / Service Locator.

מקור: Design-Patterns-Creational.pdf, Exercise9-DesignPatternsAbstractFactory.pdf

Builder

מטרה: להפריד את הבנייה שלב-אחר-שלב של אובייקט מורכב מהייצוג הסופי שלו — כך שאותו תהליך בנייה יכול לייצר ייצוגים שונים. למה? כדי להימנע מבנאי-טלסקופ (telescoping constructor) — בנאי ענק עם רשימת פרמטרים ארוכה ומבלבלת כמו new Pizza(cheese, crust, tomato, olives, ...). במקומו קוראים לשלבים ברורים בשמם.

המשתתפים (Participants):

  • Builder — ממשק אבסטרקטי המכריז על מתודות לבניית כל חלק (buildWheels(), buildEngine()…). מתודות ברירת-המחדל יכולות להיות ריקות, כך שכל בונה דורס רק את מה שצריך.
  • ConcreteBuilder — מממש את השלבים ומרכיב את החלקים; מספק getResult() להחזרת המוצר המוגמר. CarBuilder ו-BikeBuilder ממלאים את אותם שלבים אחרת → ייצוגים שונים.
  • Director — הבמאי: מחזיק את סדר השלבים. ה-construct() שלו קורא ל-buildWheels/buildEngine/… על ה-Builder לפי סדר קבוע. חלוקת האחריות: ה-Director אחראי על הרצף, ה-Builder אחראי על החלקים.
  • Product — האובייקט המורכב שנבנה (למשל Vehicle); לרוב אין לו מחלקה אבסטרקטית משותפת כי המוצרים שונים מדי במבנה (זהו החיסרון-המפתח לפי המרצה).

דוגמה מהעולם: בניית רכב. VehicleBuilder מכריז על buildWheels/buildEngine/buildSeats; CarBuilder שם 4 גלגלים ומנוע בנזין, BikeBuilder שם 2 גלגלים ומנוע קטן. ה-Director קורא לשלבים באותו סדר, והלקוח מקבל רכב מוגמר דרך getVehicle() — בלי לדעת את סדר ההרכבה הפנימי.

struct VehicleBuilder {                      // ← Builder
  virtual void buildWheels() {}   // ברירת-מחדל ריקה
  virtual void buildEngine() {}
  virtual Vehicle* getResult() = 0;
};
struct CarBuilder : VehicleBuilder {         // ← ConcreteBuilder
  Vehicle* v = new Vehicle();
  void buildWheels() override { v->wheels = 4; }
  void buildEngine() override { v->engine = "V6"; }
  Vehicle* getResult() override { return v; }
};
struct Director {                            // ← Director (מחזיק את הרצף)
  void construct(VehicleBuilder& b) { b.buildWheels(); b.buildEngine(); }
};
// Client:
CarBuilder cb; Director d; d.construct(cb);
Vehicle* car = cb.getResult();

Abstract Factory יוצרת משפחת מוצרים בבת-אחת; Builder בונה מוצר מורכב יחיד בשלבים. המנגנון המזהה של Builder במבחן: Director + ממשק Builder. יתרונות נוספים: שליטה עדינה בכל שלב, הפרדת-אחריות (תחזוקה קלה), עידוד אי-שינוייות (immutability) ו-ממשקים שוטפים (fluent API) עם קריאות משורשרות.

החסרונות:

  • ריבוי מחלקות: נדרשות Builder + ConcreteBuilder + Director + Product — עומס קוד ניכר.
  • הנדסת-יתר לאובייקטים פשוטים: עבור אובייקט טריוויאלי זה מוגזם ומקשה על מתחילים לעקוב אחר היצירה.
  • לא בטוח-לחוטים (not thread-safe): שרשור a().b().c() אינו אטומי — Builder עם מצב שמשותף בין חוטים עלול להישחת.
  • אין מוצר אבסטרקטי: לרוב אין מחלקת-על משותפת למוצרים כי הם שונים מדי במבנה, וההרכבה עדיין דורשת מישהו (Director/לקוח) לנהל את הרצף.
