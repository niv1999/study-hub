הבחנות בין תבניות דומות

  • Aggregation (◇) מול Composition (◆): ◇ = חלק עצמאי; ◆ = חלק נהרס עם השלם.
  • Decorator מול Proxy: שניהם עוטפים — Decorator מוסיף התנהגות, Proxy שולט בגישה.
  • Adapter מול Facade: Adapter מתאים ממשק אחד לתאימות; Facade מפשט תת-מערכת שלמה.
  • Strategy מול Template Method: Strategy = הרכבה בזמן ריצה; Template Method = הורשה בזמן קומפילציה.
  • Abstract Factory מול Builder: Factory יוצרת משפחת מוצרים; Builder בונה מוצר מורכב יחיד בשלבים.

מקור: OOAD_Course_Summary.pdf

הבחנות בין דיאגרמות UML

  • Sequence מול Communication: אותו מידע — Sequence מדגיש ציר זמן, Communication מדגיש מבנה קישורים (הודעות ממוספרות).
  • Activity מול State: Activity = זרימת תהליך/פעולות; State = מצבי אובייקט ומעברים.
  • Class מול Object: Class = תבנית (טיפוסים, פעולות); Object = תצלום מופעים עם ערכים.
  • Component מול Deployment: Component = רכיבים לוגיים וממשקים; Deployment = חומרה פיזית (nodes).
  • include מול extend: include = תמיד כלול (חובה); extend = מותנה (אופציונלי).

מקור: OOAD_Course_Summary.pdf

מצביעים חכמים — מלכודות

  • unique_ptr אינו ניתן להעתקה — רק std::move.
  • מעגל של shared_ptr דולף — שוברים אותו עם weak_ptr על ה-back-pointer.
  • חייבים lock() על weak_ptr לפני שימוש — הוא עלול להצביע לאובייקט שכבר נהרס.
  • עדיף unique_ptr כברירת מחדל; shared_ptr רק כשצריך בעלות משותפת (יש תקורת מונה אטומי).

מקור: OOAD_Course_Summary.pdf

זיהוי עקרון SOLID

בשאלות מסוג «איזה עקרון מופר» — חפשו את הסימן:

  • מחלקה שעושה יותר מדי דברים → SRP.
  • צריך לשנות קוד קיים כדי להוסיף מקרה → OCP.
  • תת-מחלקה שמשבשת את התנהגות הבסיס → LSP.
  • ממשק ענק עם מתודות לא-רלוונטיות → ISP.
  • תלות ישירה במחלקה קונקרטית במקום בהפשטה → DIP.
