הבחנות בין תבניות דומות

  • Aggregation (◇) מול Composition (◆): ◇ = חלק עצמאי; ◆ = חלק נהרס עם השלם.
  • Decorator מול Proxy: שניהם עוטפים — Decorator מוסיף התנהגות, Proxy שולט בגישה.
  • Adapter מול Facade: Adapter מתאים ממשק אחד לתאימות; Facade מפשט תת-מערכת שלמה.
  • Strategy מול Template Method: Strategy = הרכבה בזמן ריצה; Template Method = הורשה בזמן קומפילציה.
  • Abstract Factory מול Builder: Factory יוצרת משפחת מוצרים; Builder בונה מוצר מורכב יחיד בשלבים.

מקור: OOAD_Course_Summary.pdf

הבחנות בין דיאגרמות UML

  • Sequence מול Communication: אותו מידע — Sequence מדגיש ציר זמן, Communication מדגיש מבנה קישורים (הודעות ממוספרות).
  • Activity מול State: Activity = זרימת תהליך/פעולות; State = מצבי אובייקט ומעברים.
  • Class מול Object: Class = תבנית (טיפוסים, פעולות); Object = תצלום מופעים עם ערכים.
  • Component מול Deployment: Component = רכיבים לוגיים וממשקים; Deployment = חומרה פיזית (nodes).
  • include מול extend: include = תמיד כלול (חובה); extend = מותנה (אופציונלי).

מקור: OOAD_Course_Summary.pdf

מצביעים חכמים — מלכודות

  • unique_ptr אינו ניתן להעתקה — רק std::move.
  • מעגל של shared_ptr דולף — שוברים אותו עם weak_ptr על ה-back-pointer.
  • חייבים lock() על weak_ptr לפני שימוש — הוא עלול להצביע לאובייקט שכבר נהרס.
  • עדיף unique_ptr כברירת מחדל; shared_ptr רק כשצריך בעלות משותפת (יש תקורת מונה אטומי).

מלכודת דיאגרמות: אותה תמונה, שאלה אחרת

שאלות רבות מציגות שתי דיאגרמות דומות ושואלות «מה ההבדל». הטבלה הבאה מפרקת את שלושת הזוגות הקלאסיים — לכל אחד דוגמה קונקרטית:

1. State ≠ Sequence

  • תרשים מצבים (State Diagram) = אובייקט אחד שמתפתח לאורך זמן דרך מצבים (states) ומעברים ביניהם. השאלה שהוא עונה: «באילו מצבים האובייקט הזה יכול להיות?».
    דוגמה: הזמנה יחידה (Order) עוברת New → Paid → Shipped → Delivered. אובייקט אחד, ציר של מצבים.
  • תרשים רצף (Sequence Diagram) = כמה אובייקטים שמחליפים הודעות (messages) לאורך זמן. השאלה שהוא עונה: «מי קורא למי, ובאיזה סדר?».
    דוגמה: תשלום — הדפדפן (Browser) שולח pay() ל-Server, שקורא ל-charge() על PaymentGateway, שמעדכן את ה-DB. שלושה אובייקטים שמדברים.
  • מפריד המבחן: אובייקט אחד שמשנה מצב ⟵ State. כמה אובייקטים ששולחים הודעות ⟵ Sequence.

2. Object ≠ Class

  • תרשים מחלקות (Class Diagram) = המבנה הכללי — שמות טיפוסים, תכונות (attributes) עם טיפוסים, ופעולות (methods). בלי ערכים קונקרטיים.
    דוגמה: תיבת המחלקה Student עם name: String, id: int, והפעולה +enroll().
  • תרשים אובייקטים (Object Diagram) = תצלום מצב (snapshot) ברגע נתון — מופעים ספציפיים עם ערכים ממשיים. שם המופע מסומן עם קו תחתון: dana : Student.
    דוגמה: dana : Student עם name = "דנה", id = 305 — מופע קונקרטי אחד ברגע מסוים.
  • מפריד המבחן: טיפוסים ומבנה כללי ⟵ Class. מופעים עם ערכים ברגע נתון ⟵ Object.

3. Component ≠ Package (וגם ≠ Deployment)

  • תרשים רכיבים (Component Diagram) = יחידות ניתנות-לפריסה בזמן ריצה (runtime) — קבצים כמו .dll / .exe / .jar — עם ממשקים: provided (סוכריה על מקל ⟜○) ו-required (חצי-עיגול ⟜⊃).
    דוגמה: הרכיב PaymentService.dll חושף ממשק provided בשם IPayment, ודורש ממשק required בשם IDatabase.
  • תרשים חבילות (Package Diagram) = קיבוץ לוגי בזמן קומפילציה — namespaces / תיקיות שמארגנות מחלקות. אין כאן ממשקי ריצה, רק תלויות «use» בין חבילות.
    דוגמה: החבילה com.shop.payment שמכילה את המחלקות Invoice, Charge, Receipt — ארגון קוד, לא יחידת פריסה.
  • ולהשלמה — Deployment = חומרה פיזית: צמתים (nodes) כמו שרת ומכשיר, עם ה-artifacts שרצים עליהם.
    דוגמה: הצומת «device» Client PC שעליו רץ ה-artifact browser.exe, מחובר לצומת «server» AppServer.
  • מפריד המבחן: ממשקים ויחידות ריצה ⟵ Component. קיבוץ קוד/namespace בזמן קומפילציה ⟵ Package. חומרה פיזית ⟵ Deployment.

מקור: UML-Diagrams-State-Object-Package-Component.pdf p43

מלכודת תבניות: עוטפים שנראים אותו דבר

ארבע תבניות מבניות (Structural) «עוטפות» אובייקט אחר — וקל להתבלבל ביניהן במבחן. ההבדל הוא בכוונה, לא בקוד. לכל אחת דוגמה מהעולם האמיתי:

Decorator מול Proxy — שניהם עוטפים ומיישמים את אותו ממשק

  • Decorator = מוסיף התנהגות/אחריות חדשה בזמן ריצה, בלי לשנות את האובייקט המקורי.
    דוגמה: כוס קפה — Milk(Sugar(Espresso)). כל עוטף מוסיף מחיר ותיאור. תמיד מפעיל את הפנימי ואז מוסיף משהו.
  • Proxy = שולט בגישה לאובייקט — אותה התנהגות בדיוק, רק מתווך אליה (טעינה עצלה, מטמון, הרשאות, קריאה מרחוק).
    דוגמה: ImageProxy שטוען את התמונה הכבדה מהדיסק רק כשקוראים ל-draw() בפעם הראשונה (lazy loading). אותו ממשק כמו Image, בלי התנהגות חדשה.
  • מפריד: «מוסיף פונקציונליות» ⟵ Decorator. «מסנן/מתזמן את הגישה בלי לשנות מה שקורה» ⟵ Proxy.

Adapter מול Facade — שניהם «נותנים ממשק אחר»

  • Adapter = מתאים ממשק אחד קיים לממשק אחר שהלקוח מצפה לו. עוטף מחלקה אחת כדי לפתור אי-תאימות.
    דוגמה: מתאם שקע — תקע אירופאי לשקע אמריקאי. או XmlToJsonAdapter שעוטף ספריית XML ישנה כדי שתתאים לקוד שמצפה ל-JSON.
  • Facade = מפשט תת-מערכת שלמה — נותן ממשק אחד פשוט מעל הרבה מחלקות מסובכות.
    דוגמה: HomeTheater.watchMovie() שמדליק מגבר, מוריד מסך, מפעיל נגן DVD ומכבה אורות — הלקוח קורא לפעולה אחת במקום לחמש.
  • מפריד: «התאמה של ממשק אחד לא-תואם» ⟵ Adapter. «כיסוי פשוט מעל תת-מערכת מרובת-מחלקות» ⟵ Facade.

סדר ה-Decorator משנה!

  • מכיוון שכל דקורטור עוטף את הקודם, סדר העטיפה משנה את התוצאה.
    דוגמה: Encrypt(Compress(data)) ≠ Compress(Encrypt(data)). דחיסה-ואז-הצפנה עובדת; הצפנה-ואז-דחיסה כמעט לא דוחסת (מידע מוצפן נראה אקראי). אותם עוטפים — סדר הפוך — תוצאה שונה.

סכנת ה-«אובייקט-אל» (God Object) ב-Facade וב-Proxy

  • Facade אמור להאציל (delegate) לתת-המערכת — לא לספוג את הלוגיקה. אם הוא מתחיל להכיל חישובים וכללי-עסק, הוא הופך ל-God Object שמפר SRP. הכלל: «האצל — אל תספוג».
  • Proxy שעושה גם רישום (logging), גם הרשאות (auth) וגם מטמון (caching) בבת-אחת — כבר לא «שולט בגישה» אלא ערבב שלוש אחריות ⟵ הפרת SRP. פרוקסי אמור להיות דק וייעודי.
  • יותר מדי דקורטורים בשרשרת יוצרים מחסנית-קריאות עמוקה שקשה לנפות (debug) — עטיפה של 6 שכבות מקשה על מעקב אחרי מי עשה מה.
  • Adapter כקביים: במערכת חדשה עדיף לתכנן ממשק תואם מראש; שימוש ב-Adapter נכון כשמשלבים קוד קיים/צד-שלישי, לא כתירוץ לתכן ממשק גרוע.

מקור: Design-Patterns-Structural-Patterns.pdf p47

מלכודות תבניות יצירה (Creational)

  • Singleton: כדי למנוע מופע שני חייבים לחסום גם את בנאי-ההעתקה — Singleton(const Singleton&) = delete; וגם את אופרטור ההשמה. אחרת auto s2 = s1; יוצר עותק שני.
  • Prototype (שכפול): העתקה עמוקה (deep copy) של אובייקט עם הפניות מעגליות קשה — עלולים ליפול ללולאה אינסופית או להעתיק לא-נכון. צריך מעקב אחר מה שכבר הועתק.
  • Abstract Factory: הוספת סוג מוצר חדש (לא רק וריאנט) מחייבת שינוי של כל היררכיית המפעלים — בכל מפעל קונקרטי צריך להוסיף פעולה חדשה. זו החולשה המרכזית שלה.
  • Builder: לרוב אין מחלקת-מוצר מופשטת משותפת שימושית — מוצרים שנבנים ב-Builder שונים כל-כך שממשק אב משותף לא נותן ערך.

מקור: Design-Patterns-Creational.pdf p60

מצא את הטעות — קוד תבניות פגום

המבחן והתרגולים מציגים קטעי קוד שגויים ומבקשים לזהות את הבאג. הנה שתי מלכודות קלאסיות:

1. Builder בלי איפוס (reset) — מוצרים «נדבקים» זה לזה:

class CarBuilder {
  Car* car = new Car();          // ❌ נוצר פעם אחת בלבד
public:
  void addWheels() { car->wheels = 4; }
  Car* getResult() { return car; } // ❌ מחזיר את אותו car שוב ושוב
};
// שימוש:
CarBuilder b;
b.addWheels(); Car* c1 = b.getResult();
b.addWheels(); Car* c2 = b.getResult(); // c2 == c1 — אותו אובייקט!

הטעות: ה-Builder לא מאפס את המוצר אחרי getResult(), ולכן בנייה שנייה מזהמת את הראשונה — שני הלקוחות מקבלים אותו אובייקט. התיקון: ליצור car חדש בתחילת כל בנייה (או ב-reset() אחרי החזרה).

2. Decorator שלא מאציל למרכיב שהוא עוטף:

class MilkDecorator : public Beverage {
  Beverage* inner;               // המשקה הנעטף
public:
  MilkDecorator(Beverage* b) : inner(b) {}
  double cost() {
    return 0.5;                  // ❌ מחזיר רק את מחיר החלב
  }                              //    ולא קורא ל-inner->cost()!
};

הטעות: דקורטור חייב להאציל לפנימי ואז להוסיף את שלו: return inner->cost() + 0.5;. בלי הקריאה ל-inner, השרשרת נשברת ומחיר הקפה הבסיסי נעלם. באגים נפוצים נוספים: אי-מימוש ממשק ה-Component, וניהול-בעלות דולף על ה-pointer הפנימי (עדיף unique_ptr).
