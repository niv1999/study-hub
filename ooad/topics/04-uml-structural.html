Object Diagram — תצלום מצב

תרשים אובייקטים (Object Diagram) הוא «תצלום» (snapshot) של המערכת ברגע מסוים — מופעים ממשיים עם ערכים קונקרטיים, לא תבניות מחלקה. קופסת אובייקט: instanceName : ClassName (מודגש/קו תחתון), ומוצגים ערכי התכונות (לא הטיפוסים). בין אובייקטים יש links (לא associations), ואין תא פעולות.

שימוש: להמחיש תרחיש קונקרטי, להבהיר תרשים מחלקות מעורפל, או לנפות מצב אובייקטים (שלב תכן ובדיקה).

דוגמה מהעולם: תצלום של הזמנה בחנות

נניח מחלקה Customer (עם תכונות name, email) ומחלקה Order (עם date, total). תרשים מחלקות יראה את התבנית; תרשים אובייקטים מצלם רגע אחד קונקרטי — עם ערכים אמיתיים:

alice : Customer
name = "Alice Cohen"
email = "alice@shop.com"
—— places ——>
o1 : Order
date = "2026-07-14"
total = 249.90

שימו לב להבדל: ב-Class Diagram כתוב total : double (טיפוס); כאן כתוב total = 249.90 (ערך ממשי). הקו בין alice ל-o1 הוא link — מופע קונקרטי של האסוציאציה «places» מתרשים המחלקות. שם המופע מודגש עם קו תחתון: alice : Customer.

העשרה — הכלי של המרצה (PlantUML): בתרגילים המרצה מצייר תרשימים ב-PlantUML (שפת טקסט שמייצרת תרשים UML). למשל תרחיש מלון: alice : Guest מחוברת ל-b1 : Booking עם status = "confirmed", checkIn = "2026-08-01", וקישורים לחדרים ולפריטי מיני-בר — כל אחד אובייקט עם ערכים משלו.

Package Diagram — ארגון מודולרי

תרשים חבילות (Package Diagram) מציג את המבנה המודולרי ברמה גבוהה — איך המערכת מחולקת לחבילות (namespaces) וכיצד הן תלויות זו בזו. סימון: מלבן עם «לשונית», וחצי תלות «use»/«import». מאפשר הגדרת שכבות (layering) וזיהוי מעגלי תלות.

שימוש: תכן מוקדם — ארכיטקטים מגדירים גבולות מודולים וכיווני תלות מותרים.

מה זו «חבילה»?

חבילה (package) היא בעצם תיקייה של מחלקות — קבוצת מחלקות שקשורות נושאית ומקובצות יחד תחת מרחב-שם (namespace) אחד. בקוד זה מתורגם ל-namespace ב-C++ או package ב-Java. הרעיון: לחלק מערכת גדולה ליחידות מובנות שאפשר להבין ולתחזק בנפרד.

דוגמה קנונית: ארכיטקטורת שכבות

הדוגמה הקלאסית היא מערכת מחולקת ל-4 שכבות, כשכל חבילה תלויה רק בזו שמתחתיה — שרשרת תלות חד-כיוונית:

כלל מבחן (Q34): חץ תלות «use» מ-A ל-B פירושו ש-A תלוי באלמנטים שבתוך B — A משתמש במחלקות של B, לא להפך. אז UI «use» Application אומר: UI מכיר את Application ומשתמש בו; Application לא יודע על UI. הכיוון הוא הכל.

העשרה — לבחור חבילות מתרחיש: בתרגיל מערכת קורסים המרצה פירק ל-6 חבילות לפי תחום: auth (התחברות), user (פרופילים), course (קורסים), content (שיעורים), admin (ניהול), utils (עזר משותף). התלויות מסודרות כך ש-utils לא תלוי באיש, וכולם עשויים לתלות בו. החצים חשובים יותר מהקופסאות — הם מגדירים את הארכיטקטורה. תלות הפוכה שנוצרת בטעות = tight coupling, ומעגל תלות (A→B→A) הוא ריח-רע שיש לשבור.

Component Diagram — רכיבים וממשקים

תרשים רכיבים (Component Diagram) מציג רכיבים לוגיים/פיזיים והממשקים ביניהם — כיצד חלקי מערכת גדולים מתחברים דרך ממשקים.

  • רכיב (component): קופסה עם «component».
  • ממשק מסופק (provided): «סוכריה על מקל» —◯ (lollipop).
  • ממשק נדרש (required): «שקע» —⊂ (socket).

שימוש: שלב תכן — פירוק המערכת לרכיבים ניתנים להחלפה עם חוזי ממשק.

מה זה רכיב, בעצם? רכיב הוא חלק מהמערכת שאפשר לפרוס (deployable) ולהחליף כיחידה עצמאית — שירות, מודול, או מיקרו-שירות. אמירה שימושית: «הרכיבים שלך הם המיקרו-שירותים שלך», ותרשים הרכיבים הוא map של ארכיטקטורת המיקרו-שירותים.

דוגמה מהעולם: מפת מיקרו-שירותים של חנות אונליין

מערכת e-commerce מפורקת לרכיבים, כל אחד נפרס בנפרד ומדבר עם השאר רק דרך ממשקים:

  • Frontend מספק את ה-UI ונדרש (socket ⊂) להתחבר לממשק שה-API Gateway מספק (lollipop ○).
  • Auth Service מספק את הממשק IAuth (התחברות/הרשאות); Payment Service מספק IPay (חיוב).
  • הרכיב עם «database» — Transactions — הוא רכיב-אחסון סטריאוטיפי שמספק IDB.
  • אפשר להחליף את Payment Service בספק אחר בלי לגעת בשאר — כל עוד הוא מספק את אותו IPay. זו עוצמת הרכיבים.

כלל מבחן (Q35): «סוכריה על מקל» —○ (lollipop) = ממשק מסופק (provided) — «אני מציע את השירות הזה». «שקע» —⊂ (socket) = ממשק נדרש (required) — «אני צריך שמישהו יספק לי את זה». שקע שמתחבר לסוכריה = חיבור בין רכיבים.

העשרה — שני סוגי תלות ברכיבים: usage dependency (חץ מקווקו) — רכיב משתמש בממשק שרכיב אחר מספק. realization / implementation — רכיב מממש ממשק (הקו מהרכיב אל ה-lollipop). הראשון = «אני צרכן»; השני = «אני הספק שכתב את הקוד מאחורי הממשק».

Deployment Diagram — פריסה פיזית

תרשים פריסה (Deployment Diagram) מציג את ההפצה הפיזית של התוכנה — אילו artifacts (קבצים/הרצה) רצים על אילו nodes (חומרה), ואת נתיבי התקשורת ביניהם.

  • Node/Device: קופסה תלת-ממדית עם «device».
  • Artifact: קובץ/הרצה שנפרס על node.
  • Communication path: קו עם תווית פרוטוקול.

שימוש: תכן מאוחר ומסירה — ארכיטקטים ו-DevOps מתכננים טופולוגיית פריסה.

דוגמה מהעולם: פריסת אפליקציית ווב

אותה חנות אונליין — אבל עכשיו השאלה היא איפה כל דבר רץ בפועל, על איזו חומרה:

  • Node = חתיכת חומרה (קופסה תלת-ממדית): הדפדפן, שרת הווב, שרת מסד-הנתונים.
  • Artifact = הפריט המוחשי שנפרס על ה-node: shop.war (קובץ ההרצה של האפליקציה) רץ על ה-WebServer; orders.db יושב על ה-DB Server.
  • Communication path = הקו בין nodes, מתויג בפרוטוקול: HTTPS בין הדפדפן לשרת, JDBC בין השרת למסד.

כלל מבחן (Q36): תרשים פריסה ממפה artifacts תוכנתיים אל nodes פיזיים של חומרה — «מה רץ איפה». זה התרשים היחיד שמדבר על חומרה אמיתית; כל השאר לוגיים. אם השאלה מזכירה שרתים/מכשירים/פרוטוקולי-רשת — זה Deployment.

העשרה — פריסה מודרנית (ענן): היום ה-nodes לרוב אינם שרת פיזי אלא משאבי ענן: מכונות וירטואליות ושירותים מנוהלים ב-AWS / Azure / GCP, containers (Docker) ו-pods ב-Kubernetes, ואף nodes מיוחדים עם GPU/TPU להרצת מודלי-AI. נהוג לצייר nodes מקוננים: למשל artifact WebUI.war בתוך «container» Docker, שיושב בתוך «cloud» EC2. סטריאוטיפים נפוצים: «container», «cloud», «database», ונתיבים מתויגים HTTPS / gRPC / JDBC / HLS.

ארבעת התרשימים במבט-על — מתי כל אחד

ארבעת התרשימים בעמוד הזה עונים על ארבע שאלות שונות על אותה מערכת:

  • Object — «איך המערכת נראית ברגע נתון?» תצלום מופעים עם ערכים.
  • Package — «איך המערכת מחולקת למודולים ומי תלוי במי?» גבולות ותלויות.
  • Component — «מאילו יחידות ניתנות-לפריסה היא בנויה ואיך הן מתחברות בממשקים?» מפת מיקרו-שירותים.
  • Deployment — «איפה כל דבר רץ בפועל, על איזו חומרה?» מיפוי artifacts ל-nodes.

העשרה — סדר ניתוח מומלץ: להבנת מערכת קיימת נוח להתחיל «מבחוץ פנימה»: Deployment (איפה זה רץ) ← Component (מאילו שירותים) ← Package (איך הקוד מאורגן) ← ולבסוף Object/State (מה קורה בזמן ריצה). כל תרשים הוא «עדשה» אחרת על אותה מערכת.
