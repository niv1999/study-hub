ספירת הפניות (Reference Counting)

⏱ זמן משוער: 30 דק'

מקור: Reference-Counting.pdfמקור: OOAD_Course_Summary.pdf

איך זה עובד

ספירת הפניות (reference counting) היא טכניקת ניהול זיכרון שבה כל אובייקט שומר מונה של כמה הפניות מצביעות אליו:

  • כל אובייקט מחזיק מונה שלם (reference count).
  • המונה עולה בכל העתקה או השמה חדשה שמצביעה לאובייקט.
  • המונה יורד כשהפניה משוחררת.
  • האובייקט מושמד ברגע שהמונה מגיע ל-0.

זהו המנגנון שמאחורי shared_ptr.

מקור: Reference-Counting.pdf

המוטיבציה: שיתוף ערכים במקום שכפול

מעבר לניהול בעלות, ספירת הפניות פותרת בעיה שנייה — בזבוז זיכרון על ערכים כפולים. נניח חמישה עצמי מחרוזת שכולם מקבלים את אותו ערך:

String a, b, c, d, e;
a = b = c = d = e = "Hello";   // חמש מחרוזות — חמישה עותקים נפרדים של "Hello"

במימוש נאיבי כל אחד מחמשת העצמים מקצה ומחזיק עותק עצמאי של הבתים 'H','e','l','l','o' — פי חמישה זיכרון, ופי חמישה זמן העתקה, לגמרי לחינם. הרעיון: אם לכמה עצמים יש ערך זהה, שכולם יצביעו לייצוג יחיד משותף בזיכרון.

  • הערך המשותף נושא בתוכו שדה refCount — כמה עצמים מפנים אליו כרגע (במקרה שלנו: 5).
  • כשעצם משתחרר, הוא מפחית את refCount. כשהמונה מגיע ל-0, הערך משמיד את עצמו — אין צורך ב"בעלים" חיצוני שיזכור למחוק אותו. זה גם פתרון לבעיית הבעלות (ownership): מי מותר לו לשחרר עצם שהוקצה דינמית? התשובה — העצם מנהל את עצמו.

דוגמה מוחשית: מעבד תמלילים שבו כל מילה במסמך היא עצם String. במסמך של 10,000 מילים המילה "the" חוזרת אלפי פעמים — עם שיתוף ערכים כל המופעים חולקים ייצוג אחד בזיכרון, והמונה שלו הוא מספר המופעים.

מקור: Reference-Counting.pdf

איפה יושב המונה — עם הערך, לא עם העצם

נקודת המפתח במימוש: המונה חייב להיות מוצמד לערך המשותף, לא לכל עצם String בנפרד. אם היינו שמים refCount בתוך String עצמו — היינו סופרים "כמה הפניות לעצם ה-String הזה", וזה תמיד יהיה עצם אחד. אנחנו רוצים לספור "כמה עצמים חולקים את הערך הזה". לכן מגדירים מבנה פנימי נסתר שמחזיק את הנתונים יחד עם המונה:

class String {
  struct StringValue {        // נסתר מהמשתמש — נושא את הערך + המונה
    int   refCount;           // כמה String-ים חולקים אותי
    char* data;               // הבתים עצמם
    StringValue(const char* s): refCount(1) { /* deep-copy של s */ }  // ערך חדש נולד עם 1
    ~StringValue() { delete[] data; }
  };
  StringValue* value;         // כל String מצביע לערך משותף
};
  • ערך חדש נולד עם refCount = 1 (מי שיצר אותו הוא ההפניה הראשונה).
  • בנאי ההעתקה (copy ctor) לא מקצה כלום: הוא רק מעתיק את המצביע value ומגדיל ++value->refCount. אין deep-copy, אין הקצאה — זו כל החיסכון.
  • ההורס (destructor) עושה if (--value->refCount == 0) delete value; — משחרר את הערך רק כשאיש כבר לא מפנה אליו.
  • אופרטור ההשמה operator= חייב קודם להפחית ואולי למחוק את הערך הישן, ורק אז להצביע לערך החדש ולהגדיל את מונהו. בדיקת if (value == rhs.value) return *this; בתחילתו מטפלת גם בהשמה-עצמית (self-assignment).

תובנת מבחן: שלושת הגדולים נעלמים

שאלה קלאסית במבחן. ב-C++, מחלקה שמנהלת משאב גולמי (מצביע raw pointer שהיא new/delete בעצמה) חייבת להגדיר את שלושת הגדולים (The Big Three): בנאי העתקה, אופרטור השמה והורס — אחרת ההעתקה תהיה רדודה ותוביל למחיקה כפולה או לדליפה.

ברגע שמחליפים את המצביע הגולמי בעצם מצביע חכם (למשל value שהוא RCPtr<StringValue> או shared_ptr) — שלושת הגדולים נעלמים לגמרי מהמחלקה. המצביע החכם כבר יודע להעתיק את עצמו (משתף ומגדיל מונה), להשים את עצמו ולהרוס את עצמו. המחלקה String לא צריכה עוד להכריז על אף אחד מהשלושה, והמהדר המחדל שלו עושה בדיוק את הנכון.

יתרון נוסף: המעבר שקוף לחלוטין ללקוח — קוד שמשתמש ב-String לא משתנה בכלל, רק צריך לקמפל מחדש.

Copy-on-Write — שיתוף עד לרגע הכתיבה

אם חמישה עצמים חולקים ערך אחד, ואחד מהם משנה את הערך — הוא ישבש את הערך גם עבור ארבעת האחרים! הפתרון: Copy-on-Write (COW), "העתקה בעת כתיבה". הרעיון פשוט:

  • כל עוד רק קוראים את הערך — כולם ממשיכים לחלוק ייצוג אחד (חינם).
  • ברגע שעצם עומד לכתוב, בודקים: אם refCount > 1 (הערך משותף) — קודם מפצלים עותק פרטי משלנו (deep-copy, מפחיתים את מונה הערך הישן ב-1), ורק אז כותבים לעותק הפרטי. אם אנחנו כבר הבעלים היחידים (refCount == 1) — כותבים ישירות.

הקושי: ב-String, האופרטור operator[] הלא-const אינו יודע להבחין בין קריאה (char c = s[0];) לבין כתיבה (s[0] = 'x';) — הרי שניהם מחזירים הפניה לתא. לכן חייבים להניח שכל גישה לא-const היא כתיבה ולבצע את הפיצול ליתר ביטחון. הגרסה ה-const של operator[], לעומת זאת, בטוחה שהיא קריאה בלבד ולעולם לא מפצלת.

מלכודת ה-aliasing והדגל shareable: אם המשתמש שמר מצביע/הפניה פנימה לערך המשותף — char* p = &s[1]; — ואז מישהו העתיק את s, כתיבה דרך *p = 'x' תשבש את כל שותפי הערך, ואי אפשר לזהות זאת בזמן ההעתקה. התיקון: מוסיפים ל-StringValue דגל bool shareable, מאותחל ל-true; ברגע שגישה לא-const "מדליפה" הפניה החוצה, מסמנים shareable = false לתמיד, ובנאי ההעתקה כבר לא ישתף את הערך הזה אלא יעתיק אותו עמוקות.

העשרה: אותו רעיון בדיוק חוזר במערכות הפעלה (fork() שמשתף דפי זיכרון עד לכתיבה), ובמכונות וירטואליות/מסדי נתונים (snapshots). COW הוא דפוס אופטימיזציה כללי, לא רק תעלול של String.

מכונה לשימוש חוזר: RCObject ו-RCPtr

כדי לא לשכתב את לוגיקת הספירה עבור כל מחלקה מחדש, מפצלים אותה לשני רכיבים לשימוש חוזר:

  • RCObject — מחלקת בסיס שכל ערך שרוצים לספור יורש ממנה. היא מחזיקה את refCount ואת shareable ומספקת addReference() / removeReference(). כש-removeReference מוריד את המונה ל-0 היא מבצעת delete this (הערך משמיד את עצמו). דקויות מבחן: המונה מאותחל ל-0 (היוצר אחראי להעלות אותו), ו-operator= שלה ריק — כי השמה בין שני ערכים משנה את התוכן, לא את מי-שמצביע-עליהם, ואסור לה לגעת ב-refCount של אף אחד מהם.
  • RCPtr<T> — מצביע חכם תבניתי שעוטף T* (כאשר T יורש מ-RCObject) ומבצע את כל תחזוקת המונה אוטומטית: בבנייה/השמה הוא קורא ל-addReference, בהריסה ל-removeReference. הוא מעמיס את operator-> ואת operator* כדי להיראות בדיוק כמו מצביע רגיל. זה בדיוק המצביע החכם שגורם לשלושת הגדולים להיעלם מ-String.

העשרה — כשאי אפשר לגעת בקוד המקור: אם צריך להוסיף ספירת הפניות למחלקה X שאסור/אי-אפשר לשנות (ולכן היא לא יכולה לרשת מ-RCObject), משתמשים ב-RCIPtr<T> ("indirect"): הוא מכניס בין המצביע לערך שכבה נוספת — CountHolder שכן יורש מ-RCObject ומחזיק X*. המונה חי אינדירקציה אחת רחוק יותר, וה-operator->/operator* יכולים לבצע COW אוטומטית (גישה לא-const מפצלת עותק לפני הכתיבה).

הבעיה: הפניות מעגליות

מעגל (cycle): A ← מצביע ל-B ו-B ← מצביע ל-A. גם כשאין הפניות חיצוניות, המונה של כל אחד נשאר 1 — לעולם לא מגיע ל-0, ושני האובייקטים דולפים.

הפתרון: להפוך אחת מההפניות (למשל back-pointer) ל-weak_ptr, שאינו מעלה את הספירה — וכך המעגל נשבר.

כיצד weak_ptr ממומש — מונה שני: בבלוק-הבקרה המשותף (control block) של shared_ptr יש למעשה שני מונים: מונה חזק (strong count) שסופר את ה-shared_ptr-ים שמחזיקים בבעלות, ומונה חלש (weak count) שסופר את ה-weak_ptr-ים הצופים. weak_ptr "עוטף" את ה-shared_ptr ומעלה רק את המונה החלש. כשהמונה החזק מגיע ל-0 העצם מושמד — אך בלוק-הבקרה עצמו נשאר בחיים כל עוד המונה החלש > 0, כדי ש-weak_ptr יוכל לדעת אם העצם עדיין קיים. הקריאה lock() מנסה לשדרג לחזרה ל-shared_ptr: אם העצם עדיין חי — מקבלים אותו ומגדילים את המונה החזק; אם כבר נמחק — מקבלים nullptr. שימוש טיפוסי: cache שרוצה לצפות בפריט בלי למנוע את שחרורו.

מתי ספירת הפניות משתלמת?

ספירת הפניות אינה חינמית — כל ערך נושא מונה (עלות זיכרון), וכל העתקה/השמה/שחרור מעדכנת אותו (עלות זמן). לכן היא משתלמת רק כשהיתרון מכסה את המחיר. שלושת התנאים:

  • ערכים גדולים — ככל שהערך המשותף גדול יותר, כך החיסכון על כל שיתוף גדול יותר (מונה של כמה בתים חוסך מגה-בייט של תמונה).
  • שיתוף רחב — הרבה עצמים חולקים את אותו ערך (כמו המילה "the" במסמך).
  • הרבה העתקות/השמות — פעולות שהיו דורשות deep-copy יקר הופכות להגדלת מונה זולה.

מתי זה מזיק: ערכים קטנים, שכמעט לא משותפים, שממעטים להיות מועתקים — שם תקורת הזיכרון של המונה ותקורת הזמן של עדכונו (במיוחד אטומי בריבוי תהליכונים) עלולות דווקא להאט את התוכנית. כשהבעלות יחידה — unique_ptr עדיף, אפס תקורה.

מקור: Reference-Counting.pdf

Thread safety וביצועים

  • Thread safety: בתוכניות מקביליות עדכוני המונה חייבים להיות אטומיים (atomic). shared_ptr משתמש בהגדלה/הקטנה אטומית של המונה באופן פנימי.
  • מחיר ביצועים: פעולות אטומיות מוסיפות תקורה. כשאין צורך בבעלות משותפת — עדיף unique_ptr (אפס תקורה).
