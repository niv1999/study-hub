יסודות תכנות מונחה-עצמים — ארבעת העמודים (4 Pillars of OOP)

לפני שנדבר על עקרונות עיצוב, צריך את ארבעת עמודי-התווך של תכנות מונחה-עצמים (Object-Oriented Programming, OOP). כל אחד מהם נבחן במבחן:

1. Encapsulation — כימוס (Q29)

כימוס = לארוז יחד את הנתונים ואת המתודות שפועלות עליהם במחלקה אחת, ולהגביל גישה ישירה לנתונים. השדות מוגדרים private, והגישה אליהם עוברת דרך מתודות ציבוריות שמבצעות בקרה/ולידציה.

דוגמה מהעולם: חשבון בנק (BankAccount) מסתיר את היתרה (balance) — אי-אפשר לגעת בה ישירות, רק דרך deposit() ו-withdraw(), שבודקות שהסכום חוקי.

class BankAccount {
  double balance;                        // מוסתר — private
public:
  void deposit(double a){ balance += a; }
  bool withdraw(double a){               // ולידציה מבפנים
    if (a > balance) return false;      // אין משיכה ביתר
    balance -= a; return true;
  }
};

2. Abstraction — הפשטה (Q8)

הפשטה = לחשוף רק את התכונות החיוניות ולהסתיר את המורכבות הפנימית. המשתמש רואה «מה» האובייקט עושה, לא «איך».

דוגמה מהעולם: מכונית (Car) חושפת start() ו-drive() — הנהג לא צריך לדעת דבר על בעירה פנימית, ניצוצות או תזמון שסתומים. המנגנון הפנימי מופשט מאחורי ממשק פשוט.

כימוס מול הפשטה: הפשטה עוסקת בעיצוב — אילו פרטים לחשוף; כימוס עוסק במימוש — כיצד להגן על הפרטים המוסתרים. הם משלימים זה את זה.

3. Inheritance — הורשה (Q40)

הורשה = יחס «הוא-סוג-של» (is-a) שמאפשר שימוש חוזר בקוד: תת-מחלקה יורשת שדות והתנהגות ממחלקת-על ומרחיבה אותם.

דוגמה מהעולם: Dog יורש מ-Animal — כלב הוא חיה, ולכן ניתן להשתמש ב-Dog בכל מקום שמצפים ל-Animal. בהורשה מבטלים גם כפילות: חלקים משותפים עולים למחלקת-העל.

class Animal { public: virtual void eat(); };
class Dog : public Animal { public: void bark(); };  // Dog IS-A Animal

4. Polymorphism — רב-צורתיות (Q21)

רב-צורתיות = ממשק אחד, צורות רבות. אותה קריאה למתודה מפעילה מימוש שונה בהתאם לטיפוס האמיתי של האובייקט בזמן ריצה — זו רב-צורתיות בזמן ריצה (runtime polymorphism), הממומשת ב-C++ באמצעות מתודות virtual ו-override וקישור דינמי (dynamic binding).

דוגמה מהעולם: shape.draw() — אותה שורה מציירת עיגול אם הצורה היא Circle ואת ריבוע אם היא Square. ההחלטה איזה draw() להריץ נופלת בזמן ריצה, לא בזמן קומפילציה.

class Shape  { public: virtual void draw() = 0; };   // ממשק מופשט
class Circle : public Shape { void draw() override; };
class Square : public Shape { void draw() override; };

Shape* s = new Circle();
s->draw();   // בזמן ריצה נקרא draw() של Circle — dynamic binding

קישור דינמי (dynamic / late binding): קביעת המתודה שתופעל נדחית לזמן ריצה לפי הטיפוס בפועל. זה מה שמאפשר רב-צורתיות. בלי virtual ב-C++ הקישור סטטי — נקרא draw() של הטיפוס המוצהר, לא של הטיפוס האמיתי.

מקור: Exercise7-SoftwareDesignAndOOP.ppt

מודל האובייקטים — ארבעת המרכיבים (The Object Model)

מודל האובייקטים (Booch) בנוי על ארבעה מרכיבים עיקריים. שימו לב: כימוס והפשטה חוזרים כאן, ומצטרפים אליהם עוד שניים:

  • Abstraction (הפשטה): מדגישים את המאפיינים החיוניים ומתעלמים משאר הפרטים. דוגמה: Employee מתאר שם ומשכורת, לא את צבע העיניים.
  • Encapsulation (כימוס): מסתירים את פרטי המימוש מאחורי ממשק. דוגמה: Stack חושף push/pop, מסתיר את המערך הפנימי.
  • Modularity (מודולריות): מפרקים מערכת ליחידות עצמאיות ומלוכדות. דוגמה: מודול תשלומים נפרד ממודול המשלוחים בחנות מקוונת.
  • Hierarchy (היררכיה): מסדרים הפשטות בדירוג — הורשה (is-a) והרכבה (has-a). דוגמה: Car יורש מ-Vehicle ומכיל Engine.

מקור: Software-Design.pdf

הרכבה על פני הורשה (Composition over Inheritance) (Q19)

כלל אצבע מרכזי בעיצוב: כשאפשר לבחור, העדיפו הרכבה (has-a) על פני הורשה (is-a).

למה?

  • גמישות רבה יותר: אפשר להחליף את הרכיב המורכב בזמן ריצה (למשל להחליף מנוע), דבר בלתי-אפשרי עם הורשה שנקבעת בזמן קומפילציה.
  • צימוד חלש יותר (less tight coupling): תת-מחלקה תלויה בפרטים הפנימיים של מחלקת-העל; אובייקט מורכב תלוי רק בממשק של הרכיב.
  • נמנעים מהיררכיות עמוקות ושבירות (fragile deep hierarchies): שינוי במחלקת-על גבוהה שובר את כל הצאצאים («בעיית מחלקת-הבסיס השבירה»).

דוגמה מהעולם: מכונית אינה סוג של מנוע, ולכן class Car : public Engine שגוי ומצמיד בחוזקה. הנכון: מכונית מכילה מנוע — Car HAS-A Engine — ואפשר להחליף אותו.

class Car : public Engine { };   // ✗ Car IS-A Engine? לא הגיוני, צימוד חזק
class Car { Engine* engine; };   // ✓ Car HAS-A Engine — גמיש, ניתן להחלפה

מקור: Software-Design.pdf

מושגי יסוד נוספים

  • גנריות (Genericity) / תבניות (Templates): כתיבת קוד שעובד על טיפוסים שונים בלי לשכפל אותו — הטיפוס הוא פרמטר. דוגמה: max<T> אחד עובד ל-int, double ו-string. 
    template<typename T>
T max(T a, T b){ return a > b ? a : b; }   // עובד לכל טיפוס שתומך ב->

העשרה — מורכבות חיונית מול מקרית (Essential vs Accidental Complexity): מורכבות חיונית טבועה בבעיה עצמה (ניהול טיסות באמת מסובך) ואי-אפשר להיפטר ממנה. מורכבות מקרית נובעת מהכלים והפתרון שלנו (קוד מפותל, טכנולוגיה מסורבלת) — ואותה כן אפשר לצמצם. עיצוב טוב מכוון להסיר מורכבות מקרית.

מקור: Software-Design.pdf

למה צריך עיצוב טוב?

עקרונות עיצוב הם הנחיות לבניית תוכנה ניתנת לתחזוקה, גמישה וברת-הרחבה. הם מנחים את החלטות התכן — לפני ולצד בחירת תבנית עיצוב ספציפית. שני מדדי-איכות מרכזיים: coupling (צימוד — כמה מחלקות תלויות זו בזו; רוצים נמוך) ו-cohesion (לכידות — כמה מחלקה ממוקדת בתפקיד אחד; רוצים גבוה).

עיצוב גרוע לא «נשבר» מיד — הוא מתנקם עם הזמן, ככל שהמערכת גדלה ומתחזקים אותה. אלה «ריחות התכן» (design smells) שמסמנים עיצוב רקוב:

  • נוקשות (Rigidity): כל שינוי קטן גורר שרשרת שינויים במקומות רבים.
  • שבירות (Fragility): שינוי במקום אחד שובר קוד במקום אחר, לא-קשור לכאורה.
  • חוסר-ניידות (Immobility): קשה לחלץ רכיב לשימוש חוזר כי הוא כרוך בכל השאר.
  • צמיגות (Viscosity): קל יותר לעשות «טלאי» מלוכלך מאשר את התיקון הנכון.

עקרונות SOLID נועדו בדיוק כדי למנוע את המחלות האלה — ולהשאיר קוד ניתן לתחזוקה, גמיש ובר-הרחבה.

העשרה — עקרונות לחיות לפיהם: «מה שאתה לא יודע לא יכול לפגוע בך» — הסתרת מידע (Information Hiding) היא עמוד-התווך של תכן טוב; אל תשכפל (DRY — Don't Repeat Yourself); קודם שימוש חוזר; ואיכות לפני מהירות. הורשה עצמה היא כלי להסרת כפילות: חלקים משותפים עולים למחלקת-העל.

מקור: Software-Design.pdf

SOLID — חמשת העקרונות

SOLID הוא ראשי-תיבות של חמישה עקרונות עיצוב שנועדו בדיוק כדי למנוע את «ריחות התכן» שלמעלה. במבט מהיר:

  • S — Single Responsibility (אחריות יחידה): למחלקה צריכה להיות סיבה אחת ויחידה להשתנות.
  • O — Open/Closed (פתוח/סגור): פתוח להרחבה, סגור לשינוי — מוסיפים התנהגות בלי לשנות קוד קיים.
  • L — Liskov Substitution (החלפת ליסקוב): תת-טיפוס חייב להיות ניתן להחלפה בטיפוס-העל בלי לשבור את נכונות התוכנית.
  • I — Interface Segregation (הפרדת ממשקים): עדיף כמה ממשקים קטנים וממוקדים מאשר ממשק אחד גדול וכללי.
  • D — Dependency Inversion (היפוך תלות): להסתמך על הפשטות (abstractions), לא על מימושים קונקרטיים. מודולים ברמה גבוהה לא יתלו במודולים ברמה נמוכה.

ההגדרות היבשות נחרתות רק כשרואים את ההפרה ואת התיקון. לכל עיקרון: הקוד הבעייתי, ומולו הקוד המתוקן.

S — Single Responsibility (Q3, Q26)

ההפרה: מחלקת Employee גם מחשבת שכר, גם שומרת ל-DB, וגם מפיקה תלוש PDF — שלוש סיבות שונות להשתנות (שינוי בכללי-שכר / בסכימת-DB / בפורמט-PDF).

class Employee {                 // ✗ שלוש אחריויות
  double calculatePay();         // חוקי שכר
  void   saveToDatabase();       // התמדה
  void   generatePayslipPDF();   // עיצוב תלוש
};

התיקון: אחריות אחת לכל מחלקה — לכידות גבוהה, פחות שבירות.

class Employee { /* נתוני עובד בלבד */ };
class PayCalculator      { double calculate(Employee&); };
class EmployeeRepository { void   save(Employee&); };
class PayslipReporter    { void   toPDF(Employee&); };

O — Open/Closed (Q4)

ההפרה: פונקציה עם if/else על סוג-הצורה — כל צורה חדשה מחייבת לשנות קוד קיים ולבדוק אותו מחדש.

double area(Shape s){                    // ✗ סגור להרחבה
  if      (s.type == CIRCLE) return 3.14*s.r*s.r;
  else if (s.type == SQUARE) return s.side*s.side;
  // Triangle? צריך לחזור ולערוך את הפונקציה הזו...
}

התיקון: מחלקת-בסיס מופשטת עם מתודה virtual שכל צורה דורסת. צורה חדשה = מחלקה חדשה, אפס שינוי בקוד הקיים.

class Shape  { public: virtual double area() = 0; };
class Circle : public Shape { double area() override { return 3.14*r*r; } };
class Square : public Shape { double area() override { return side*side; } };
class Triangle : public Shape { double area() override { return b*h/2; } }; // חדש — בלי לגעת בשאר

L — Liskov Substitution (Q5)

ההפרה: תת-מחלקה שאינה יכולה לכבד את חוזה הבסיס. הסימן הקלאסי במבחן: מתודה דרוסה שזורקת חריגה (throws exception) או «לא עושה כלום». Ostrich יורש מ-Bird אבל לא יכול לעוף — כל קוד שמצפה ל-Bird.fly() יקרוס.

class Bird { public: virtual void fly(); };
class Ostrich : public Bird {
  void fly() override { throw NotSupported(); }  // ✗ מפר LSP — לא ניתן להחלפה
};

התיקון: לתקן את ההיררכיה כך שכל תת-טיפוס יכבד את חוזה הבסיס — למשל להפריד את היכולת «לעוף» לרמה שבה היא באמת נכונה.

class Bird { /* מכנה משותף אמיתי */ };
class FlyingBird : public Bird { public: virtual void fly(); };
class Sparrow : public FlyingBird { void fly() override; };  // עף
class Ostrich : public Bird { /* בלי fly() כלל */ };         // ✓

סימן-היכר למבחן: תת-מחלקה שדורסת מתודה וזורקת ממנה חריגה, מחזירה ערך «ריק», או מחלישה את ההתנהגות — מפרה את LSP. גם Square extends Rectangle שמשבש את setWidth הוא הפרת LSP.

I — Interface Segregation (Q6)

ההפרה: ממשק «שמן» (fat interface) שמכריח מממשים לספק מתודות שאינן רלוונטיות להם. מדפסת ישנה נאלצת לממש scan() ו-fax() שהיא לא תומכת בהם.

class IMachine {                      // ✗ ממשק שמן
public:
  virtual void print() = 0;
  virtual void scan()  = 0;
  virtual void fax()   = 0;
};
class OldPrinter : public IMachine {
  void print() override;                 // תומך
  void scan() override { throw ...; }     // נאלץ לממש בכפייה
  void fax()  override { throw ...; }     // — וזו גם הפרת LSP!
};

התיקון: לפצל לממשקים קטנים וממוקדים; כל מחלקה מממשת רק את מה שהיא באמת צריכה.

class IPrinter { public: virtual void print() = 0; };
class IScanner { public: virtual void scan()  = 0; };
class IFax     { public: virtual void fax()   = 0; };
class OldPrinter : public IPrinter { void print() override; };        // רק הדפסה
class AllInOne   : public IPrinter, public IScanner, public IFax { /*...*/ };

D — Dependency Inversion (Q7, Q26)

ההפרה: מודול ברמה גבוהה (OrderService) יוצר בעצמו (new) ותלוי במחלקה קונקרטית (MySQLDatabase) — אי-אפשר להחליף בסיס-נתונים או לבדוק בבידוד.

class OrderService {
  MySQLDatabase db;                    // ✗ תלות ישירה בקונקרטי
public:
  void save(Order o){ db.insert(o); }
};

התיקון: להגדיר ממשק (IDatabase), שהמחלקה הקונקרטית מממשת, ולהזריק אותו דרך הבנאי (dependency injection). גם הרמה הגבוהה וגם הנמוכה תלויות בהפשטה.

class IDatabase { public: virtual void insert(Order) = 0; };
class MySQLDatabase : public IDatabase { void insert(Order) override; };

class OrderService {
  IDatabase& db;                                 // ✓ תלות בהפשטה
public:
  OrderService(IDatabase& d) : db(d) {}          // הזרקת תלות
  void save(Order o){ db.insert(o); }
};
// בהרכבה: OrderService svc(mysql);  — אפשר להחליף ל-PostgresDatabase בלי לגעת ב-OrderService

מ-SOLID ל-OOP — על מה נשען כל עיקרון

כל עיקרון SOLID נשען על מנגנון OOP קונקרטי:

  • SRP ← לכידות (cohesion) ומודולריות: מחלקה ממוקדת = אחריות אחת.
  • OCP ← הפשטה + רב-צורתיות: מרחיבים דרך מחלקות חדשות שדורסות virtual.
  • LSP ← הורשה נכונה: יחס is-a ששומר על חוזה הבסיס.
  • ISP ← ממשקים (interfaces / abstract classes): ממשקים קטנים וממוקדים.
  • DIP ← הפשטה + כימוס: תלות בממשק, הזרקת המימוש מבחוץ.

מקור: Exercise7-SoftwareDesignAndOOP.ppt

קשר לתבניות עיצוב

SOLID והתבניות עובדים יחד: Open/Closed ו-Dependency Inversion הם הבסיס לרוב התבניות — למשל Strategy (הזרקת אלגוריתם דרך ממשק), Decorator (הרחבה בלי שינוי), ו-Factory (תלות בהפשטה במקום ב-new קונקרטי). התבניות הן «יישום» מוחשי של העקרונות.

מקור: Software-Design.pdf

בדיקה עצמית

מהו העיקרון Single Responsibility?

«פתוח להרחבה, סגור לשינוי» הוא:
