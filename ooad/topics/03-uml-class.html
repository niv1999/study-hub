מהו UML ושתי משפחות הדיאגרמות

UML (Unified Modeling Language) היא שפה גרפית סטנדרטית לתיאור, ניתוח ותכן של מערכות תוכנה. הדיאגרמות מתחלקות לשתי משפחות:

  • דיאגרמות מבנה (structural): Class, Object, Package, Component, Deployment — מה קיים במערכת.
  • דיאגרמות התנהגות (behavioral): Use Case, Sequence, Communication, Activity, State — מה המערכת עושה ואיך.

תרשים מחלקות — המבנה

תרשים מחלקות (Class Diagram) הוא ה«שרטוט» המבני של המערכת — המחלקות, התכונות, הפעולות והקשרים ביניהן. הדיאגרמה הנפוצה ביותר. כל מחלקה מיוצגת בקופסה בת שלושה תאים:

+-------------------------+
|        Circle           |   <- שם המחלקה
+-------------------------+
| - radius : float        |   <- תכונות (attributes)
| - center : Point        |
+-------------------------+
| + area() : double       |   <- פעולות (operations)
| + setRadius(r : float)  |
+-------------------------+

נראוּת (visibility): + public, - private, # protected.

מקור: Exercise4-ClassDiagrams.pdf

קשרים בין מחלקות

חמשת הקשרים המרכזיים (חשוב לזהות את הסימון!):

  • הורשה / הכללה (Generalization): חץ עם משולש ריק △ מהתת-מחלקה אל העל-מחלקה («is-a»).
  • אסוציאציה (Association): קו רגיל — → קשר מבני בין מחלקות («knows-a»).
  • צבירה (Aggregation): מעוין ריק ◇— — «has-a» חלש; החלק יכול להתקיים בלי השלם.
  • הרכבה (Composition): מעוין מלא ◆— — «has-a» חזק; החלק נהרס יחד עם השלם.
  • תלות (Dependency): חץ מקווקו --▶ — שימוש זמני («uses»).

Aggregation מול Composition: ההבדל הוא במשך החיים — בהרכבה (◆) החלק «שייך בלעדית» ונהרס עם השלם (למשל חדרים בבית); בצבירה (◇) החלק עצמאי (למשל שחקנים בקבוצה).

מקור: Exercise4-ClassDiagrams.pdf

ריבוי (Multiplicity)

על קצות האסוציאציה מציינים ריבוי: כמה מופעים משתתפים בקשר.

  • 1 — בדיוק אחד. 0..1 — אפס או אחד (אופציונלי).
  • * או 0..* — אפס או יותר. 1..* — אחד או יותר.

למשל: Order 1 —— * LineItem — הזמנה אחת מכילה מספר שורות.

מי המציא את UML, ומה הוא בעצם

UML נוצר בשנות ה-90 על ידי «שלושת האמיגוס» (the 3 Amigos) — שלושה חוקרים שאיחדו שיטות סימון מתחרות לשפה אחת:

  • Grady Booch — שיטת Booch.
  • Ivar Jacobson — תרם את תרשים ה-Use Case.
  • James Rumbaugh — שיטת OMT.

מאז UML הוא תקן רשמי המנוהל על ידי ארגון התקינה OMG (Object Management Group).

UML הוא שפה, לא תהליך. הוא נותן לך את הכלים (הסימונים) לבטא רעיונות תכן, אבל אינו אומר לך כיצד לנתח או באיזה סדר לעבוד. את שיטת העבודה (למשל Agile או RUP) בוחרים בנפרד. זו הבחנה שנשאלת במבחן.

  • הנדסה קדימה (forward engineering): מייצרים שלד קוד מתוך הדיאגרמה.
  • הנדסה לאחור (reverse engineering): מייצרים דיאגרמה מתוך קוד קיים כדי להבין מערכת.

העשרה — כלל ה-0 > 1: המרצה ממליץ להעדיף תמיד כלי אוטומטי (0 התערבות אנושית) על פני עבודה ידנית (1) — בני אדם טועים, וסנכרון ידני בין קוד לדיאגרמה נשבר מהר.

שלב ותפקיד: תרשים המחלקות משמש לאורך כל הפרויקט — בשלב הניתוח (Analysis) כדי למדל מושגים מהתחום, ובשלב התכן (Design) כדי למדל את מבנה התוכנה. משתמשים בו אדריכלי-תוכנה ומפתחים כאחד.

מקור: UML.pdf, OOAD_Course_Summary.pdf

אנטומיה של קופסת המחלקה — סדר התאים

קופסת המחלקה נקראת תמיד מלמעלה למטה בסדר קבוע של שלושה תאים. שינון הסדר הזה נשאל ישירות במבחן:

סדר התאים: שם המחלקה (Class Name) ← תכונות (Attributes) ← פעולות/מתודות (Methods / Operations). תמיד בסדר הזה.

תחביר מלא של שורה בתא: visibility name : Type = initialValue — כולל ערך התחלתי אופציונלי:

+---------------------------------+
|            Account              |   <- 1. שם המחלקה
+---------------------------------+
| - owner   : string              |   <- 2. תכונות (attributes)
| - balance : double = 0.0        |   <- ערך התחלתי אחרי ה-=
+---------------------------------+
| + deposit(amt : double) : void  |   <- 3. פעולות (operations)
| + getBalance() : double         |
+---------------------------------+

נראוּת (visibility): + ציבורי (public), - פרטי (private), # מוגן (protected).

כשלא מעניינים אותנו הפרטים, מותר לצייר גרסה מקוצרת — קופסה עם שם המחלקה בלבד (ללא תאי תכונות ופעולות).

חמשת הקשרים — דוגמה אמיתית לכל אחד

לכל אחד מחמשת הקשרים יש סימון גרפי ייחודי ומשמעות שונה. הטבלה הבאה נותנת דוגמה קונקרטית מהעולם לכל אחד — זהו הלב של תרשים המחלקות במבחן:

  • הורשה / הכללה (Generalization): חץ עם משולש ריק △ מהתת-מחלקה אל העל-מחלקה, קשר «is-a».
    דוגמה: SavingsAccount ו-CheckingAccount יורשים מ-BankAccount — «חשבון חיסכון הוא חשבון בנק».
  • אסוציאציה (Association): קו רגיל — קשר מבני קבוע בין מחלקות, «knows-a».
    דוגמה: Student —enrolls→ Course — סטודנט רשום לקורס; שניהם קיימים בנפרד אך מכירים זה את זה.
  • צבירה (Aggregation): מעוין ריק ◇ בצד השלם, «has-a» חלש. החלק עצמאי — שורד גם בלי השלם, ואף יכול להיות משותף.
    דוגמה: Department ◇— Professor — פרופסור שייך למחלקה, אך אם המחלקה נסגרת הפרופסור ממשיך להתקיים (ואף יכול להשתייך לשתי מחלקות).
  • הרכבה (Composition): מעוין מלא ◆ בצד השלם, «has-a» חזק ובלעדי. החלק נהרס יחד עם השלם.
    דוגמה: House ◆— Room — חדר הוא חלק בלעדי מבית; אם הבית נהרס, החדרים נהרסים איתו.
  • תלות (Dependency): חץ מקווקו --▶, קשר זמני/חלש «uses». מחלקה אחת משתמשת בשנייה לרגע (פרמטר, ערך מוחזר, משתנה מקומי) בלי להחזיק אותה.
    דוגמה: Order ┈▶ PaymentService — הזמנה קוראת ל-PaymentService.charge() בזמן התשלום, אך אינה מחזיקה אותו כתכונה.

העשרה — מילים נרדפות לצבירה: צבירה לוגית = צבירה משותפת (logical / shared aggregation) — החלק יכול להתקיים בלי השלם. הרכבה = צבירה פיזית = צבירה לא-משותפת (physical / non-shared aggregation). אם המבחן כותב «shared aggregation» הכוונה לצבירה רגילה (מעוין ריק).

הורשה בקוד — BankAccount (דוגמת המבחן)

כך מתרגמים את משולש ההורשה לקוד C++ — כל תת-מחלקה מוסיפה תכונות ופעולות משלה:

class BankAccount {            // על-מחלקה בסיסית
protected:
    string owner;             // בעל החשבון
    double balance;           // יתרה
public:
    virtual void withdraw(double amt); // משיכה
};

class SavingsAccount : public BankAccount { // «is-a» חשבון
    double interestRate;      // ריבית שנתית
public:
    void addInterest();       // צבירת ריבית
};

class CheckingAccount : public BankAccount {
    double overdraftLimit;    // מסגרת אשראי (overdraft)
public:
    bool allowOverdraft(double amt); // האם לאשר משיכת יתר
};

בתרשים, איזה סימון ראש-חץ מציין הורשה (generalization)?

בית מורכב מחדרים; אם הורסים את הבית, החדרים חדלים להתקיים. איזה קשר זה?

עידונים על אסוציאציה

קו האסוציאציה אינו רק קו — אפשר לתלות עליו מידע נוסף. ארבעה עידונים חשובים:

  • שם תפקיד (role name): תווית על קצה הקו שאומרת איזה תפקיד ממלא הצד הזה.
    דוגמה: בין Order ל-Customer — הקצה של הלקוח נושא את התפקיד orderedFor (ההזמנה «הוזמנה עבור» הלקוח).
  • ניווט (navigability): ראש חץ פתוח על קצה האסוציאציה מציין מי «מכיר» את מי.
    דוגמה: Order ——▸ Product — הזמנה מחזיקה הפניה למוצר וניגשת אליו, אך המוצר אינו יודע באילו הזמנות הוא מופיע. בקוד אסוציאציה ממומשת בדרך כלל כמצביע/הפניה (pointer / reference).
  • ריבוי (multiplicity): ראו הסעיף הייעודי — כמה מופעים בכל קצה.
  • אילוץ (constraint): תנאי בסוגריים מסולסלים על הקו.
    דוגמה: {ordered} על קצה של Playlist ——* Song — השירים שמורים בסדר קבוע.

זיקה עצמית (self-association)

מחלקה יכולה להיות בקשר עם עצמה — הקו יוצא מהמחלקה וחוזר אליה, עם שמות תפקיד בשני הקצוות.

  • דוגמה: Employee — עובד מנהל עובדים אחרים: קצה אחד manager, הקצה השני subordinate.
  • דוגמה: Person — קצה אחד husband והשני wife.

מחלקת זיקה (association class)

לפעמים לקשר עצמו יש מידע שאינו שייך לאף אחת משתי המחלקות. אז מצמידים לקו מחלקה בקו מקווקו — היא «מחלקת הזיקה».

  • דוגמה: Student *——* Course. הציון (grade) אינו שייך לסטודנט (יש לו ציון שונה בכל קורס) ולא לקורס — הוא שייך להרשמה. מחלקת הזיקה Enrollment{ grade } נתלית על הקו.
  • דוגמה נוספת: בין Person ל-Company — מחלקת הזיקה Job{ salary } מחזיקה את המשכורת של אותה העסקה.

תכונה או אסוציאציה? אם הערך הוא טיפוס פשוט/פרימיטיבי שאיננו ממדלים (מחרוזת, תאריך, סכום כסף, מספר) — כתבו אותו כתכונה בתוך הקופסה. אם הוא מחלקה אחרת שכבר קיימת בתרשים — קשרו אותה באסוציאציה (בקוד: מצביע/הפניה). זו שאלה קלאסית במבחן.

מקור: UML.pdf, UML-examples-updated.pdf

ממשק ומחלקה מופשטת בתרשים

שני מבנים מיוחדים מסומנים אחרת מהמחלקה הרגילה:

  • ממשק (interface): קופסה עם הסטריאוטיפ «interface» מעל השם. הממשק מגדיר פעולות ללא מימוש.
    דוגמה: «interface» PaymentMethod עם הפעולה pay(amount).
  • מימוש / רֵאליזציה (realization): חץ עם משולש ריק על קו מקווקו (בניגוד להורשה שהיא קו מלא) — «המחלקה מממשת את הממשק».
    דוגמה: CreditCard ┈┈▷ «interface» PaymentMethod.
  • מחלקה מופשטת (abstract class): שם המחלקה נכתב בנטוי (או עם התווית {abstract}). לא ניתן ליצור ממנה מופע ישירות.
    דוגמה: Shape (מופשטת) עם draw() מופשטת, ותת-המחלקות Circle/Square מממשות אותה.
  • סוכריה על מקל (lollipop) —○: עיגול על מקל מציין ממשק מסופק. נפוץ בתרשים רכיבים, אך משמש גם כקיצור לממשק בתרשים מחלקות.

סטריאוטיפ (stereotype) הוא תווית ב-«guillemets» (סוגריים זוויתיים כפולים) שמרחיבה את המשמעות של אלמנט — למשל «interface», «abstract» או «use». זהו מנגנון ההרחבה של UML.

מקור: UML.pdf, Exercise4-ClassDiagrams.pdf

תרשים אובייקטים מול תרשים מחלקות

שתי דיאגרמות דומות ויזואלית אך שונות במהות — הבחנה שנשאלת במבחן:

  • תרשים מחלקות (Class Diagram): מתאר את המבנה הכללי — אילו מחלקות קיימות, תכונותיהן וקשריהן. תבנית, לא נתונים.
  • תרשים אובייקטים (Object Diagram): «צילום מצב» של מופעים קונקרטיים עם ערכים ממשיים ברגע נתון בזמן.

שם המופע נכתב objectName : ClassName (עם קו תחתון), והתכונות מקבלות ערכים:

  תרשים מחלקות (כללי)          תרשים אובייקטים (מופע קונקרטי)
  +----------------+           +--------------------+
  |     Circle     |           |   c1 : Circle      |   <- קו תחתון
  +----------------+           +--------------------+
  | - radius:float |           | radius = 5.0       |   <- ערך ממשי
  +----------------+           +--------------------+

אותה מחלקה Circle יכולה להוליד אינספור אובייקטים — c1 (radius=5), c2 (radius=12) וכו׳.

דוגמה מלאה — תרשים מחלקות של ספרייה

נאסוף הכול לתרשים אחד. תרחיש: ספרייה בנויה ממדורים; חבר יכול להשאיל עד 5 ספרים; לספר יש מחבר (שיכול להשתייך לספרים רבים); ספר הוא פיזי או דיגיטלי. כל דרישה הופכת לקשר:

              +-------------+
              |   Library   |
              +-------------+
                 ◆ 1 | 1..*        הרכבה: Section נהרס עם Library
              +-------------+
              |   Section   |
              +-------------+

 +----------+  borrows      +----------+          +----------+
 |  Member  |1 --------- 0..5|   Loan   |* ----- 1|   Book   |
 +----------+               +----------+          +----------+
 | -name    |               | -dueDate |               △  הורשה (is-a)
 +----------+               +----------+          +----+-----+
                                                  |          |
                                     +----------------+  +-------------+
                                     |  PhysicalBook  |  |    EBook    |
                                     | -shelfLoc:str  |  | -url:string |
                                     +----------------+  +-------------+

 +----------+  writes                +----------+
 |  Author  |◇ 1 ------------------ *|   Book   |    צבירה: Author חי בלי Book
 +----------+                        +----------+     (ומשותף לספרים רבים)

קריאת התרשים — איזו דרישה הפכה לאיזה קשר:

  • הרכבה ◆ — Library ◆— Section: מדור אינו יכול להתקיים בלי הספרייה; אם הספרייה נסגרת, המדורים נעלמים.
  • אסוציאציה + ריבוי — Member 1 — 0..5 Loan: חבר אחד מחזיק בין 0 ל-5 השאלות; הריבוי 0..5 אוכף את «מגבלת 5 הספרים».
  • אסוציאציה — Loan * — 1 Book: כל השאלה מפנה לספר אחד.
  • הורשה △ — PhysicalBook / EBook ▷ Book: שניהם «הם» ספר; הפיזי מוסיף מיקום מדף, הדיגיטלי מוסיף קישור הורדה.
  • צבירה ◇ — Author ◇— Book: מחבר קיים בפני עצמו, יכול להיות משותף לספרים רבים, וממשיך להתקיים גם אם ספר נמחק.

העשרה — מודל מושגי (conceptual model): לפני ציור תרשים המחלקות המפורט, בשלב הניתוח מציירים לעיתים מודל מושגי — מחלקות התחום בלבד (שמות מושגים וקשרים), עדיין בלי טיפוסים, נראוּת או פעולות. הוא ממופה בהמשך לתרשים מחלקות תכן מלא.
