0. מהו Node.js — JavaScript מחוץ לדפדפן

Node.js היא סביבת הרצה (runtime) של JavaScript בצד השרת — כלומר מריצה קוד JS ללא דפדפן. הסינטקס הוא של JavaScript, אבל Node עצמה כתובה ב-C++ ובנויה מעל מנוע V8 של Google Chrome. תפקיד V8 הוא לקחת קוד JavaScript ולתרגם אותו לקוד מכונה שהמחשב יכול להריץ; V8 כתוב ב-C++ לפי סטנדרט ECMAScript והוא open source. Node מוסיפה מעל V8 שכבה המותאמת לצד שרת: טיפול בקבצים, ב-HTTP, ב-streams ובתקשורת רשת — יכולות שאינן קיימות ב-JS של הדפדפן.

שלוש התכונות המרכזיות של Node (למבחן):

  • א-סינכרונית (Asynchronous).
  • Single Thread — חוט ריצה יחיד.
  • Non-Blocking — אינה חוסמת את הריצה כשפעולה לוקחת זמן, אלא ממשיכה וממתינה לאירועים. מתאפשר בזכות events queue, thread pool ו-callbacks.

Callback = פונקציה שתופעל רק כשהפעולה תסתיים / התוצאה תהיה מוכנה. Node ממשיכה לקרוא קוד, וכשהפעולה הושלמה היא "קופצת" ל-callback שהוגדר — וזה מה שמאפשר לחוט יחיד להיות א-סינכרוני.

גרסה — LTS ולא החדשה ביותר. בקורס עובדים עם גרסת LTS = Long Term Support: Node v20.14.0 (קוד "Iron", עם npm 10.8.1) — ולא עם הגרסה החדשה ביותר (22.3.0). כמו כן: לא כל שרת אחסון יכול להריץ Node — בקורס נפרוס ל-Render.

מקור: 9.2_Lec_Node.js (עמ' 4–10)

1. הרצת קוד — REPL ו-node file.js

אחרי התקנת Node (מ-nodejs.org, גרסת LTS), בודקים שהכל תקין מכל תיקייה: 

node -v      # → v20.14.0
npm -v       # → 10.8.1  (npm מותקן יחד עם Node)

ל-Node שני מצבי הרצה. הראשון הוא REPL (Read-Eval-Print-Loop) — סביבה אינטראקטיבית שנכנסים אליה בהקלדת node לבד בטרמינל, ובה אפשר לבדוק שורות קוד תוך כדי כתיבה: 

C:\Users\me>node
Welcome to Node.js v20.14.0.
> const x = 10, y = 20; x + y;
30

המצב השני — הרצת קובץ שלם. יוצרים קובץ (למשל node1.js) עם console.log("Hello world!");, ומהטרמינל — מתוך התיקייה שבה שמור הקובץ — מריצים: 

node node1.js     # → Hello world!

את סימן ה-$ (או >) בתחילת דוגמאות טרמינל לא מקלידים — הוא רק מסמן ששורה זו נכתבת ב-CMD/טרמינל. ולזכור: node file.js מחייב שתהיו בתיקייה של הקובץ.

מקור: 9.2_Lec_Node.js (עמ' 11), 9.3_Tar_Node.js (עמ' 3–5)

2. מערכת המודולים — require ו-module.exports

מודול הוא קובץ (או כמה קבצי) JS שמאוגדים לשימוש חוזר (reusable code). המודולים עצמאיים: קוד ממודול אחד אינו נוגע במודול אחר — רק מה שמייצאים במפורש נגיש מבחוץ.

2.1 require — טעינת מודול

require היא פונקציה גלובלית (מזכירה import / include) שמחזירה reference למודול. הפרמטר היחיד הוא נתיב: 

require("./my_mdl");     // מודול בספרייה הנוכחית
require("../my_mdl");    // מודול בספריית האב
require("./data.json");  // אפשר לטעון גם קובץ JSON
require("express");      // ללא נתיב → חיפוש ב-node_modules

כשקוראים ל-require("express") בלי נתיב, Node מחפש את תיקיית node_modules הקרובה ביותר; אם לא מצא — עולה בהיררכיית התיקיות מעלה ומעלה עד למודולים הגלובליים.

2.2 שתי דרכי ייצוא: module.exports מול exports

module.exports = ... — כותבים כשמייצאים רפרנס יחיד (למשל פונקציה אנונימית אחת): 

// index.js — הקובץ המפעיל
const hotNews = require("./news/index");  // עם או בלי .js
hotNews("Snow in Jerusalem");             // → Hot news: Snow in Jerusalem

// news/index.js — המודול
const prefixMsg = "Hot news: ";
module.exports = msg => console.log(prefixMsg + msg);

exports.name = ... — כותבים כשמייצאים כמה פונקציות בשמות: 

// drive/index.js
const slow = msg => console.log(`Dan said: ${msg}`);
exports.slowly = slow;
exports.fast = msg => console.log(`Dan cried: ${msg}`);

מלכודת מבחן — מה שלא יוצא נשאר פרטי. אם המודול מגדיר פונקציה אך לא מייצא אותה, קריאה אליה מהקובץ המפעיל תיפול: 

// index.js
require("./mdl.js");
news();     // ReferenceError: news is not defined

// mdl.js  (news מוגדרת אך לא מיוצאת → פרטית)
const news = () => console.log("Hello World");

בנוסף: את שורות ה-exports כותבים אחרי כל פונקציה שרוצים לייצא (לא בסוף הקובץ), כדי למנוע בעיות סנכרון.

מקור: 9.2_Lec_Node.js (עמ' 14–16, 33–36), 9.3_Tar_Node.js (עמ' 14–16)

3. מודולים מובנים ואובייקטים גלובליים

חשוב להבחין בין שני סוגי "מובנה", כי זו נקודת מבחן:

  • אובייקטים גלובליים (global objects) — זמינים מכל מקום בלי require. למשל: require, global, process, console, module.
  • מודולים מובנים (built-in modules) — מגיעים עם Node אבל מחייבים require. למשל: fs (קבצים), http (שרת/רשת), path (נתיבים), os, url. 
const os = require("os");   // built-in → חייב require
// process גלובלי → ללא require
const msg = `version: ${process.version}, OS: ${os.type()}`;
console.log(msg);           // → version: v20.14.0, OS: Windows_NT

process הוא אובייקט גלובלי שמציג מידע על התהליך שרץ (הקוד רץ ב-command line, בתוך תהליך — ולא בדפדפן). בנוסף, Buffer הוא סוג משתנה ייחודי ל-Node לאחסון חלקי streams או קבצים.

העשרה — scope הפוך מהדפדפן. בדפדפן, משתנה שלא הוצהר הופך אוטומטית לגלובלי (global scope). Node תוכננה הפוך — לוקאלי כברירת מחדל. זו גם הסיבה שמודולים מותקנים לוקאלית בכל פרויקט: אם פרויקט אחד צריך גרסה 1 של מודול ואחר גרסה 2, לכל פרויקט יהיה עותק משלו, בלי התנגשויות.

מקור: 9.2_Lec_Node.js (עמ' 12, 18–24), 9.3_Tar_Node.js (עמ' 9)

4. npm, package.json ו-node_modules

להבחנה חשובה בשמות: מודולים / חבילות הן ספריות קוד לשיתוף ושימוש חוזר, ואילו npm (Node Package Manager) הוא מנהל ההתקנה שלהן. npm מותקן יחד עם Node. שתי חבילות שנפגוש: Express (מרחיב עבודה עם HTTP) ו-nodemon (מריץ מחדש את השרת אוטומטית בכל שמירת שינוי).

4.1 npm init ו-package.json

כל פרויקט חדש מאותחל ב-npm init — פקודה שיוצרת את הקובץ package.json, "תעודת הזהות + הוראות ההתקנה" של הפרויקט. npm init -y עונה "yes" לכל השאלות. הקובץ מכיל metadata (name, version, author, license), scripts, ורשימות dependencies: 

{
  "name": "init1",
  "version": "1.0.0",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1",
    "dev": "nodemon index.js"
  },
  "author": "Keren",
  "license": "ISC",
  "dependencies": { "express": "^4.19.2" },
  "devDependencies": { "nodemon": "^3.1.3" }
}

package.json הוא JSON חוקי — לא אובייקט JS. מפתחות במרכאות כפולות, בלי פסיק אחרי האיבר האחרון, בלי הערות.

4.2 scripts: npm start מול npm run dev

סקריפט הוא קיצור לפקודה. "start" ו-"test" הם סקריפטים "שמורים" ומורצים ישירות (npm start), אבל סקריפט מותאם אישית כמו "dev" דורש את המילה run: 

npm start        # מריץ את הסקריפט "start"
npm run dev      # מריץ את "dev": "nodemon index.js"

4.3 npm install ו-node_modules

התקנת מודול מורידה אותו לתיקיית node_modules ומוסיפה את התלות ל-package.json. בפעם הראשונה נוצרות גם node_modules וגם package-lock.json (רשימת כל התלויות המדויקת — לא נוגעים בו): 

npm install express            # → dependencies (זמן ריצה)
npm install nodemon --save-dev # → devDependencies (רק פיתוח)
npm install                    # מתקין את כל התלויות של פרויקט קיים

למה מריצים npm install אחרי clone, ולמה node_modules ב-.gitignore? התיקייה node_modules כבדה מאוד וניתנת לשחזור מלא מתוך רשימת ה-dependencies שב-package.json. לכן אף פעם לא מעלים אותה לגיט / לשרת — במקום זאת מעלים רק את package.json, ומי שמקבל את הפרויקט מריץ npm install ומקבל את node_modules מחדש.

dependencies מול devDependencies: ב-dependencies נמצאים מודולים הנחוצים בזמן ריצה (למשל Express, שנקרא ב-require()); ב-devDependencies רק כלי פיתוח (למשל nodemon). דגלים: -D / --save-dev ל-devDependencies, -g / --global להתקנה גלובלית (לשימוש מה-cmd — לא מומלץ כשלא חייבים).

מקור: 9.2_Lec_Node.js (עמ' 22–31), 9.3_Tar_Node.js (עמ' 10–13)

5. שרת HTTP בסיסי ב-Node

אחד השימושים המרכזיים של Node הוא לשמש שרת ווב. המודול המובנה http (הנטען ב-require("http")) מאפשר להקים שרת שמאזין לפורט, מקבל requests ומחזיר responses: 

const http = require("http");
const port = process.env.PORT || 3000;

const server = http.createServer((req, res) => {
  res.writeHead(200, { "Content-Type": "text/plain" });
  res.end("Hello from Node!");
});

server.listen(port, () => console.log(`listening on ${port}`));

השרת מקבל callback שרץ בכל בקשה עם req (הבקשה) ו-res (התשובה). הדפוס process.env.PORT || 3000 — לקחת פורט ממשתנה הסביבה אם הוגדר, אחרת 3000 — יחזור גם בשרתי Express.

העשרה. כתיבת שרת "יד ביד" עם המודול http מסורבלת: צריך לנתב נתיבים ומתודות ידנית, לקבוע כותרות ולפרסר גוף בקשה בעצמנו. בדיוק בשביל זה קיים Express — הוא עוטף את http ומספק routing, middleware ופרסור גוף בשורות קצרות. זה הנושא של היחידה הבאה.

מקור: 9.2_Lec_Node.js (עמ' 8–9, 20) · דוגמת השרת — העשרה

6. Postman — בדיקת ה-API

כשכותבים את צד השרת רוצים לבדוק שהקריאות תקינות — לפני שקיים צד לקוח. Postman הוא כלי לביצוע קריאות לשרת ולצפייה בתשובות, ומשמש כ"לקוח זמני". לכל בקשה קובעים:

  • Method — GET / POST / PUT / DELETE (חייבת להתאים למתודה שהוגדרה ב-route; GET לנתיב שהוגדר POST יחזיר Cannot GET /...).
  • URL / route — הכתובת שאליה פונים.
  • Body — הגוף (למשל raw + JSON) לבקשות POST/PUT.
  • Headers — כותרות (למשל Content-Type, אימות).

התשובה מוצגת עם status (למשל 200 OK), זמן, גודל וגוף ה-JSON. ההיררכיה ב-Postman: workspace → collection → requests → examples. תשובה מוצלחת שומרים כ-Save as example, ומומלץ לשמור גם תשובות ריקות/שגויות. את מסמך ה-documentation (ההוראות לכל בקשה: מתודה, route, תפקיד ולפחות דוגמת תשובה) אפשר לפרסם בכתובת ציבורית ב-Publish.

דגשים מהתרגול: כדי להשתמש ב-View documentation חובה להיות מחוברים ובתוך workspace ("You need to be in a workspace…"). שם הבקשה מתאר מה היא עושה ("Get all posts"), לא את ה-route. ובהגשה — הבקשות חייבות להצביע לשרת המפורסם ב-Render; API ל-localhost לא ייבדק.

מקור: 9.5_Tar_Postman (עמ' 2–13), 9.3_Tar_Node.js (עמ' 6–7, 24–25)
