0. הארכיטקטורה — שלוש השכבות וזרימת הבקשה מקצה לקצה

אפליקציית full-stack מורכבת משלוש שכבות שמדברות זו עם זו: לקוח (client — ה-HTML/CSS/JS שרץ בדפדפן), שרת (server — Node/Express שמאזין לבקשות ומגיש API), ומסד נתונים (DB — כאן MongoDB). בפיתוח תיקיות client ו-server יושבות זו לצד זו, אך בפריסה (deploy) כל אחת יושבת במקום אחר — הלקוח באחסון סטטי והשרת ב-Render.

זרימת בקשה טיפוסית של הוספת רשומה, מקצה לקצה:

Browser (client)  ──fetch(POST)──►  Express server  ──Mongoose──►  MongoDB
      ▲                                    │                            │
      └────────── JSON (עם id) ◄───────────┴──────── document ◄─────────┘

הלקוח שולח fetch אל נתיב ה-API, השרת מתרגם את הבקשה לפעולת CRUD מול מסד הנתונים דרך Mongoose, ומחזיר JSON ללקוח — שמעדכן את ה-DOM. את חלוקת האחריות הזו מתעד המרצה בעזרת מיני-SDD (Software Design Document): שלושה תרשימים — Database design, Use case diagram ו-MVC diagram (Model–View–Controller). כל use case אמור לספק תוצאה אחת במערכת.

מלכודת CORS. הלקוח והשרת חיים ב-origin שונה (למשל דף סטטי מול 127.0.0.1:8081), ולכן השרת חייב להחזיר כותרות CORS כדי שהדפדפן יאשר את הבקשה: Access-Control-Allow-Origin: * ו-Access-Control-Allow-Methods: GET, POST, PUT, DELETE. בלעדיהן הדפדפן חוסם את התשובה — גם אם השרת עצמו החזיר 200 OK.

מקור: 12.1_Lec_Full-Stack (עמ' 12–18), 12.2_Tar_Full-Stack (עמ' 2, 7), 12.4_MongoDB_Mongoose

1. CRUD מלא מול ה-API — מהלקוח לשרת ובחזרה

CRUD = Create / Read / Update / Delete. בתרגיל ה-posts הלקוח מפעיל את כל הפעולות ב-fetch: Read ב-GET /api/posts, Create ב-POST, Delete ב-DELETE /api/posts/:id. בקשת POST חייבת שלושה חלקים: method, כותרת Content-Type: application/json, וגוף מומר במחרוזת: 

fetch("http://127.0.0.1:8081/api/posts", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify(defaultPostBody)
})
.then(response => response.json())
.then(data => addCreatedPostToList(data));

למה תשובת true לא מספיקה. בגרסה הראשונה השרת החזיר בוליאני מ-POST. הבעיה: ללקוח אין את ה-id של הרשומה החדשה, ולכן לא יוכל למחוק/לעדכן/להביא אותה בהמשך. התיקון: השרת בודק response[0].affectedRows === 1, שולף את ה-id החדש מ-response[0].insertId, ומחזיר את הרשומה המלאה ב-res.json(row). עכשיו הלקוח בונה את הקישור עם ה-id האמיתי: post.html?postId=${postData.id}.

שימו לב לזרם ה-Read: .then(response => response.json()) ממיר את גוף התשובה לאובייקט JS, והשרשור השני משתמש בנתונים. response.json() עצמו מחזיר Promise — לכן צריך then נוסף ולא לגשת לנתונים ישירות.

מקור: 12.1_Lec_Full-Stack (עמ' 20–31), 12.2_Tar_Full-Stack (עמ' 5–18)

2. MongoDB — מסד נתונים NoSQL של מסמכים

MongoDB הוא מסד נתונים NoSQL: הנתונים מאוחסנים במסמכים (documents) גמישים דמויי JSON — לא בשורות ועמודות כמו ב-SQL. אין מבנה טבלה קבוע: שני מסמכים באותו אוסף יכולים להכיל שדות שונים לגמרי. הוא בנוי להרחבה אופקית (scale-out) על פני שרתים, ולכן מאורגן סביב אשכולות.

ההיררכיה, מלמעלה למטה:

Cluster  ─►  Database  ─►  Collection  ─►  Document  { name, price, ... }

יצירה אוטומטית. MongoDB יוצר את ה-Database ואת ה-Collection בפעם הראשונה שמוסיפים מסמך — אין צורך להגדיר אותם מראש (או שיוצרים ידנית בממשק Atlas). ב-Atlas: רק כתובות IP שנוספו ל-Access List יכולות להתחבר, וה-connection string חייב להחליף את <db_password> בסיסמה האמיתית (URL-encoded): 

mongodb+srv://<username>:<db_password>@the-vault.xxxx.mongodb.net/?appName=The-Vault

מקור: 12.4_MongoDB_Mongoose (עמ' 2–7)

3. Mongoose — Schema, Model ופעולות CRUD

Mongoose הוא ODM (Object Data Modeler) — שכבה מעל הדרייבר המקורי שנותנת לנתונים צורה וכללים: סכמות, אימות וברירות מחדל מובנים, המרת טיפוסים (casting) ומתודות מודל פשוטות. הדרייבר המקורי (npm install mongodb, new MongoClient) מוצג בקורס להשוואה בלבד — לא בשימוש.

מגדירים Schema ואז יוצרים ממנו Model: 

const productSchema = new mongoose.Schema({
  name:      { type: String,  required: true },
  price:     { type: Number,  required: true, min: 0 },
  category:  { type: String,  enum: ["food", "tech", "toys"] },
  inStock:   { type: Boolean, default: true },
  createdAt: { type: Date,    default: Date.now }
});

module.exports = mongoose.model("Product", productSchema);
  • required: true — השדה חובה; ניסיון לשמור בלעדיו נכשל באימות.
  • default — ערך שנקבע אוטומטית כשלא סופק (כאן inStock=true, createdAt=Date.now).
  • enum — מגביל את הערכים החוקיים לרשימה סגורה, אך אינו הופך את השדה לחובה.

ארבע פעולות ה-CRUD על ה-Model — כולן עם await:

await Product.create({ name: "Tea", price: 5 });   // CREATE
await Product.find({ category: "food" });          // READ
await Product.findByIdAndUpdate(id, { price: 6 }); // UPDATE
await Product.findByIdAndDelete(id);               // DELETE

חיבור אחד לכל האפליקציה. קוראים ל-mongoose.connect(process.env.MONGO_URI) פעם אחת בעליית השרת — לא בכל בקשה. שגיאות ראשוניות נתפסות ב-.catch, ושגיאות ריצה ב-mongoose.connection.on("error", ...).

מקור: 12.4_MongoDB_Mongoose (עמ' 9–13)

4. משתני סביבה ו-.env — ולמה לא מעלים ל-GitHub

משתני סביבה הם זוגות שם/ערך שהאפליקציה משתמשת בהם אך שלא רוצים שיופיעו במפורש בקוד — סיסמאות DB, מפתחות API, ה-connection string. בקוד ניגשים אליהם דרך process.env.NAME, למשל process.env.MONGO_URI או process.env.DB_PASSWORD.

בפיתוח מקומי מרכזים אותם בקובץ .env. ⚠️ לעולם לא מבצעים לו commit ל-Git/GitHub — הוא מכיל סודות; מוסיפים אותו ל-.gitignore ושומרים אותו בנפרד בצורה מאובטחת. גם ב-Atlas ההנחיה מפורשת: "Do not commit this file to version control".

העשרה — מהיכן מגיע process.env? אלו משתנים שהמערכת מזריקה לתהליך Node בזמן ריצה. בענן (Render) מגדירים אותם בממשק ולא בקובץ; בפיתוח מקומי נהוג לטעון .env עם החבילה dotenv (require("dotenv").config()), שקוראת את הקובץ וממלאת את process.env. כך אותו קוד עובד גם מקומית וגם בפריסה — משתנים בלבד, לא הקוד, מחליפים ערכים בין הסביבות.

מקור: 12.3_Tar_Render-Deploy (עמ' 13), 12.4_MongoDB_Mongoose (עמ' 5)

5. פריסה ל-Render — deploy של צד השרת

Render הוא שירות חינמי (לצרכי הקורס) ל-deploy של צד השרת. נרשמים עם חשבון GitHub, בוחרים Web Services, ומקשרים ריפו — כך שכל push לברנץ' הראשי מפעיל deploy אוטומטי של הקוד העדכני (קיים גם Manual Deploy). מאפייני ההעלאה:

  • Build Command: npm install — מתקין תלויות.
  • Start Command: הרצת השרת (למשל node index.js).
  • Runtime: Node (מזוהה אוטומטית), Instance Type: Free (0$, 512MB RAM, 0.1 CPU).
  • Environment Variables / Secret Files: כאן מגדירים את הסודות (במקום להעלות .env).

PORT מגיע מ-process.env. Render מקצה את הפורט דרך משתנה הסביבה PORT (בפועל 10000). לכן אסור לקבע פורט; כותבים: 

const port = process.env.PORT || 8080;
app.listen(port);

כך בענן משתמשים בפורט של Render, ומקומית נופלים ל-8080. בלוגים רואים "Detected service running on port 10000 — Your service is live", וכתובת השירות היא https://<service-name>.onrender.com.

Spin down. בגרסה החינמית השרת "נרדם" אחרי חוסר פעילות, והבקשה הראשונה יכולה להתעכב ב-~50 שניות עד שהוא מתעורר. בנוסף: deploy יכול "להצליח" והשירות live — אך חיבור ה-DB נכשל אם משתנה הסביבה חסר. בדיוק זה קורה בשגיאת הלוגים: MongooseError: The 'uri' parameter to 'openUri()' must be a string, got "undefined" — כי process.env של ה-URI לא הוגדר ב-Render.

מקור: 12.3_Tar_Render-Deploy (עמ' 2–13), 12.1_Lec_Full-Stack (עמ' 5)

6. async/await נכון — לא לערבב עם then

פעולות מסד הנתונים אסינכרוניות, ולכן כולן עם async/await. המרצה מדגים בכוונה קוד שמערבב await עם שרשור then — כדי להראות ששתי הצורות שקולות — אבל מסמן זאת כאנטי-דפוס: 

async addPost(req, res) {
  const connection = await dbConnection.createConnection();
  await connection
    .execute(`INSERT INTO tbl_99_posts ...`)
    .then(response => {                       // ⚠️ ערבוב await + then
      if (response[0].affectedRows === 1) {
        getPostById(response[0].insertId)
          .then(row => res.json(row));        // ⚠️ then מקונן
      }
    });
}

הכלל של המרצה: בחרו שיטה אחת — עדיף async/await — ואל תשרשרו הרבה רמות של אסינכרוניות (כמו 5 then מקוננים). זה עובד גם מעורבב, אבל פוגע בקריאוּת ובניהול שגיאות. הגרסה הנקייה: const [rows] = await connection.execute(...) ואז res.json(rows[0]), בתוך try/catch.

מקור: 12.1_Lec_Full-Stack (עמ' 28–30), 12.2_Tar_Full-Stack (עמ' 13)
