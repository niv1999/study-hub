0. מהו Express — והרמת שרת ב-3 שורות

Express הוא, בלשון התיעוד הרשמי, "a minimal and flexible Node.js web application framework" — framework מינימלי וגמיש שרץ מעל Node.js. בפועל הוא מודול קליל שעוטף את מודול ה-http המובנה של Node ומאפשר לנהל בקלות נתיבים (Routes), בקשות ותשובות (Requests / Responses), Cookies ועוד — במקום לכתוב הכול ידנית מעל http.

מתקינים אותו כמו כל חבילה: npm i express (החבילה נרשמת ב-dependencies של package.json). הרמת שרת דורשת שלוש שורות בלבד: לייבא, ליצור מופע, ולהאזין ל-port: 

const express = require('express');
const app = express();

app.listen(8080);

השורה app.listen(8080) מקשרת (binds) חיבור HTTP ל-port. זהו בדיוק אותו חיבור שהיה נוצר עם server.listen() של מודול http — למעשה app.listen(8080) שקול ל-http.createServer(app).listen(8080).

בפרקטיקה כמעט תמיד לוקחים את ה-port ממשתנה סביבה, כדי שאותו קוד ירוץ גם מקומית וגם בענן: 

const port = process.env.PORT || 8080;
app.listen(port);
console.log(`listening on port ${port}`);

לזכור: const app = express(); יוצר את האפליקציה, app.listen(port) מרים את השרת. express אינו מחליף את Node — הוא רץ מעליו ועוטף את http.

מקור: 10.1_Lec_Express (עמ' 4–7), 11.1_Lec_Node_and_Client (עמ' 17)

1. Routing — app.get / post / put / delete

Routing הוא ההתאמה בין שיטת HTTP + נתיב לבין פונקציית טיפול (callback). לכל שיטת CRUD יש פונקציה מוכנה, וכולן באותה חתימה: app.method("/path", (req, res) => { … }): 

app.get('/', (req, res) => res.send('Server root'));
app.get('/login', (req, res) => res.send('Login page'));
app.post('/contact', (req, res) => res.json({ firstname: "Alf" }));
app.put('/contact', (req, res) => res.send('Update contact'));
app.delete('/contact', (req, res) => res.send('Delete contact'));

app.all('*', (req, res) => res.send('Global handler for all routes'));

app.all('*', …) תופס כל שיטה וכל נתיב — הוא ה"רשת" האחרונה. העיקרון הקריטי שהמרצה מדגיש:

כל טיפול ב-route הוא כמו if / else. Express נכנס רק ל-route הראשון שמתאים לפי סדר ההגדרה, מריץ אותו — ואם ה-callback שלח תשובה וללא next(), השרשרת נעצרת שם. שאר ה-routes כלל לא נבדקים.

מכאן שאלת המבחן הקלאסית — "מה יודפס?". עבור הקוד למעלה: PUT /contact → "Update contact", אבל PUT /other → "Global handler" (אין ראוט PUT /other, ולכן נופל ל-app.all('*')). וגם PATCH /contact → "Global handler" (אין app.patch מוגדר).

הסדר קובע! אם app.all('*') מוגדר לפני app.get('/'), גם בקשה ל-/ תיתפס ע"י ה-all ותחזיר "Global handler" — כי הוא הראשון שמתאים. החלפת הסדר בין השניים משנה את התוצאה.

מקור: 10.1_Lec_Express (עמ' 9–14)

2. req ו-res — ותשובה אחת בלבד!

ל-callback עוברים תמיד שני פרמטרים בסדר קבוע: req (אובייקט הבקשה — הפרמטר הראשון) ו-res (אובייקט התשובה — השני). דרך res שולחים תגובה ללקוח:

  • res.send(...) — שולח מחרוזת, Buffer, אובייקט או מערך. קובע אוטומטית את ה-Content-Type (אובייקט → application/json).
  • res.json(obj) — ממיר אובייקט JS למחרוזת JSON ושולח, עם Content-Type: application/json.
  • res.status(code) — קובע את קוד הסטטוס. ניתן לשרשר: res.status(200).json({...}) או res.status(500).json({...}).
  • res.redirect(url) — מפנה את הלקוח. הפניה לצמיתות מחייבת סטטוס 301: res.status(301).redirect(...).
  • res.set({...}) — קובע ידנית כותרות HTTP של התשובה.

חובה לשלוח בדיוק תשובה אחת. אחרי res.send()/res.json() התשובה נשלחת וקוד הטיפול לא ממשיך רלוונטית בצד השרת. שתי סכנות למבחן:

  • אפס תשובות — אם ה-callback מסיים בלי לקרוא ל-res.send/json, הבקשה נשארת "תלויה" (pending) והלקוח ממתין עד timeout. Express אינו שולח תשובה אוטומטית.
  • שתי תשובות — קריאה שנייה ל-res.send אחרי שהתשובה כבר נשלחה זורקת "Cannot set headers after they are sent to the client".

חשוב תמיד לשלוח סטטוס — לא רק כשהוא שונה מ-200. משפחות הסטטוסים: 1XX informational, 2XX success, 3XX redirection, 4XX client error, 5XX server error.

מקור: 10.1_Lec_Express (עמ' 25–31), 11.1_Lec_Node_and_Client (עמ' 6, 8)

3. שלוש הדרכים לקבל פרמטרים — params / query / body

כדי לא להגדיר אינספור נתיבים, שולחים לשרת פרמטרים בשלוש דרכים:

3.1 req.params — פרמטר בתוך הנתיב (route parameter)

מגדירים חלק דינמי בנתיב עם נקודתיים, למשל :userId, ושולפים אותו מ-req.params: 

app.get("/user/:userId", (req, res) => {
    res.send(`Showing results for user: ${req.params.userId}`);
});
// GET /user/5  →  "Showing results for user: 5"

3.2 req.query — מחרוזת השאילתה (querystring)

פרמטרים שמגיעים אחרי ? בכתובת נשלפים מ-req.query: 

app.get("/find", (req, res) => {
    res.send(`Author: ${req.query.author}. Title: ${req.query.title}`);
});
// GET /find?author=Stephen%20King&title=it
//   →  "Author: Stephen King. Title: it"

פרמטר חסר = undefined, לא שגיאה. קריאה ל-/find בלי querystring מחזירה "Author: undefined. Title: undefined". לעומת זאת, בקשה לנתיב שאין לו ראוט כלל (למשל GET / ללא app.get('/')) מחזירה את הודעת ברירת המחדל "Cannot GET /".

3.3 req.body — גוף הבקשה (ב-POST/PUT)

פרמטרים ב-POST/PUT נשלחים בגוף ההודעה ונשלפים מ-req.body (למשל req.body.username). אבל — כדי שזה יעבוד חייבים middleware לפרסור הגוף, אחרת req.body ריק. על כך בסעיף הבא.

מקור: 10.1_Lec_Express (עמ' 17–22, 25–26)

4. Middleware ו-app.use

Middleware הוא, בהגדרת המרצה, "Code that sits between two layers of software" — ב-Express, קוד שיושב בין ה-request לבין ה-response. פונקציות ה-middleware יוצרות שרשרת callbacks, ומגדירים אותן עם app.use. החתימה שלהן היא שלושה פרמטרים — (req, res, next): 

app.all("*", (req, res, next) => {
    console.log("1- All route");
    next();          // בלי זה — נעצרים כאן!
});
app.get("/", (req, res) => {
    console.log("2- Get route");
    res.send({ result: "WOW!" });
});
// הפלט: "1- All route"  ואז  "2- Get route"

הפרמטר next והקריאה next() הם מה שמעביר את הזרימה ל-handler הבא בשרשרת. בלי next() — ובלי שנשלחה תשובה — הבקשה תיתקע. כך "מנהלים" מצב שבו כמה handlers מתאימים לאותה בקשה.

4.1 express.json() — למה req.body ריק בלעדיו

כדי לקרוא גוף בקשה חייבים middleware לפרסור. Express כולל שני middleware מובנים (built-in, מגרסה 4.16.0+): express.json() לגוף JSON גולמי, ו-express.urlencoded() לטפסי x-www-form-urlencoded: 

app.use(express.json());              // מפרסר גוף JSON גולמי
app.use(express.urlencoded({ extended: true })); // מפרסר טפסי x-www-form-urlencoded

בדיפולט req.body הוא undefined. אם שולחים POST עם JSON אבל שוכחים את app.use(express.json()), req.body.username יהיה undefined — באג נפוץ. הערה: המודול body-parser מובנה מגרסה 4 — אין להשתמש בו ישירות (זה קוד ישן).

4.2 middleware ממוקד-נתיב וסדר ההגדרה

ל-app.use אפשר לתת נתיב ראשון, כך שירוץ רק עליו (לכל שיטת HTTP): 

app.use('/user/:id', (req, res, next) => {
    console.log('Request Type:', req.method);
    next();
});

סדר ה-middleware קריטי. Express מריץ middleware ו-routes לפי סדר ההגדרה בקוד. middleware שאמור להשפיע על ראוט חייב להיות מוגדר לפניו — נראה זאת חד בבעיית ה-CORS (סעיף 6).

העשרה — express.static להגשת קבצים סטטיים. כדי ש-Express יגיש קבצים "כמו שהם" (HTML, CSS, JS, תמונות) מתיקייה, משתמשים ב-middleware מובנה נוסף: app.use(express.static('public')). קובץ public/style.css יוגש אז בכתובת /style.css. נהוג לצרף נתיב מוחלט כדי לא להיות תלויים בתיקיית ההרצה: express.static(path.join(__dirname, 'public')), כאשר __dirname הוא הנתיב המוחלט של תיקיית הקובץ הנוכחי. (מעבר לשקפים — להעשרה בלבד.)

מקור: 10.1_Lec_Express (עמ' 15–16, 21–23), 11.1_Lec_Node_and_Client (עמ' 4–5) · express.static — העשרה

5. Express Router — ארגון הראוטים

ככל שהמערכת גדלה, ריכוז כל הראוטים בקובץ אחד הופך למסורבל. ה-Express Router הוא קלאס שעוזר ליצור ולנהל ראוטים בקובץ נפרד. הרעיון: req → index.js → router → controller → res — הראוטר מפנה כל בקשה לקונטרולר המתאים. 

// routers/postsRouter.js
const { Router } = require('express');
const { postsController } = require('../controllers/postsController.js');

const postsRouter = new Router();

postsRouter.get('/', postsController.getPosts);
postsRouter.get('/:postId', postsController.getPost);
postsRouter.post('/', postsController.addPost);
postsRouter.put('/:postId', postsController.updatePost);
postsRouter.delete('/:postId', postsController.deletePost);

module.exports = { postsRouter };

ב-index.js מחברים את הראוטר לנתיב-בסיס עם app.use, ומוסיפים middleware "תופס-הכול" בסוף לבקשות שלא טופלו: 

app.use('/api/posts', postsRouter);       // כל /api/posts/... עובר לראוטר

app.use((req, res) => {                   // catch-all — חייב להיות אחרון
    res.status(400).send('Something is broken!');
});

מקור: 11.1_Lec_Node_and_Client (עמ' 33–37)

6. CORS — למה מקור/פורט שונה נחסם

CORS = Cross Origin Resource Sharing. מטעמי אבטחה, הדפדפן חוסם בקשות AJAX (fetch) בין שני origins שונים — כאשר origin = פרוטוקול + host + port. לדוגמה, לקוח שרץ על Live Server ב-127.0.0.1:5500 שפונה לשרת Express ב-127.0.0.1:8080 — פורט שונה = origin שונה = נחסם: 

Access to fetch at 'http://127.0.0.1:8080/books' from origin
'http://127.0.0.1:5500' has been blocked by CORS policy:
No 'Access-Control-Allow-Origin' header is present on the requested resource.

נקודה למבחן: הבקשה כן מגיעה לשרת והקוד בשרת רץ (רואים בטרמינל "Got the request...") — אבל התשובה נחסמת בדפדפן. לכן פותרים את CORS בצד השרת בלבד, לא בלקוח.

הפתרון: middleware שמוסיף לתשובה כותרות הרשאה — ובעיקר Access-Control-Allow-Origin (דומיין ספציפי או '*' לכולם) — לפני הגדרת הראוטים: 

app.use((req, res, next) => {
    res.set({
        'Access-Control-Allow-Origin': '*',
        'Access-Control-Allow-Headers': 'Origin, X-Requested-With, Content-Type, Accept',
        'Access-Control-Allow-Methods': "GET, POST, PUT, DELETE",
        'Content-Type': 'application/json'
    });
    next();
});

app.get("/books", (req, res) => res.json(booksData));

הסדר שוב קריטי: אם ה-app.use של CORS מוגדר אחרי ה-route, ה-route ישלח תשובה ראשון וה-middleware של CORS לא ירוץ כלל — והשגיאה תישאר. קיים גם מודול cors ב-npm (לא נלמד בקורס).

מקור: 11.1_Lec_Node_and_Client (עמ' 9–13), 11.2_Tar_Node_and_Client (עמ' 8–11)

7. בדיקה עצמית

בהינתן הקוד הבא, מה יישלח ללקוח עבור GET localhost:8080/?

app.all('*', (req, res) => res.send("Global handler"));
app.get('/',  (req, res) => res.send("Server root"));

שולחים POST /users עם גוף JSON {"username":"Alf"}, אך התשובה היא {"user": undefined}. מה הבאג?

const app = express();
app.post("/users", (req, res) => {
    res.json({ user: req.body.username });
});

מה יישלח ללקוח עבור הבקשה GET /user/42?sort=name?

app.get("/user/:userId", (req, res) => {
    res.send(`user ${req.params.userId}, sort ${req.query.sort}`);
});

לקוח ב-:5500 עושה fetch לשרת שקודו למטה, ומקבל שגיאת CORS. מה קורה בפועל?

app.get("/books", (req, res) => res.json(booksData));

app.use((req, res, next) => {                 // ← אחרי הראוט!
    res.set('Access-Control-Allow-Origin', '*');
    next();
});

מה חווה הלקוח כשהוא פונה ל-GET /data עם ה-callback הבא?

app.get("/data", (req, res) => {
    const result = compute();
    console.log(result);        // אין res.send / res.json !
});
