1. Git מול GitHub — מקומי מול ענן

אלה שני דברים שונים ששמם דומה, וזו מלכודת קלאסית. Git הוא מערכת ניהול גרסאות (version control system) שמותקנת ופועלת מקומית על המחשב שלך — היא מנהלת היסטוריה של כל שינוי בקוד ומאפשרת לחזור אחורה בלחיצת כפתור. GitHub, לעומת זאת, הוא שירות אחסון ריפוזיטוריז מבוסס-ענן בלבד (a Git repository hosting service) — מקום מרוחק ומרכזי שבו הקוד יושב, ניתן לשיתוף ולעבודה צוותית.

אנלוגיה: Git הוא הכלי שכל מפתח מריץ אצלו על המחשב; GitHub הוא "הרשת החברתית" (social coding) שאליה כולם דוחפים את הקוד. אפשר להשתמש ב-Git בלי GitHub (עבודה מקומית טהורה), אבל כדי לשתף — צריך שרת מרוחק כמו GitHub.

שתי עובדות שהמרצה מדגישה (אינפוגרפיקה Git vs. GitHub):

  • Git — פותח ב-2005, מותקן ומתוחזק על המערכת המקומית (לא בענן), ומצטיין ב-מודל ה-branching שלו.
  • GitHub — שירות אירוח ריפו מבוסס-ענן בלבד (exclusively cloud-based), שמאפשר לשתף קוד ולתת לאחרים להגיה ולערוך.

מלכודת מבחן: "commit שולח את הקוד ל-GitHub" — שגוי. commit הוא תמיד פעולה מקומית; רק push מעלה לשרת המרוחק. פירוט בבלוק הבא.

מקור: 2.2_Lec_Git_GitHub (עמ' 14–16)

2. ארבעת המרחבים וזרימת הפקודות

הקוד "נע" בין ארבעה מרחבים, וכל מעבר הוא פקודת Git אחת. זהו לב-ליבו של הנושא ומקור לשאלות רבות: 

Working Directory  --(git add)-->   Staging Area
Staging Area       --(git commit)--> Local Repository
Local Repository   --(git push)-->   Remote Repository (GitHub)
Remote Repository  --(git pull)-->   Local Repository
  • Working Directory — תיקיית העבודה בפועל, הקבצים שאתה עורך כרגע.
  • Staging Area — "אזור המתנה" שאליו בוחרים אילו שינויים ייכנסו ל-commit הבא. מגיעים לכאן עם git add.
  • Local Repository — הריפו המקומי (תיקיית .git) עם כל ההיסטוריה. מגיעים לכאן עם git commit.
  • Remote Repository — הריפו בענן ב-GitHub. מגיעים לכאן עם git push, ומורידים ממנו עם git pull.
פקודהמאיפהלאןמקומי / מרוחק
git add .Working DirStagingמקומי
git commit -mStagingLocal Repoמקומי
git pushLocal RepoRemoteמרוחק
git pullRemoteLocal Repoמרוחק

staging area — למה בכלל? הוא מאפשר לבחור חלק מהשינויים לקומיט: git add index.html מוסיף קובץ בודד, ואילו git add . מוסיף את כל הקבצים ששונו. רק מה שנמצא ב-staging נכנס ל-commit הבא.

העשרה — clone: כדי להתחיל לעבוד על ריפו קיים שכבר נמצא ב-GitHub, מריצים git clone [URL]. הפקודה מורידה עותק מלא של הריפו כולל כל ההיסטוריה ומחברת אותו אוטומטית ל-remote (origin) — כך שאין צורך ב-git init או git remote add אחריו. (מעבר לשקפים, ידע כללי סטנדרטי.)

מקור: 2.2_Lec_Git_GitHub (עמ' 6), 2.4_Tar_Git_GitHub (עמ' 19)

3. זרימת העבודה המלאה בטרמינל — מ-init ל-push

לפני הכול, פעם אחת בכל מחשב, מגדירים זהות. הגדרה זו נכתבת פעם אחת ומשמשת את כל הפרויקטים: 

git config --global user.name  "YOUR NAME"
git config --global user.email "YOUR EMAIL"

וזהו הרצף המלא בתוך תיקיית פרויקט חדש:

cd C:\Users\Me\Desktop\my-project   # ניווט לתיקייה
git init                            # רק פעם אחת בפרויקט — יוצר את תיקיית .git
git status                          # (לא חובה) מציג קבצים untracked
git add .                           # כל הקבצים אל ה-staging area
git commit -m "first commit"        # "מתייקים" מקומית
git remote add origin [URL]         # קושרים לריפו המרוחק ב-GitHub (פעם אחת)
git push -u origin master           # מעלים לענן
  • git init מריצים רק בפעם הראשונה בתיקייה; הוא יוצר את תיקיית .git המוסתרת (ספריית ה-logs). אם אינך רואה אותה — זה תקין, היא מוסתרת.
  • הדגל -m ב-git commit -m "..." מצמיד את הודעת ה-commit.
  • git remote add origin [URL] מקשר את התיקייה המקומית לריפו המרוחק — פעם אחת בפרויקט.

הסיסמה ב-push אינה סיסמת GitHub! כשה-cmd מבקש username ו-password בעת ה-push, בשדה הסיסמה יש להזין Personal Access Token — טוקן שמייצרים ב-Settings → Developer Settings → Personal Access Tokens → Tokens (classic) → Generate new token. אי אפשר לשחזר אותו לאחר היצירה — לכן שמרו אותו במקום סודי.

README: קובץ README.md (בפורמט Markdown) הוא "כרטיס הביקור" של הריפו — GitHub מציג אותו אוטומטית בעמוד הראשי. בטופס יצירת ריפו יש צ'קבוקס Add a README file, ובשורת הפקודה מוסיפים אותו כך: 

echo "# my-project" >> README.md

מקור: 2.4_Tar_Git_GitHub (עמ' 2, 6–10, 14–15), 2.2_Lec_Git_GitHub (עמ' 11)

4. .gitignore — מה לא מעלים

.gitignore הוא קובץ שמפרט תבניות של קבצים/תיקיות ש-Git יתעלם מהם לחלוטין — הם לא ייכנסו ל-staging, לא ל-commit ולא ל-GitHub. בטופס יצירת ריפו ב-GitHub אף מוצע .gitignore template מוכן לכל שפה.

העשרה — שני החשודים המיידיים בפרויקט Node (ידע מעשי שהמרצה מצפה לו):

  • node_modules — תיקיית התלויות. היא ענקית ומיותרת בריפו: כל מי שמושך את הפרויקט פשוט מריץ npm install, וזו משחזרת אותה מדויק מתוך package.json. אין טעם לדחוף אלפי קבצים לענן.
  • .env — קובץ משתני הסביבה עם סודות (connection string ל-MongoDB, מפתחות API, סיסמאות). דחיפתו ל-GitHub — במיוחד לריפו ציבורי — חושפת את הסודות לכולם.
# .gitignore
node_modules/
.env

שים לב: ערכי ה-.env לא נעלמים — פשוט לא מעלים אותם דרך Git. בפריסה (למשל Render) מגדירים את אותם משתני סביבה ידנית בלוח הבקרה של השירות. את .gitignore עצמו כן מעלים.

מקור: 2.4_Tar_Git_GitHub (עמ' 4) · העשרה מעבר לשקפים

5. GitHub Desktop — אותה זרימה, בלי טרמינל

GitHub Desktop הוא ממשק גרפי (GUI) שמבצע בדיוק את אותן פעולות Git בלחיצות כפתור. השימוש בו אינו חובה — אפשר לעבוד עם פקודות Git בטרמינל או עם כלי אחר (למשל SourceTree). כל אלה חלופות שקולות. המיפוי המדויק:

ב-GitHub Desktopשקול לפקודה
File → New repositorygit init
סימון ה-V ליד קובץ ב-Changesgit add .
Publish repositorygit remote add origin [URL]
Commit to mastergit commit -m "message"
Publish branch / Push origingit push -u origin master

גם כאן commit הוא לוקאלי! לחיצה על Commit to master ב-GitHub Desktop לא שולחת כלום לשרת — עדיין צריך Push origin / Publish branch. ה-V (צ'קבוקס) ליד כל קובץ קובע אילו קבצים ייכנסו לקומיט — בדיוק כמו staging.

רמז ויזואלי לזרימה: לאחר Publish repository הכפתור מתחלף ל-Publish branch; לאחר ה-push הראשון הוא הופך ל-Fetch origin.

מקור: 2.5_Tar_GitHub_Desktop (עמ' 3–10)

6. Branches, Merge וקונפליקטים

כדי לעבוד על שינוי בלי לסכן את הקוד היציב, פותחים branch (ענף) חדש שיוצא מ-master, עובדים עליו, ולבסוף ממזגים (merge) חזרה. פקודות המפתח: 

git checkout -b feature   # יוצר ענף חדש "feature" וגם עובר אליו מיד
git checkout master       # עובר לענף קיים (בלי ליצור)
git merge feature         # ממזג את feature לתוך הענף הנוכחי
git branch -d feature     # מוחק ענף

שתי דקויות שנבחנות: (1) git checkout -b feature יוצר וגם עובר לענף — בשונה מ-git checkout feature שרק עובר לענף קיים. (2) git merge feature ממזג את feature לתוך הענף שאתה עומד עליו כרגע — לא להיפך.

כששני חברי צוות שינו את אותו אזור בקוד, ה-merge נכשל ונוצר merge conflict שיש לפתור ידנית. Git מסמן את שני הצדדים בקובץ: 

<<<<<<< HEAD
    return [{          // הקוד שלך (Current Change)
=======
    return [           // הקוד הנכנס (Incoming Change)
>>>>>>> feature

בוחרים איזה צד להשאיר (או משלבים ידנית), מוחקים את שורות הסימון (<<<<<<<, =======, >>>>>>>), ואז מקמטים ודוחפים. VS Code מציע כפתורים Accept Current / Incoming / Both לזירוז.

מקור: 2.2_Lec_Git_GitHub (עמ' 8–12), 2.4_Tar_Git_GitHub (עמ' 12)

7. הגשות הקורס והקשר לפריסה (Render)

בהגשות בקורס תמיד מצרפים את הלינק לריפו ב-GitHub, ו-חובה לוודא שהריפו אינו פרטי: ריפו פרטי ללא הרשאות פירושו שהבודקים אינם רואים את הקבצים ולא יכולים לתת ציון. מומלץ לפתוח את הכתובת גם כמשתמש אנונימי כדי לוודא שהוא באמת ציבורי (או לעבוד דרך GitHub Classroom, שם למרצה יש גישת אדמין אוטומטית).

העשרה — למה GitHub קריטי לפריסה: Render (וכל שירותי הפריסה המודרניים) לא מעלים אליהם קבצים ידנית — הם מושכים את הקוד ישירות מריפו ב-GitHub. מגדירים חיבור בין השירות לענף (למשל master) פעם אחת, ומאותו רגע auto-deploy פועל: כל git push לענף המקושר מפעיל אוטומטית build ופריסה מחדש של האתר.

ההשלכה המעשית (מלכודת נפוצה): אם עשית git commit מקומי אבל שכחת git push — האתר ב-Render לא יתעדכן, כי ל-GitHub עדיין אין את הקומיט שלך. "עשיתי commit והאתר לא זז" = כמעט תמיד שכחת push.

מקור: 2.2_Lec_Git_GitHub (עמ' 16) · הקשר Render — העשרה (ר' מודול 11)

8. בדיקה עצמית

מפתח הריץ בהצלחה git init, git add . ו-git commit -m "v1". הוא פותח את עמוד הריפו ב-GitHub ורואה אותו ריק. מה קרה?

מה עושה בדיוק הפקודה git checkout -b feature?

בעת git push מה-cmd, מתבקשים username ו-password. מה צריך להזין בשדה הסיסמה?

בפרויקט Node/Express, אילו פריטים נכון להוסיף ל-.gitignore כך שלא יעלו ל-GitHub?

‏Render מחובר לריפו שלך ב-GitHub עם auto-deploy. תיקנת באג, הרצת git add . ו-git commit -m "fix" — אך האתר החי ב-Render עדיין מציג את הבאג. מדוע?
