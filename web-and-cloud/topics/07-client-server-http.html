0. מודל לקוח–שרת — מה רץ איפה

אפליקציית ווב מפוצלת לשני צדדים. הלקוח (client) הוא הדפדפן: שם רצים ה-HTML, ה-CSS וה-JavaScript שרואה המשתמש. השרת (server) מריץ את ה-API, את הלוגיקה העסקית ואת בסיס הנתונים. שני הצדדים מדברים ביניהם בבקשות ותשובות על-גבי פרוטוקול HTTP דרך האינטרנט. 

Web app in Browser  ⇄  Internet  ⇄  API  ⇄  Web Server  ⇄  Database
      (Request →)                                (← Response)

0.1 הפרדת שכבות — MVC

המבנה המקובל מפריד אחריות לפי Separation of concerns: Model (הנתונים/מסד), View (התצוגה) ו-Controller (הלוגיקה שמקשרת ביניהם). בפרויקט ה-AJAX של הקורס המיפוי מוחשי: ה-HTML הוא ה-View, קובץ ה-JSON הוא ה-Model, וה-JS הוא ה-Controller.

כלל זהב מהתרגול: אין חיבור ישיר בין ה-HTML ל-JSON — ה-JS הוא תמיד ה-Controller ביניהם: HTML → JS → JSON.

0.2 REST

REST (Representational State Transfer) היא ארכיטקטורה ששולחת הודעות על-גבי HTTP. השירות מצפה לקבל URI עם פרמטרים או אובייקטי JSON, ובדרך כלל מחזיר תשובה כ-XML או JSON. כיום JSON מועדף — הוא קטן, יעיל ומינימליסטי.

מקור: 9.1_Lec_Client_Server_2 (עמ' 4–7), 7.1_Lec_Client_Server_1 (עמ' 4–7), 7.2_Tar_Ajax_Json (עמ' 3)

1. הודעת HTTP — Request ו-Response

לכל הודעת HTTP אותו מבנה משלושה חלקים: שורת פתיחה, כותרות (headers), ואז ה-body. הבקשה נשלחת מהלקוח לשרת; שדות הבקשה מוגדרים בצד הלקוח. התשובה חוזרת מהשרת; שדות התשובה מוגדרים בצד השרת.

1.1 בקשה (Request)

מתחילה ב-Request Line עם הפועל (GET, וכו'), הנתיב וגרסת הפרוטוקול:

GET /doc/test.html HTTP/1.1
Host: www.test101.com
Accept: image/gif, image/jpeg, */*
Accept-Language: en-us
Content-Length: 35

bookId=12345&author=Tan+Ah+Teck

1.2 תשובה (Response)

מתחילה ב-Status Line עם גרסת הפרוטוקול וקוד הסטטוס:

HTTP/1.1 200 OK
Date: Sun, 08 Feb xxxx 01:11:12 GMT
Server: Apache/1.3.29 (Win32)
Content-Length: 35
Connection: close
Content-Type: text/html

<h1>My Home page</h1>

נקודת מבחן: שורה ריקה (blank line) היא מה שמפריד בין ה-headers ל-body — בשתי ההודעות, גם בבקשה וגם בתשובה.

1.3 הכותרת Content-Type

הכותרת Content-Type מצהירה על סוג המדיה של ה-body — למשל text/html לדף HTML, או application/json לתשובת JSON. הצד המקבל קורא לפיה כיצד לפרש את גוף ההודעה.

מקור: 9.1_Lec_Client_Server_2 (עמ' 9–10), 7.1_Lec_Client_Server_1 (עמ' 9–10)

2. מתודות HTTP — GET / POST / PUT / DELETE

מערכות RESTful מתקשרות בפעלים (HTTP Verbs). לא כל הדפדפנים תומכים בכולם — כולם תומכים לפחות ב-GET וב-POST, והרוב תומכים בארבע השימושיות:

  • GET — קבלת מידע (קריאה).
  • POST — הוספת מידע חדש.
  • PUT — עדכון מידע קיים.
  • DELETE — מחיקת מידע.

מיפוי CRUD ↔ HTTP (חובה לזכור): Create → POST · Read → GET · Update → PUT · Delete → DELETE.

דוגמת routes ל-RESTful API של משאב customers:

פעולהמתודה + נתיב
יצירת לקוח חדשPOST /customers
קבלת רשימת הלקוחותGET /customers
קבלת לקוח בודדGET /customers/{id}
עדכון לקוח קייםPUT /customers/{id}
מחיקת לקוחDELETE /customers/{id}

מלכודת: כל בקשה שנשלחת משורת הכתובת של הדפדפן היא תמיד GET. כדי לשלוח POST/PUT/DELETE צריך קוד או כלי כמו Postman / Advanced REST client. טופס (form) לעומת זאת יכול לשלוח GET או POST.

העשרה — קווים מנחים ל-API: להשתמש בשמות עצם ולא בפעלים (/articles), בצורת רבים, וב-nesting להיררכיה (/articles/:id/comments). כשה-API מתועד היטב, הלקוח והשרת יכולים להיכתב בשפות שונות לגמרי — אין כאן "שפת שרת" מול "שפת לקוח".

מקור: 9.1_Lec_Client_Server_2 (עמ' 22, 27–30), 7.1_Lec_Client_Server_1 (עמ' 27–30)

3. מבנה URL ו-Query String

ה-query string הוא החלק ב-URL שמאפשר לצרף פרמטרים בצורת מפתח-ערך: 

https://www.domain.com/page?key1=value1&key2=value2
                          │        │       │
                          ?        =       &
              query string begins  value   separator
                                separator

? פותח את ה-query string, = מפריד מפתח מערך, ו-& מפריד בין זוגות. כך מעבירים מידע בין דפים — למשל קישור book.html?bookId=456.

בקוד קוראים את הפרמטר מתוך ה-URL כך (מהתרגול):

function getBookId() {
  const aKeyValue = window.location.search.substring(1).split('&');
  const bookId = aKeyValue[0].split("=")[1];
  return bookId;              // "456"  ← מחרוזת
}

שימו לב: substring(1) מוריד את ה-? שבתחילת location.search, ואז מפצלים לפי & ואז לפי =. הערך שחוזר הוא תמיד מחרוזת — לכן ההשוואה בהמשך נעשית עם == רופף (bookObj.id == selectionBookId).

מקור: 9.1_Lec_Client_Server_2 (עמ' 32), 7.2_Tar_Ajax_Json (עמ' 11, 14)

4. קודי סטטוס (Status Codes)

ה-status code שבשורת ה-Status Line אומר ללקוח מה עלה בגורל הבקשה. הקודים הנפוצים:

קודמשמעותמתי
200 OKהצליחבקשה תקינה שהוחזרה בהצלחה (למשל GET)
201 Createdנוצר משאבטיפוסי ל-POST שיצר משאב חדש
400 Bad Requestבקשה שגויההלקוח שלח קלט לא תקין
404 Not Foundלא נמצאהמשאב/הנתיב לא קיים בשרת
500 Internal Server Errorשגיאת שרתתקלה בצד השרת

העשרה — קריאת המשפחות: הספרה הראשונה מקבצת את הקודים: 2xx = הצלחה, 4xx = שגיאת לקוח (הבקשה עצמה בעייתית), 5xx = שגיאת שרת. לכן 404 היא "אשמת" הלקוח (ביקש נתיב לא קיים) ואילו 500 היא תקלה אצל השרת. (חלוקה זו היא ידע כללי מעבר לשקפים, שמובא להעשרה.)

מקור: 9.1_Lec_Client_Server_2 (עמ' 10, קוד 200 OK) · פירוט 201/400/404/500 — העשרה

5. JSON — מבנה, stringify ו-parse

JSON (JavaScript Object Notation) הוא פורמט קל להחזקת ושיתוף מידע במבנה מפתח-ערך (key-value). התחיל ב-JavaScript אך היום פורמט עצמאי, ומחליף את XML ברוב המקרים.

  • ממירים לאובייקט עם JSON.parse ולמחרוזת עם JSON.stringify.
  • הערכים יכולים להיות: Strings, Objects, Numbers, Booleans, Array, null.
  • מבנה נתונים לא-רלציוני (NoSQL, למשל Key-Value store).

חובה למבחן: ב-JSON חובה מרכאות כפולות סביב המפתחות וגם סביב מחרוזות. שימו לב לטיפוסים: id נשמר כמחרוזת ("123") בעוד price נשמר כמספר (16.65). 

var student = {
  "name": "dan sagiv",
  "address": "Haifa",
  "grades": [
    {"math": 95},
    {"english": 90},
    {"statistics": 85}
  ]
};

console.log(student.name);            // dan sagiv
console.log(student.grades.length);   // 3
console.log(student.grades[0].math);  // 95

מקור: 9.1_Lec_Client_Server_2 (עמ' 13–14), 7.1_Lec_Client_Server_1 (עמ' 13–14)

6. AJAX, fetch ו-Promises

AJAX (Asynchronous JavaScript And XML) מאפשר להביא דאטה מהשרת בלי לטעון מחדש את הדף (למשל השלמה אוטומטית בחיפוש גוגל). ה-JS יוצר בקשה, מקבל תשובה, ומעדכן את התוכן — הכול ברקע, אסינכרונית.

6.1 XMLHttpRequest

הדרך הוותיקה לקרוא דאטה היא אובייקט XMLHttpRequest (XHR) — שלמרות שמו משמש לכל אינטראקציה עם השרת, לא רק ל-XML: 

var xhttp = new XMLHttpRequest();
xhttp.onreadystatechange = function() {
  if (this.readyState == 4 && this.status == 200) {
    document.getElementById("demo").innerHTML = xhttp.responseText;
  }
};
xhttp.open("GET", "filename", true);
xhttp.send();

מצבי readyState: 0 UNSENT, 1 OPENED, 2 HEADERS_RECEIVED, 3 LOADING, 4 DONE. בודקים readyState == 4 (התשובה מוכנה) וגם status == 200 (הצליח).

6.2 fetch ו-Promise

בקורס משתמשים בפונקציה fetch לבקשת AJAX פשוטה. fetch מחזירה Promise — אובייקט שמייצג תוצאה שעדיין-לא-מוכנה, ונמצא באחד משלושה מצבים: pending (ממתין), fulfilled (הצליח) או rejected (נכשל). 

fetch("data/books.json")
  .then(response => response.json())
  .then(data => console.log(data));

למה שני .then? גם fetch(url) מחזירה Promise, וגם response.json() מחזירה Promise נוסף. ה-.then הראשון ממיר את ה-Response ל-JSON, והשני מקבל את הדאטה הסופי.

מלכודות AJAX: קריאת AJAX חייבת שרת — בלי שרת (פתיחת קובץ ישירה) הבקשה לא תתבצע; לכן מריצים דרך Live Server. בנוסף, קבצי ה-HTML, ה-JS וקובץ הדאטה חייבים לשבת באותו דומיין.

העשרה — async/await: אפשר לכתוב את אותה שרשרת בסגנון סינכרוני-למראה עם await. שקול לחלוטין לשרשרת ה-.then: 

async function load() {
  const res  = await fetch("data/books.json");
  const data = await res.json();
  console.log(data);
}

מלכודת ד׳ (העשרה — חשוב מאוד למבחן): fetch אינה נדחית (rejected) על 404 או 500! קוד סטטוס של שגיאת HTTP עדיין נחשב תשובה שהתקבלה — ה-Promise מתקיים (fulfilled). רק כשל רשת ממשי דוחה אותה. לכן חובה לבדוק ידנית: 

const res = await fetch("/missing");
console.log(res.ok);      // false
console.log(res.status);  // 404
// ה-catch לא ירוץ! צריך לבדוק res.ok בעצמנו

מקור: 9.1_Lec_Client_Server_2 (עמ' 16–21), 7.1_Lec_Client_Server_1 (עמ' 16–21), 7.2_Tar_Ajax_Json (עמ' 5) · async/await + fetch-404 — העשרה

7. איפה מבצעים ולידציה רגישה?

ולידציה בצד הלקוח (HTML5 עם required, בדיקות JS) היא נחמדה לחוויית משתמש — משוב מיידי בלי סיבוב לשרת. אבל היא אינה אבטחה.

העשרה — כלל ברזל: כל ולידציה רגישה (הרשאות, מחירים, בעלוּת על משאב, טיפוסי נתונים) חייבת להתבצע בשרת. הלקוח נשלט בידי המשתמש: אפשר לעקוף כל בדיקת JS דרך DevTools או לשלוח בקשה "נקייה" ישירות עם Postman. הכלל: never trust the client — ולידציית לקוח לחוויה, ולידציית שרת לאמת.

מקור: 7.1_Lec_Client_Server_1 (עמ' 24–25, מודל השרת כמקור-אמת) · דגש הביטחון — העשרה
