0. האינטואיציה — לומדים מדוגמאות, לא ממישהו שמכתיב חוקים

עד עכשיו כל סוכן שפגשנו קיבל את הידע שלו מוכן: חוקי לוגיקה, פונקציית מעבר, יוריסטיקה. עכשיו השאלה הפוכה — איך סוכן בונה ידע בעצמו מתוך דוגמאות? זהו תפקידו של הסוכן הלומד (Learning Agent): רכיב Critic מקבל Performance standard ותחושות מה-Sensors, ומזין feedback ל-Learning element — הרכיב שבאמת "לומד" ומשנה את התנהגות ה-Performance element (שמפעיל את ה-Actuators). ה-Learning element גם מזין learning goals ל-Problem generator, שמציע ניסויים חדשים ל-Performance element.

שלושה סוגי למידה לפי סוג הפידבק שהסוכן מקבל: Supervised learning — דוגמאות של קלט ופלט ראוי, כשהתשובות ניתנות ע"י מורה או ע"י ניסיון הסוכן; Unsupervised learning — אין תשובות נכונות, לומדים תבניות מתוך הקלט בלבד; Reinforcement learning — למידה מפרסים וקנסות, כשצריך לדעת כיצד הסביבה עובדת. Supervised מתפצל ל-Classification (פלט בדיד) ו-Regression (פלט רציף); Unsupervised מוביל ל-Clustering. עצי החלטה, נושא הפרק, הם דוגמה קלאסית ל-supervised classification.

בכל למידה אינדוקטיבית מקבלים זוגות (x, f(x)) — f היא פונקציית המטרה, ודוגמה היא הזוג (x, f(x)). הבעיה: מצאו היפותזה h כך ש-h ≈ f, בהינתן אוסף דוגמאות המכונה training set. היפותזה h עקבית אם כל נקודות הדוגמאות נמצאות עליה בדיוק — אבל כפי שמדגים שקף רגרסיה קלאסי, כשיש כמה היפותזות עקביות (קו ישר, פולינום חלק, עקומה משוננת שמשתלטת על כל רעש בדוגמאות) לא ברור איזו עדיפה:

Ockham's razor: מבין ההיפותזות העקביות, העדיפו את הפשוטה ביותר. לרוב יש trade-off בין סיבוכיות ההיפותזה למידת התאמתה לקלט — עקביות מלאה לבדה אינה קריטריון מספיק לבחירה, כפי שמראה העקומה המשוננת: היא עקבית לחלוטין אך לא בהכרח שימושית לניבוי דוגמאות עתידיות.

כשההיפותזה עצמה נבחרת מראש להיות קו ישר (רגרסיה לינארית), נשאר לבחור את הקו הטוב ביותר מבין כל הקווים האפשריים — לפי סכום ריבועי השגיאות Σⱼ (yⱼ − w₀xⱼ − w₁)², ובוחרים את LOSS = Min₍w1,w0₎ Σⱼ(yⱼ−w₀xⱼ−w₁)² — ההיפותזה שממזערת את פונקציית השגיאה (משטח קעור שממנו קל למצוא מינימום).

1. עצי החלטה — מבנה ובעיית ההמתנה במסעדה

עץ החלטה: הקלט הוא סיטואציה המתוארת ע"י מאפיינים (attributes), הפלט הוא החלטה — סיווג הקלט לאחד מכמה פלטים אפשריים (פונקציה בדידה). קודקוד פנימי הוא מבחן על אחד המאפיינים, וענפיו היוצאים הם הערכים האפשריים למבחן הזה; עלה מכיל את ערך הפלט שמוחזר אם מגיעים אליו.

דוגמת העבודה המרכזית של הפרק — "האם נחכה לשולחן במסעדה?" עשרה מאפיינים: Alternate (יש מסעדה חלופית קרובה), Bar (יש בר להמתנה), Fri/Sat (סופ"ש), Hungry (רעבים), Patrons (None/Some/Full), Price ($/$$/$$$), Raining, Reservation, Type (French/Italian/Thai/Burger) ו-WaitEstimate (0-10/10-30/30-60/>60).

שימו לב: זהו העץ ה"אמיתי" — הפונקציה שיצרה את הדוגמאות. בהמשך נבנה עץ מהדוגמאות עצמן באמצעות אלגוריתם למידה, ונשווה בין השניים — הם לא זהים.

3. בחירת המאפיין הראשון — Patrons מול Type

הרעיון המרכזי בבניית העץ: לבדוק ראשון את המאפיין הכי חשוב — זה שמחלק את הדוגמאות לתתי-קבוצות טהורות ככל האפשר (רק חיוביות או רק שליליות). כך נקווה שלא נצטרך לבדוק מאפיינים רבים נוספים, והעץ יהיה נמוך וקומפקטי.

נשווה בין שני מועמדים: Patrons (3 ערכים) ו-Type (4 ערכים).

  • Patrons: None → 2 שליליות בלבד (טהור!); Some → 4 חיוביות בלבד (טהור!); Full → 2 חיוביות + 4 שליליות (מעורב).
  • Type: בכל אחד מארבעת הערכים (French, Italian, Thai, Burger) יש תמהיל מעורב של חיוביות ושליליות — הפיצול לא מבודד כלום.

זו הערכה ויזואלית בלבד — עדיין לא כימתנו "כמה יותר טוב". זה בדיוק מה שהאנטרופיה ו-Information Gain, בבלוקים הבאים, נותנים לנו: מספר מדויק להשוואה בין כל שני מאפיינים.

4. תורת האינפורמציה — נוסחת האנטרופיה

ב-1948 הגדיר קלוד שאנון את הביט כיחידת מידע ואת האנטרופיה של מקור מידע — כמות הביטים המינימלית הנדרשת לקודד מסר. הרעיון שאנו שואלים: כמה מידע חסר? כמה מידע נדרש כדי לדעת את התשובה בוודאות. כמות המידע נמדדת בביטים: 0 משמעו מערכת ודאית — אין צורך במידע כלל; 1 משמעו אי-ודאות מלאה — צריך מידע מושלם.

אם לתשובה אפשרית xᵢ יש הסתברות P(xᵢ), המידע הכולל:

I(P(x1),...,P(xn)) = Σ −P(xi)·log2 P(xi)

לדוגמה, הטלת מטבע הוגן: I(0.5,0.5) = −0.5·log₂(0.5) − 0.5·log₂(0.5) = 0.5+0.5 = 1 ביט — אי-ודאות מלאה, בדיוק כמו שמצפים ממטבע הוגן.

בעץ החלטה בינארי (סיווג yes/no) עם p דוגמאות חיוביות ו-n שליליות, ההסתברויות הן היחסים p/(p+n) ו-n/(p+n): 

I(p/(p+n), n/(p+n)) = −[p/(p+n)]·log2[p/(p+n)] − [n/(p+n)]·log2[n/(p+n)]

בבעיית המסעדה יש 12 דוגמאות, p=n=6: I(6/12, 6/12) = I(0.5,0.5) = 1 ביט — כמות הידע הנדרשת מקסימלית, בדיוק כי הדוגמאות מאוזנות בול 50/50.

מלכודת נפוצה: לחשוב ש-"מידע = 1" הוא "טוב". למעשה 1 ביט הוא המקסימום של אי-ודאות — התחלה הכי גרועה שאפשר. ככל שהמידע החסר נמוך יותר, כך המצב "יותר ידוע" מראש.

נסו בעצמכם: שנו את יחס הדוגמאות החיוביות/שליליות ב-slider וצפו איך האנטרופיה משתנה — ואז חשבו remainder ו-Information Gain של מאפיין דמיוני:

5. Remainder ו-Information Gain — החישוב שהמבחן אוהב

מאפיין A מחלק את סט הדוגמאות E לתתי-קבוצות E₁,...,Eᵥ לפי ערכיו. בכל Eᵢ יש pᵢ דוגמאות חיוביות ו-nᵢ שליליות, ולכן אם ממשיכים בענף הזה עדיין נצטרך מידע נוסף של I(pᵢ/(pᵢ+nᵢ), nᵢ/(pᵢ+nᵢ)) ביטים. בממוצע, אחרי בחינת המאפיין A, מספר הביטים הדרוש: 

remainder(A) = Σ_{i=1}^{v} [(pi+ni)/(p+n)] · I(pi/(pi+ni), ni/(pi+ni))

Information Gain — ההפרש בין כמות המידע שנדרשה לפני השימוש במאפיין לבין הכמות הנדרשת לאחריו:

IG(A) = I(p/(p+n), n/(p+n)) − remainder(A)

נבחר תמיד את המאפיין עם ה-Information Gain המקסימלי (שקול לבחירת remainder מינימלי).

חישוב עבור Patrons (עמ' 51, בדיוק 3 ספרות אחרי הנקודה):

rem(Patrons) = (2/12)·I(0/2,2/2) + (4/12)·I(4/4,0/4) + (6/12)·I(2/6,4/6)
             = 1/6·0 + 1/3·0 + 1/2·I(1/3,2/3)
             = 0.5 · 0.918 = 0.459

חישוב עבור Type (עמ' 52–53) — כל ארבעת הענפים שומרים על יחס 50/50:

rem(Type) = (2/12)·I(1/2,1/2) + (2/12)·I(1/2,1/2) + (4/12)·I(1/2,1/2) + (4/12)·I(1/2,1/2)
          = (12/12)·I(1/2,1/2) = 1

Information Gain — ההשוואה הסופית (I(6/12,6/12)=1):

IG(Patrons) = 1 − 0.459 = 0.541 bits
IG(Type)    = 1 − 1     = 0 bits

Patrons עדיף בהרבה על Type כשורש העץ — 0.541 ביט לעומת 0. ענף "טהור" (רק חיוביים או רק שליליים) תורם תמיד I=0 ל-remainder — לכן מאפיין ששני ענפים מתוך שלושה שלו טהורים ינצח כמעט תמיד מאפיין ששומר על תערובת מלאה בכל ענף. זהו בדיוק הרעיון "מדוע Patrons ולא Type" מבלוק 3 — עכשיו עם מספר מדויק מאחוריו.

מקור: 15-decision-trees.pdf (עמ' 47–55)

6. אלגוריתם LEARN-DECISION-TREE (ID3/DTL) — ובניית עץ המסעדה

הפסאודו-קוד (מילה במילה מהשקף):

function LEARN-DECISION-TREE(examples, attributes, parent_examples) returns a tree

  if examples is empty then return PLURALITY-VALUE(parent_examples)
  else if all examples have the same classification then return the classification
  else if attributes is empty then return PLURALITY-VALUE(examples)
  else
      A ← argmax_{a in attributes} IMPORTANCE(a, examples)
      tree ← a new decision tree with root test A
      for each value v of A do
          exs ← {e : e in examples and e.A = v}
          subtree ← LEARN-DECISION-TREE(exs, attributes − A, examples)
          add a branch to tree with label (A = v) and subtree subtree
      return tree

ארבעת המקרים בבניית העץ (ניסוח שקוף):

  1. אם כל הדוגמאות הנותרות באותו סיווג (כולן חיוביות או כולן שליליות) — העלה מקבל את הסיווג.
  2. אם לא נותרו דוגמאות כלל — לא נצפתה דוגמה כזו; מחזירים ערך default (PLURALITY-VALUE של דוגמאות ההורה).
  3. אם יש גם חיוביות וגם שליליות — בוחרים את המאפיין הטוב ביותר (IG מקסימלי), ובונים תת-עץ חדש לכל ערך שלו מהדוגמאות המתאימות בלבד.
  4. אם לא נותרו מאפיינים לבדוק ועדיין יש דוגמאות משני הסוגים — יש בעיה של רעש (noise) או שהמאפיינים אינם מספקים מספיק מידע. מחזירים ערך default.

בונים בפועל: אחרי בחירת Patrons כשורש — הענפים None ו-Some טהורים, הופכים מיד לעלים (No ו-Yes בהתאמה). הענף Full (2 חיוביות + 4 שליליות: x2,x4,x5,x9,x10,x12) עדיין מעורב — דורש המשך רקורסיבי מהמאפיינים הנותרים.

ממשיכים רקורסיבית על 6 הדוגמאות של Full בלבד, עם המאפיינים הנותרים (בלי Patrons). לפי אותו תהליך IG מתגלה ש-Hungry הוא המאפיין הבא, ואז Type, ואז Fri/Sat — עד שכל 12 הדוגמאות מסווגות.

נקודה שחוזרת במבחן: העץ הנלמד משקף את הדוגמאות שקיבלנו, לא בהכרח את הפונקציה האמיתית שיצרה אותן. עם 12 דוגמאות בלבד יכולות להיות כמה עצים עקביים שונים — ID3 בוחר את זה שמתאים ל-Ockham's razor (הפשוט ביותר לפי IG חמדני בכל שלב), אבל אין ערובה שהוא זהה לעץ האמיתי.

מקור: 15-decision-trees.pdf (עמ' 39–40, 56–59, 82)

7. תרגול כיתה מלא — האם סמי גולש?

תרגול נוסף מההרצאה, עם שלושה מאפיינים בלבד — מתאים כתרגיל trace קצר ומלא מקצה לקצה. 8 דוגמאות של החלטות עבר של סמי:

מס'מזג אווירגובה גליםצפיפות בחוףהחלטה
1נעיםגבוהלא צפוףכן
2סוערגבוהלא צפוףכן
3מעונןנמוךלא צפוףכן
4נעיםגבוהלא צפוףלא
5סוערגבוהלא צפוףלא
6מעונןנמוךצפוףלא
7סוערנמוךצפוףלא
8מעונןנמוךצפוףלא

p=3 ("כן"), n=5 ("לא"). שלב 1 — כמות המידע בבעיה:

I(3/8, 5/8) = −0.375·log(0.375) − 0.625·log(0.625)
            = −0.375·(−1.415) − 0.625·(−0.678) = 0.954

שלב 2 — remainder לכל מאפיין:

rem(waves)   = (4/8)·I(2/4,2/4) + (4/8)·I(1/4,3/4)      = 0.5·1 + 0.5·0.811   = 0.906
rem(beach)   = (3/8)·I(0/3,3/3) + (5/8)·I(3/5,2/5)      = 0 + 0.625·0.970      = 0.606
rem(weather) = (2/8)·I(1/2,1/2) + (3/8)·I(1/3,2/3)·2    = 0.25·1 + 0.75·0.918  = 0.938

שלב 3 — Information Gain:

IG(waves)   = 0.954 − 0.906 = 0.048
IG(beach)   = 0.954 − 0.606 = 0.348   ← המקסימלי
IG(weather) = 0.954 − 0.938 = 0.016

צפיפות בחוף נבחרת כשורש. ענף "צפוף" (דוגמאות 6,7,8 — כולן "לא") טהור → עלה מיידי "לא גולש". ענף "לא צפוף" (דוגמאות 1–5, 3 כן + 2 לא) ממשיך רקורסיבית.

בתת-הבעיה (דוגמאות 1–5): I(3/5,2/5)=0.97. שני המאפיינים הנותרים (waves, weather) נותנים remainder=0.8 ו-IG=0.17 שווים בדיוק — תיקו. בוחרים גובה גלים (מנומק בשקפים). ענף "נמוך" (דוגמה 3 בלבד) טהור → גולש!; ענף "גבוה" (דוגמאות 1,2,4,5 — 2 כן + 2 לא) ממשיך עם המאפיין האחרון שנותר — מזג אוויר. בכל אחד מערכיו (נעים, סוער, מעונן) אין הכרעה חד-משמעית או שאין דוגמאות כלל — כל שלושת העלים מקבלים ערך default.

שני לקחים חשובים מהתרגול הזה: (1) כשיש תיקו ב-IG בין מאפיינים, אין כלל יחיד — נבחר את זה שיוצר עץ קומפקטי יותר (למשל פחות ענפים); (2) בשלבים מתקדמים מחשבים את I ואת ה-remainder רק מהדוגמאות הרלוונטיות לענף הנוכחי (כאן: 5, ואז 4 דוגמאות) — לא מכל 8 הדוגמאות מחדש. זו טעות נפוצה בתרגילים.

8. מאפיינים רציפים ופלט רציף

כשלמאפיין יש ערכים שלמים ומתמשכים (כמו גובה ומשקל), אי אפשר להוציא ענף לכל ערך אפשרי מקודקוד המבחן — יהיו אינסוף ענפים בעקרון. הפתרון: למצוא נקודות פיצול שמחלקות את טווח הערכים למספר קטן של תחומים (בדיוק כמו WaitEstimate שכבר חולק מראש ל-4 תחומים: 0-10, 10-30, 30-60, >60). רוצים שנקודות הפיצול עצמן ייבחרו כך שיתנו את ה-gain הטוב ביותר — פתרון זה דורש אלגוריתם תכנות דינמי, ומייקר משמעותית את הלמידה.

וכשהפלט עצמו רציף (למשל מחיר, ולא סיווג בדיד) — עץ החלטה רגיל אינו הפתרון המתאים. משתמשים בעץ רגרסיה: בכל עלה יש פונקציה ליניארית של תת-קבוצת מאפיינים מספריים, במקום ערך סופי בודד. אלגוריתם הלמידה צריך להחליט מתי להפסיק לפצל ולתת פונקציה במקום המשך פיצול.

9. הערכת ביצועים — קבוצת מבחן ועקומת למידה

כשפונקציית המטרה אינה ידועה — איך יודעים אם ההיפותזה שמצאנו טובה? בנוסף לקבוצת האימון שעל פיה בונים את ההיפותזה, אוספים קבוצת מבחן נפרדת, מאותה התפלגות כמו קבוצת האימון:

  1. אוספים הרבה דוגמאות.
  2. מחלקים לקבוצת אימון וקבוצת מבחן.
  3. מפעילים את אלגוריתם הלמידה על קבוצת האימון ליצירת h.
  4. מודדים את אחוז הדוגמאות מקבוצת המבחן שנצפו נכון ע"י h.
  5. חוזרים על 2–4 לגדלים וחלוקות שונות, ובוחרים את הטובה ביותר.

הגרף שמתקבל — אחוז ההצלחה על קבוצת המבחן כפונקציה של גודל קבוצת האימון — נקרא עקומת למידה.

מקור: 15-decision-trees.pdf (עמ' 86–88)

10. Overfitting ו-Cross-Validation

קיימת סכנה שההיפותזה שנמצא תהיה צמודה מדי לקבוצת האימון — Overfitting — ותשתמש בכללים חסרי משמעות אמיתית, שרלוונטיים רק לרעש הספציפי בקבוצת האימון ולא לפונקציה האמיתית. שלושה כיווני פתרון:

  • דוגמאות מפוזרות באופן אחיד על מרחב הבעיה (לא רק מקרים קיצוניים).
  • גיזום העץ ממאפיינים לא רלוונטיים — נמדד בכלים סטטיסטיים.
  • Cross-validation — מחלקים את קבוצת האימון לחלקים, ובכל פעם בונים עץ כשחלק אחר משמש קבוצת מבחן; מחשבים ממוצע של כל ההרצות.

ניואנס חשוב לשאלת מבחן: "היפותזה שמתאימה במדויק לכל דוגמאות האימון תמיד בעייתית" — לא נכון בהחלט. עצי החלטה בנויים להיות עקביים עם כל הדוגמאות (זו התכונה הבסיסית שלהם!). הבעיה מתעוררת רק כשההתאמה המלאה נובעת מרעש ספציפי לקבוצת האימון ולא מהפונקציה האמיתית — ואז ה-Overfitting פוגע ביכולת ההכללה לדוגמאות עתידיות. עקביות כשלעצמה אינה הבעיה; התלות ברעש היא הבעיה.

מקור: 15-decision-trees.pdf (עמ' 89–92)

11. למידה בהרכב — Ensemble ו-Boosting

כדי לשפר למידה, בונים מספר היפותזות (למשל 100 עצי החלטה שונים) ומשלבים את צפיותיהן — Ensemble Learning. בהינתן דוגמה חדשה, מריצים אותה על כולן ובוחרים במה שהרוב הכריעו. הרעיון: כדי שהשילוב יטעה, צריך שרוב העצים יטעו יחד — הרבה פחות סביר מטעות של עץ בודד.

Boosting משפר היפותזה חלשה (רק מעט טובה מהחלטה אקראית) להיפותזה טובה מאוד: לכל דוגמה j נותנים משקל wⱼ שבתחילה שווה 1 לכולן. בונים h₁ — מסווג חלק נכון וחלק שגוי. מעלים את המשקל של הדוגמאות השגויות ומורידים את משקל הנכונות, ובונים מהסט הממושקל מחדש את h₂. התהליך ממשיך עד M היפותזות. ההכרעה הסופית — ממוצע משוקלל של הכרעות כל ההיפותזות, כשמשקל כל היפותזה תלוי באחוזי הצלחתה.

מספרים לזכור לגבי בעיית המסעדה: עץ החלטה בודד — 81% הצלחה (על 100 דוגמאות אימון). Boosting עם M=5 — 93%. M=20 — הצלחה מלאה. זו הדגמה כמותית לכך ש-Ensemble Learning עדיף בעקביות על עץ בודד.

העשרה: הרעיונות האלה הם הבסיס לשיטות למידת מכונה מודרניות שלא הופיעו בשקפים במפורש: Random Forest (הרכבה של עצי החלטה עצמאיים על מדגמי נתונים אקראיים, בלי משקלים משתנים) ו-Gradient Boosting/XGBoost (גרסה מתקדמת של Boosting) — שתי המשפחות הנפוצות ביותר כיום לבעיות סיווג טבלאיות, ושתיהן מרחיבות בדיוק את שני הרעיונות שראינו כאן: הרכבה והדגשת דוגמאות קשות.

מקור: 15-decision-trees.pdf (עמ' 93–98)

12. סיכום דרך העבודה ומלכודות למבחן

סיכום דרך העבודה לבניית עץ החלטה (5 צעדים):

  1. מחשבים את המידע החסר I בבעיה (בהתחלה — מכל הדוגמאות; בשלבים מתקדמים — רק מהדוגמאות הרלוונטיות לענף הנוכחי).
  2. לכל מאפיין מחשבים כמה מידע יישאר חסר לאחר השימוש בו — remainder.
  3. לכל מאפיין מחשבים כמה מידע הוא תורם — Information Gain.
  4. בוחרים את המאפיין עם ה-IG המקסימלי כשורש תת-העץ הנוכחי.
  5. מחלקים את הדוגמאות לפי ערכי המאפיין הנבחר; ענף עם סיווג יחיד → עלה, ענף מעורב → חוזרים לצעד 1 (בלי המאפיין שכבר נוצל).
ערך Iמשמעות
I(1,0) = I(0,1) = 0ענף טהור — אין אי-ודאות כלל
I(0.5,0.5) = 1אי-ודאות מקסימלית (50/50)
I(1/3,2/3) = 0.918ערך חוזר בהרבה תרגילים

מלכודות שחוזרות במבחן:

  • מידע=1 אינו "טוב" — הוא מקסימום אי-ודאות, לא ודאות.
  • בוחרים IG מקסימלי, שקול ל-remainder מינימלי — לא ההפך.
  • ענף טהור תמיד תורם 0 ל-remainder, בלי קשר לגודלו היחסי.
  • מחשבים I ו-remainder רק מהדוגמאות הרלוונטיות לענף הנוכחי בשלבים מתקדמים — לא מכל הדאטה המקורי מחדש.
  • עקביות מלאה עם קבוצת האימון אינה "רעה" כשלעצמה — עצי החלטה בנויים להיות עקביים; הבעיה היא כשההתאמה נובעת מרעש ולא מהפונקציה האמיתית (Overfitting).
  • קבוצת אימון גדולה אינה מבטיחה עץ מוצלח — רק אם הדוגמאות מייצגות נאמנה את התנהגות הפונקציה האמיתית, לא רק מקרי קיצון.
  • Ensemble עדיף בעקביות על עץ בודד (81%→93%→100% עם Boosting) — כי כשל השילוב דורש שרוב העצים יטעו יחד, בעוד כשל עץ בודד סביר הרבה יותר.
  • גם עם דוגמאות מייצגות היטב, יש חשיבות לסדר בחירת המאפיינים — מאפיין טוב מוביל לעץ קומפקטי והכרעה מהירה, לא רק לתוצאה נכונה.

א. 5 דוגמאות "כן" ו-2 דוגמאות "לא" מתוך 7 (p=5, n=2): 

I(5/7, 2/7) = −5/7·log(5/7) − 2/7·log(2/7) = 0.863

ב. יש צורך בהרבה ידע. חוסר הוודאות גבוה מאוד — לא רחוק מהמקסימום של חוסר מידע שהוא 1.

ג. ראשית מחשבים remainder לכל אחד משלושת המאפיינים:

Rem(parking) = 4/7·I(4/4,0/4) + 3/7·I(1/3,2/3) = 4/7·0 + 3/7·0.918 = 0.393
Rem(friend)  = 4/7·I(3/4,1/4) + 3/7·I(2/3,1/3) = 4/7·0.81 + 3/7·0.918 = 0.856
Rem(songs)   = 2/7·I(1,0) + 3/7·I(2/3,1/3) + 2/7·I(1/2,1/2) = 0 + 3/7·0.918 + 2/7 = 0.679
Gain(parking) = 0.863 − 0.393 = 0.47   ← המקסימלי
Gain(friend)  = 0.863 − 0.856 = 0.007
Gain(songs)   = 0.863 − 0.679 = 0.184

יש חניה? נבחר כשורש. כן (דוגמאות 2,4,5,7 — כולן "כן") — טהור, עלה מיידי. לא (דוגמאות 1,3,6 — 1 כן + 2 לא) ממשיך רקורסיבית.

בתת-הבעיה (דוגמאות 1,3,6 בלבד): I(1/3, 2/3) = 0.918. 

Rem(friend) = 1/3·I(0/1,1/1) + 2/3·I(1/2,1/2) = 0 + 2/3·1 = 0.667
Rem(songs)  = 2/3·I(1/2,1/2) + 1/3·I(0,1)      = 2/3·1 + 0     = 0.667
Gain(friend) = 0.918 − 0.667 = 0.251
Gain(songs)  = 0.918 − 0.667 = 0.251   ← תיקו!
