0. האינטואיציה — לחפש בעיניים עצומות

בפרק הקודם ניסחנו בעיית חיפוש כרביעייה: מרחב מצבים, פונקציית עוקב, מבחן מטרה ועלות מסלול. עכשיו נשאל: איך בפועל "מטיילים" במרחב הזה כדי למצוא פתרון? חיפוש לא מיודע (uninformed / blind search) עונה בלי שום רמז נוסף על המרחק למטרה — האלגוריתם לא "יודע" כמה רחוק המטרה, הוא רק רואה את מבנה הבעיה עצמה: אילו מצבים נוצרים, ובאיזה סדר. בפרק הבא (חיפוש מיודע) נוסיף heuristic שמנחה לאן כדאי ללכת — כאן, בלי שום ניחוש.

המצגת סוקרת שישה אלגוריתמים כאלה: BFS, Bidirectional Search, DFS, Depth-Limited Search (DLS), Iterative Deepening Search (IDS) ו-Uniform-Cost Search (UCS). הבשורה הטובה: כולם אותו שלד בדיוק — עץ/גרף חיפוש שבו קודקוד נוצר, בודקים אם הוא המטרה, ואם לא — מפעילים עליו את פונקציית העוקב ומכניסים את הבנים ל"עלים" (frontier). ההבדל היחיד בין האלגוריתמים הוא מבנה הנתונים של העלים — מי נבחר "הבא בתור" להתפתחות:

  • BFS — תור FIFO (הראשון שנכנס, ראשון שיוצא) → מפתח שכבה-שכבה.
  • DFS — מחסנית LIFO (האחרון שנכנס, ראשון שיוצא) → צולל לעומק.
  • UCS — תור ממוין לפי עלות מסלול g → מפתח את הזול ביותר.
  • DLS — כמו DFS, אך עם תקרת עומק מלאכותית ℓ.
  • IDS — מריץ DLS שוב ושוב עם ℓ = 0, 1, 2, ….
  • Bidirectional — שני חיפושים (קדימה ואחורה) שנפגשים באמצע.

לאורך כל הפרק נשווה כל אלגוריתם לפי ארבעה קריטריונים קבועים: שלמות (האם תמיד מוצא פתרון אם קיים?), אופטימליות (האם הפתרון הוא הזול/הקצר ביותר?), סיבוכיות זמן וסיבוכיות מקום. שלושה משתנים חוזרים בכל חישוב: b (גורם הפיצול המרבי), d (עומק הפתרון הזול ביותר) ו-m (העומק המרבי של מרחב המצבים, שעשוי להיות ∞).

נסו בעצמכם: בחרו אלגוריתם חיפוש והריצו אותו צעד-אחר-צעד על גרף לדוגמה — עקבו אחרי מבנה הנתונים של ה"עלים" (תור/מחסנית), הקודקוד שנבחר בכל שלב, וסדר הפיתוח המלא שמתקבל.

1. Breadth-First Search (BFS) — לפתח שכבה-שכבה

BFS מפתח תמיד את העלה הכי קרוב לשורש: רק אחרי שכל השכנים של השורש פותחו עוברים לשכנים-של-שכנים, וכן הלאה. מבנה הנתונים: תור FIFO. אם המטרה לא נמצאת — BFS יסרוק את כל מרחב המצבים.

פסאודו-קוד (גרסת עץ — בדיקת מטרה מוקדמת, "early goal test"):

function BREADTH-FIRST-SEARCH(problem) returns a solution node or failure
  node ← NODE(problem.INITIAL)
  if problem.IS-GOAL(node.STATE) then return node
  leaves ← a FIFO queue, with node as an element

  while not IS-EMPTY(leaves) do
    node ← Dequeue(leaves)
    for each child in EXPAND(problem, node) do
      s ← child.STATE
      if problem.IS-GOAL(s) then return child

      add child to leaves
  return failure

הבדלי שמות מהספר: במצגת frontier הוחלף ב-leaves, ו-POP הוחלף ב-Dequeue. שימו לב: בדיקת המטרה נעשית בעת יצירת הבן — לא כשהוא נשלף מהתור. זו נקודה קריטית שמשפיעה על ההרצה למטה.

מצבים חוזרים ו-Graph Search

חוסר יכולת לזהות מצבים חוזרים יכול להפוך בעיה לינארית לבעיה אקספוננציאלית: במרחב מצבים לינארי A–B–C–D עם קשתות דו-כיווניות, עץ חיפוש שלא מזהה חזרות מייצר כפילות אקספוננציאלית של אותם מצבים (בכל רמה מכפילים שוב).

הפתרון: Graph BFS מוסיף קבוצת reached (מצבים שכבר הושגו):

function BREADTH-FIRST-SEARCH(problem) returns a solution node or failure
  node ← NODE(problem.INITIAL)
  if problem.IS-GOAL(node.STATE) then return node
  leaves ← a FIFO queue, with node as an element
  reached ← {problem.INITIAL}
  while not IS-EMPTY(leaves) do
    node ← POP(leaves)
    for each child in EXPAND(problem, node) do
      s ← child.STATE
      if problem.IS-GOAL(s) then return child
      if s is not in reached then
        add s to reached
        add child to leaves
  return failure

ההבדל היחיד מגרסת העץ: בן מוכנס לתור רק אם מצבו טרם הושג — כך נמנעים מלטפל שוב באותו Oradea שכבר הופיע דרך Sibiu.

2. Bidirectional Search — שני חיפושים נפגשים באמצע

חיפוש דו-כיווני מנהל שני חיפושים במקביל: אחד קדימה מהמצב ההתחלתי, ואחד אחורה מהמצב הסופי (המטרה) — ועוצר כששני החיפושים נפגשים. בכל שלב בודקים האם הקודקוד הבא נמצא כבר בצד השני — אם כן, יש מסלול מלא.

מבני הנתונים: לכל צד (קדימה/אחורה) יש תור עלים משלו, וטבלת גיבוב של הקודקודים ה"סגורים" (שכבר פותחו) שלו. כשמייצרים קודקוד חדש, בודקים אם הוא כבר בטבלת הסגורים או בעלים של הצד השני — אם כן, משלבים את שני חצאי המסלול. אם לא, מוסיפים אותו לתור העלים של הצד שלו.

הרצה מלאה — Arad ↔ Bucharest

אותה מפת רומניה. שני שורשים: Arad (חיפוש קדימה, נפתח ראשון) ו-Bucharest (חיפוש אחורה, נפתח שני). ההתקדמות:

  • 1 — Arad: נפתח קדימה; בניו Timisoara, Sibiu, Zerind נוצרים.
  • 2 — Bucharest: נפתח אחורה; בניו (הליכה נגד כיוון הקשתות) Fagaras, Giurgiu, Pitesti, Urziceni נוצרים.
  • 3 — Sibiu: נפתח קדימה; בניו Rimnicu Vilcea, Fagaras, Oradea נוצרים — וFagaras כבר נמצא בעלים של החיפוש האחורי!

המפגש: Fagaras נוצר בחיפוש קדימה בעודו כבר קיים בעלים של החיפוש האחורי — משלבים את שני חצאי המסלול: Arad → Sibiu → Fagaras + Fagaras → Bucharest. סה"כ רק 3 פיתוחים עד המפגש — הרבה פחות מ-5 הפיתוחים של BFS חד-כיווני!

למה זה מהיר יותר — bd/2 במקום bd

עץ חיפוש רגיל בגובה d עולה ~bd. חיפוש דו-כיווני מריץ שני עצים קטנים יותר, כל אחד רק עד עומק d/2 — ועולה ~bd/2 + bd/2, הרבה פחות מ-bd: שני "חרוטים" קטנים מכסים פחות שטח מחרוט אחד גדול.

מקור: 04-uninformed-search.pdf (עמ' 21–30)

3. Depth-First Search (DFS) — לצלול לעומק, רגיש לסדר

DFS מפתח תמיד את העלה הכי עמוק ראשון. מבנה הנתונים: מחסנית LIFO. כשמגיעים לעלה שאין לו המשך, חוזרים צעד אחורה (backtrack) עד שמגיעים לקודקוד העמוק ביותר הבא בעץ שעדיין לא מוצה.

הרצה 1 — סדר לקסיקוגרפי: מסלול קצר "במקרה"

אותה בעיה: Arad ל-Bucharest, שוברים שוויון לפי סדר אלפביתי. מ-Arad הבן הראשון לקסיקוגרפית (מבין Sibiu, Timisoara, Zerind) הוא Sibiu; מ-Sibiu הבן הראשון שאינו מוכר הוא Fagaras; מ-Fagaras מגיעים ישר ל-Bucharest — המטרה!

  • 1 Arad → 2 Sibiu → 3 Fagaras → 4 Bucharest (מטרה!)

במקרה הזה DFS מוצא מסלול קצר (4 צעדים) מהר — אבל זה מזל של סדר שבירת השוויון, לא תכונה מובטחת של האלגוריתם.

הרצה 2 — סדר אחר: מסלול ארוך ויקר פי שניים

עם שבירת שוויון אחרת, DFS "צולל" בענף אחר לגמרי:

  • 1 Arad → 2 Timisoara → 3 Lugoj → 4 Mehadia → 5 Drobeta → 6 Craiova → 7 Pitesti → 8 Bucharest (מטרה!)

הערת תיקון: בשקף המקורי הקודקוד השביעי (הבן של Craiova שממנו יוצאים ל-Bucharest) מתויג בטעות "Rim" — אך Craiova מחוברת גם ל-Rimnicu Vilcea וגם ל-Pitesti, ורק Pitesti מחוברת ל-Bucharest. כאן מוצג התיוג הנכון: Pitesti(7) → Bucharest(8).

מלכודת למבחן: DFS אינו אופטימלי — הוא עלול להחזיר מסלול ארוך פי שניים מהאופטימלי, תלוי לגמרי בסדר שבירת השוויון. אבל ייתכן ש-DFS יחזיר במקרה את אותו מסלול כמו BFS — אם המסלול שנבחר מוביל בטעות למספר הצעדים המינימלי (כמו בהרצה 1 למעלה). זה לא מובטח ולא אופייני, אבל זה נכון עקרונית.

מקור: 04-uninformed-search.pdf (עמ' 31–48)

4. Depth-Limited Search (DLS) — DFS עם תקרה

כדי למנוע מ-DFS "לצלול" לאינסוף, מוסיפים פרמטר ℓ — גובה מלאכותי של העץ. קודקוד ברמה ℓ נחשב חסר ילדים מבחינת האלגוריתם, גם אם במציאות יש לו המשך.

פסאודו-קוד (מילה במילה מהמקור — כולל שגיאות הכתיב isEmpy, cuttof):

DepthLimitedSearch(problem,limit)
  { Leaves = A Stack with MakeNode(initialState(problem)) as an element
    result = failure
    While not isEmpy (leaves)   do:
       { node = Pop(leaves)
         if GoalTest(problem,node) = true
             return Solution(node)
        if Depth[node] = limit
            result= cuttof
         else if node is not in ClosedList
             for each successor in Expand(node, problem)
                 Push(successor, leaves)
        }
     return result
     }

קודקוד ברמת limit לא מתפתח — במקום זאת מסמנים result = cuttof ולא ממשיכים דרכו. שימו לב: יש ערך החזרה שלישי מלבד פתרון/כישלון — cutoff, שאומר "לא נמצא פתרון בתוך גבול העומק, אבל ייתכן שקיים מעבר לו".

5. Iterative Deepening Search (IDS) — הכי טוב משני העולמות

השאלה שמובילה לכאן: איך משלבים את יתרונות DFS (זיכרון) ו-BFS (שלמות ומציאת פתרון רדוד)? התשובה: Iterative Deepening — הרצה חוזרת של DLS עם תקרת עומק שעולה ב-1 בכל פעם, החל מ-0. חשוב: בכל שלב מתחילים מהתחלה — כי בסיום הרצת DFS רגילה, הקודקודים אינם נשמרים בזיכרון. 

IterativeDeepeningSearch(problem)
  {
   for depth = o to ∞
      {
      result = DepthLimitedSearch (problem, depth)
      If result ≠cutoff
         return result
       }
}

מעלים את ה-limit ב-1 בכל איטרציה ועוצרים ברגע שהתוצאה אינה cutoff.

הרצה על עץ בינארי — ℓ = 0, 1, 2, 3

ℓ=0: רק השורש נבדק ומיד "נמחק" — cutoff. ℓ=1: השורש ושני בניו נבדקים ומוחקים — עדיין cutoff. עם ℓ=2 ו-ℓ=3 העץ גדל בהתאם:

אותו רעיון על מפת רומניה (הדגמה בכתב יד)

limit=0: רק Arad נבדק ונמחק. limit=1: Arad ובניו Sibiu, Timisoara, Zerind — עדיין אין פתרון. limit=2: מתווספת רמה נוספת (Fagaras, Oradea, Rimnicu Vilcea, Lugoj וכו') — עדיין closed, אין פתרון. רק ב-limit=3 מגיעים לרמה 3 ומוצאים את Bucharest — d=3.

למה "לחזור על העבודה" כמעט לא עולה כלום

נראה בזבזני שבכל איטרציה מפתחים מחדש את כל הרמות העליונות — אבל העלים הם החלק המשמעותי בעץ (אקספוננציאלית יותר קודקודים ברמה האחרונה מכל שאר הרמות ביחד). לכן פיתוח חוזר של קודקודים פנימיים לא מעלה את סיבוכיות הזמן, ובתמורה מקבלים חיסכון עצום במקום.

הנוסחה המדויקת (חובה לזכור):

N_BFS = b^0 + b^1 + b^2 + … + b^d
N_IDS = (d+1)·b^0 + d·b^1 + (d-1)·b^2 + … + 2·b^(d-1) + b^d

הדוגמה המספרית שחוזרת במבחנים — b=10, d=5:

חישובסה"כ קודקודים
N_BFS1+10+100+1,000+10,000+100,000111,111
N_IDS6+50+400+3,000+20,000+100,000123,456

Overhead = (123,456 − 111,111) / 111,111 ≈ 11% בלבד. תוספת זעירה במחיר סיבוכיות מקום כמו DFS במקום כמו BFS — עסקה מצוינת. זו דוגמה "יפה" שהמרצה חוזרת עליה — מועמדת קלאסית לשאלת מבחן.

מקור: comparison-uninformed-search.pdf (עמ' 57–69)

6. Uniform-Cost Search (UCS) — לפתח את הזול ביותר

UCS מפתח את הקודקוד בעל עלות מסלול נמוכה ביותר (g) ראשון. מבנה הנתונים: תור עלים ממוין לפי עלות מסלול. שאלת המצגת: "מה קורה כשעלות כל שלב זהה?" — התשובה: UCS מתנוון בדיוק ל-BFS.

דוגמה קטנה לחימום

עץ עם עלויות קשתות: a→b (2), a→c (1), b→f (1), b→G (2), c→d (1), c→e (2). מחפשים מסלול קצר מ-a ל-G:

שלבקודקוד שנפתח (g)open list אחרי הפיתוח
1a (0)b(2), c(1)
2c (1)b(2), d(2), e(3)
3b (2)d(2), e(3), f(3), G(4)
4d (2)e(3), f(3), G(4)
5e (3)f(3), G(4)
6f (3)G(4)
7G (4) — מטרה!ריק

המטרה נמצאה בעלות 4 — המסלול a → b → G (2+2=4).

הרצה מלאה — Arad ל-Bucharest (עלות סופית 418)

אותה מפה, אותם משקלים. כאן ההבדל הקריטי מ-BFS בולט במיוחד: בדיקת המטרה מתבצעת רק בעת שליפה מהתור (לא ביצירה!). לכן קודקוד Bucharest שנוצר מוקדם ויקר לא נבחר לפני שנמצא מסלול זול יותר:

7. טבלת השוואה מסכמת + מלכודות למבחן

המרצה מספקת טבלת השוואה ריקה כתרגול עצמי (comparison-uninformed-search.pdf) — כאן היא מולאת על בסיס כל התכונות שנדונו לעיל לכל אלגוריתם, בתוספת ניתוח פורמלי סטנדרטי (התואם את ספר הלימוד שעליו מבוססת ההרצאה) עבור פרטים שלא פורטו במפורש בשקפים.

תכונהBFSUCSBidirectionalDFSDLSIDS
שלמות כן (b סופי) כן (עלות כל צעד ≥ ε>0) כן (כמו BFS) לא (עלול "ללכת לאיבוד" בעץ אינסופי) לא (ℓ<d ⇒ cutoff, גם אם קיים פתרון) כן (כמו BFS)
אופטימליות רק אם עלויות שוות כן רק אם עלויות שוות לא לא רק אם עלויות שוות

סיבוכיות מקום O(bd) דומה ל-BFS (≈bd כשעלויות שוות) O(bd/2) O(b·m) O(b·ℓ) O(b·d) (כמו DFS)

שרשרת ההכלה שכדאי לזכור: BFS הוא מקרה פרטי של UCS — כשעלות כל פעולה זהה, המיון של UCS לפי g מתלכד עם המיון של BFS לפי עומק, ולכן UCS מתנהג בדיוק כמו BFS.

מלכודות למבחן שחוזרות שוב ושוב:

  • closed list שומר קודקודים שכבר טופלו (נבדקו ופונקציית העוקב הופעלה עליהם) — אין קשר לעלות/ערך המסלול שלהם. זו לא רשימת "קודקודים גרועים".
  • cutoff ≠ failure: failure = כל מרחב המצבים נסרק ואין פתרון; cutoff = לא נמצא פתרון בתוך גבול העומק, אך ייתכן שקיים פתרון עמוק יותר.
  • בעץ חיפוש שומרים רק מצביע להורה בכל קודקוד — לא את כל המסלול המצטבר. שחזור המסלול נעשה בדיעבד על ידי מעקב אחורה דרך שרשרת ההורים.
  • טבלת גיבוב נדרשת: ב-bidirectional (לזיהוי מפגש בין שני הצדדים) ובכל אלגוריתם הרץ על גרף (לא עץ) — עבור ה-closed list.
  • Iterative Deepening אינו "יעיל יותר" מ-BFS בזמן ריצה — סיבוכיות הזמן האסימפטוטית שווה; ההבדל האמיתי הוא בסיבוכיות המקום, לא בזמן.
  • פונקציית העוקב אינה "חכמה" — היא לא יודעת מהו מבחן המטרה ולא שואפת לפתרון יעיל, רק מחזירה מעברים חוקיים מהמצב הנוכחי.

שאלות נכון/לא נכון

א. UCS הוא מקרה פרטי של A*. — נכון. A* בוחר קודקוד עם ערך מינימלי של g(v)+h(v), ואילו UCS בוחר לפי ערך מינימלי של g(v) בלבד. אם h(v)=0 נקבל בדיוק את אותה עבודה.

ב. היוריסטיקה קטנה מ-h* אך רחוקה ממנה תגרום לפתרון לא-אופטימלי. — לא נכון. כל עוד ההיוריסטיקה קטנה מהמרחק האמיתי, A* תמיד יחזיר ערך אופטימלי; ככל שהיא רחוקה מהאמת, A* פחות ממוקד ומפתח יותר קודקודים — אבל עדיין מגיע לפתרון האופטימלי.

ג. BFS הוא מקרה פרטי של UCS. — נכון. כשעלות כל פעולה זהה, UCS (הבוחר לפי עלות מסלול מינימלית) מתנהג בדיוק כמו BFS (הבוחר לפי עומק).

ד. BFS הוא מקרה פרטי של A*. — נכון. משילוב הסעיפים א' ו-ג': BFS ⊂ UCS ⊂ A* כאשר עלות כל צעד קבועה וגם h(n)=0.

ה. מקום הזיכרון ש-A* צריך יכול להיות טוב יותר מ-DFS. — נכון. DFS דורש O(bm) (כל צאצאי המסלול הנוכחי שמורים). אם ההיוריסטיקה מושלמת (h=h*), A* מפתח רק את המסלול האופטימלי (אורך d) ודורש O(bd) — ומכיוון ש-d ≤ m, זה יכול להיות טוב יותר.
